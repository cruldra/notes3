派生而非物化

派生而非物化（Derived, not Materialized）是一条针对数据建模的判断标准：当一份数据在概念上可以从既有事实算出来时，优先把它作为查询时派生的视图，而不是落到独立存储再去维护。

它不是哪位名人原创的命名，而是数据库领域、函数式编程、前端状态管理在不同语境下反复重新发现的同一条经验法则。它是 DRY（不允许两份真相）在数据维度的具体落地，也是"单一事实源（SSOT）"原则的另一面。

它的核心主张只有一句：

能算出来的东西，不要存。

它在反对什么​

派生而非物化反对的是冗余物化派生数据——把一份本可以由源头算出的数据，复制一份独立存起来，然后试图通过应用代码维护两份的同步。

  • 在 user 表里冗余存"最近订单数" → 必须在每次订单写入 / 退款 / 取消处同步
  • 把 filter 后的列表存成独立 state → 原列表变化时必须手动更新
  • 把"用户 ↔ 智能体的关系"存成中间表 → 创建 / 购买 / 退款 / 下架要在多处镜像
  • 缓存 derived view 但缓存失效逻辑漏一处就脏数据

物化派生数据 = 双源同步债务：每漏掉一个写入点，系统就进入不一致状态。这种 bug 通常表现为"看起来一切正常，但某个角落数字对不上"——而且很难复现。

反例 vs 正例​

例 1：用户的最近订单数​

❌ 物化

class User:
    id: int
    name: str
    last_order_count: int   # 冗余字段, 由 orders 派生

# 必须在所有改动 orders 的地方同步
def create_order(user_id, ...):
    db.execute("INSERT INTO orders ...")
    db.execute("UPDATE users SET last_order_count = last_order_count + 1 WHERE id = ?", user_id)

def refund_order(order_id):
    db.execute("UPDATE orders SET refunded_at = NOW() WHERE id = ?", order_id)
    # 漏写！last_order_count 永远不会因退款而减少

退款 / 取消 / 用户合并 / 数据迁移……每个写入点都要镜像处理。漏一个就是 bug。

✅ 派生

SELECT COUNT(*) FROM orders
WHERE user_id = ? AND refunded_at IS NULL

唯一事实源是 orders 表本身。退款只写 refunded_at，查询结果自动正确。

性能担心？加索引。真不行再用物化视图——让数据库自动维护，不是应用代码自己同步：

CREATE MATERIALIZED VIEW user_order_count AS
    SELECT user_id, COUNT(*) AS cnt FROM orders
    WHERE refunded_at IS NULL GROUP BY user_id;
REFRESH MATERIALIZED VIEW CONCURRENTLY user_order_count;

这是把同步责任交给数据库，而不是散在应用代码里。

例 2：前端过滤列表​

❌ 物化

const [users, setUsers] = useState([])
const [activeUsers, setActiveUsers] = useState([])  // 派生但物化

useEffect(() => {
    setActiveUsers(users.filter(u => u.active))
}, [users])

activeUsers 是 users 的派生，但被存成独立 state，必须手动同步。漏一次依赖、forget 一次 useEffect，UI 就脏了。

✅ 派生

const [users, setUsers] = useState([])
const activeUsers = useMemo(() => users.filter(u => u.active), [users])

派生发生在渲染时。零同步代码。

例 3：用户与可访问资源的关系​

❌ 物化

建一张 user_resource_link 表，在用户每次"获得资源访问权"时（创建 / 购买 / 被赠送 / 首次使用）插入一条；权限失去时删除（退款 / 下架 / 资源删除）。

失败场景：每个写入点都要保证写入；远程调用失败 + 重试逻辑 + 退款镜像逻辑 + admin 后台旁路……同步逻辑漏一处就出 bug。

✅ 派生

SELECT DISTINCT resource_id FROM (
    -- 我创建的
    SELECT id FROM resources WHERE owner_id = :uid
    UNION ALL
    -- 我购买的（未退款的订单）
    SELECT unnest(p.bound_resources) FROM orders o
        JOIN products p ON o.product_id = p.id
        WHERE o.user_id = :uid AND o.refunded_at IS NULL
    UNION ALL
    -- 我用过的
    SELECT resource_id FROM access_log WHERE user_id = :uid
) t

事实源已经存在（resources 表 / orders 表 / access_log），关系是它们 union 出来的视图。无 sync 逻辑。

误区：性能恐惧导致过早物化​

最常见的物化理由是"派生查询会慢"——但绝大多数情况这是未经测量的恐惧：

现实真相
现代数据库的 JOIN + 索引在 10⁶ 量级数据下都是毫秒级不要凭直觉判断慢
物化的复杂度从写下那一刻就开始计费性能收益可能要 10⁴ 行数据后才显现
应用代码维护的物化必然有 sync bug物化视图（DB 管理）比手写同步靠谱得多
真到性能瓶颈时，先加索引、再换物化视图不需要在 day-1 就物化

先派生，profiling 出问题再物化，几乎永远是更好的选择。

何时可以物化（例外）​

不是绝对反对物化。下列情形物化合理：

情形为什么
快照语义"下单时的商品价格"是历史事实，不是当前 product 的派生——必须物化
跨多源聚合派生需要打多个外部 API / 数据库，查询时算成本太高
profiler 证明的瓶颈不是猜的"应该会慢"，是真量出来达不到 SLO
数据库自动维护的物化视图应用代码不参与同步，只读视图
append-only 事实日志写入只有插入没有更新/删除，没有同步问题

判断标准：物化产生的同步代码，由谁来写、谁来维护？ 由数据库 / 框架自动维护 → 可接受；由应用代码维护 → 需极强理由。

自检清单​

设计一个新字段 / 新表 / 新 state 时问自己：

  1. 这数据是从哪些事实派生的？事实源在哪里？说不清 → 它本身就是事实，不在本原则讨论范围
  2. 物化版本由谁来同步？应用代码 → 警惕；数据库 / 框架 → 安全
  3. 有多少个写入点会改变源头？>= 2 → 同步漏写概率显著上升
  4. 有没有"反向事件"（退款 / 删除 / 失效）需要镜像处理？容易漏的就是这些
  5. 物化的性能收益有 profile 数据吗？没有 → 凭感觉物化，大概率多此一举
  6. 派生查询能用一句 SQL / 一个 selector 写完吗？能 → 几乎肯定该派生

与其他原则的关系​

  • DRY 在数据维度的具体落地——派生而非物化 = 数据 DRY：同一份事实不能两处存
  • KISS 同向——派生通常是更简单的代码（少一份同步逻辑）
  • YAGNI 同向——物化往往是为"未来可能的性能问题"提前付出代价，YAGNI 反对预测式优化
  • AHA（Avoid Hasty Abstractions）的数据版本——不要急着把派生关系抽象成独立实体

小结​

永远没有 sync bug 的最佳办法，是永远不要存两份。

派生而非物化不是反对存数据，而是反对为同一份事实存两份相互独立的版本。它要求的是对事实源的诚实——承认数据库已经知道答案，应用代码不需要再算一遍存起来。

需要物化时，让数据库 / 框架替你维护；需要应用代码维护的物化，几乎总是错的设计。
