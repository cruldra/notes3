Review

Review 面板可帮助你了解 Codex 做了哪些更改、给出有针对性的反馈，并决定保留哪些内容。

Review 面板只适用于位于 Git 仓库中的项目。如果你的项目还不是 Git 仓库，Review 面板会提示你创建一个。

显示哪些更改​

Review 面板反映的是 Git 仓库的状态，而不仅是 Codex 编辑过的内容。这意味着它会显示：

  • Codex 做出的更改
  • 你自己做出的更改
  • 仓库中的任何其他未提交更改

默认情况下，Review 面板聚焦于未提交更改。你也可以将范围切换为：

  • 所有分支更改：与基准分支进行 diff
  • 上一轮更改：只显示最近一次助手回合中的更改

在本地工作时，你还可以在未暂存和已暂存更改之间切换。

导航 Review 面板​

  • 点击文件名通常会在你选择的编辑器中打开该文件。你可以在设置中选择默认编辑器。
  • 点击文件名背景可以展开或折叠 diff。
  • 按住 Cmd 并点击单行，会在你选择的编辑器中打开该行。
  • 如果你认可某项更改，可以暂存更改或还原不需要的更改。

用内联评论给出反馈​

内联评论允许你把反馈直接附加到 diff 中的具体行上。这通常是引导 Codex 精准修复问题的最快方式。

留下内联评论：

  1. 打开 Review 面板。
  2. 将鼠标悬停在要评论的行上。
  3. 点击出现的 + 按钮。
  4. 写下反馈并提交。
  5. 留完反馈后，回到线程中发送一条消息。

由于评论绑定到具体行，Codex 可以比处理一般性指令时更精确地回应。

Codex 会把内联评论视为代码审查指导。留下评论后，发送一条后续消息明确你的意图，例如：“处理这些内联评论，并保持最小范围”。

代码审查结果​

如果你使用 /review 运行代码审查，评论会直接以内联形式显示在 Review 面板中。

Pull Request 审查​

当 Codex 对你的仓库拥有 GitHub 访问权限，并且当前项目位于 Pull Request 分支时，Codex 应用可以帮助你在应用内处理 Pull Request 反馈。侧边栏会显示 Pull Request 上下文和审查者反馈，Review 面板会把评论与 diff 一起显示，因此你可以在同一个线程中要求 Codex 处理问题。

安装 GitHub CLI（gh）并通过 gh auth login 完成认证，这样 Codex 才能加载 Pull Request 上下文、审查评论和已更改文件。如果缺少 gh 或未完成认证，Pull Request 详情可能不会出现在侧边栏或 Review 面板中。

如果你想在一个地方完成完整的修复循环，可以使用以下流程：

  1. 在 Pull Request 分支上打开 Review 面板。
  2. 查看 Pull Request 上下文、评论和已更改文件。
  3. 要求 Codex 修复你想处理的具体评论。
  4. 在 Review 面板中检查生成的 diff。
  5. 准备好后，将更改暂存、提交并推送到 Pull Request 分支。

关于 GitHub 触发的审查，见在 GitHub 中使用 Codex。

暂存和还原文件​

Review 面板包含 Git 操作，方便你在提交前整理 diff。

你可以在以下层级暂存、取消暂存或还原更改：

  • 整个 diff：使用 Review 标题栏中的操作按钮，例如“Stage all”或“Revert all”
  • 单个文件：暂存、取消暂存或还原某个文件
  • 单个 hunk：暂存、取消暂存或还原某个 hunk

当你想接受部分工作时使用暂存；当你想丢弃更改时使用还原。

已暂存和未暂存状态​

Git 可以在同一个文件中同时表示已暂存和未暂存更改。发生这种情况时，面板可能会在已暂存和未暂存视图中显示“同一个文件两次”。这是正常的 Git 行为。
