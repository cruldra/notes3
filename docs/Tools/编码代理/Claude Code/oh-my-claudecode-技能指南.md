> oh-my-claudecode（OMC）是 Claude Code 的多代理编排增强插件。本文介绍所有内置技能的作用与最佳使用场景。
>
> 调用方式：在对话中直接输入关键词，或使用 `/oh-my-claudecode:<技能名>` 命令。

## 一、执行模式技能

### `autopilot` - 全自动执行

**触发词**：autopilot、build me、I want a

**作用**：从想法到可运行代码的全自动流程。Claude 会自动规划、执行、验证，中间不打断你。

**最佳场景**：
- 你有一个清晰的功能需求，不想参与中间过程
- 构建一个新的独立模块或服务
- 快速原型开发

**示例**：
```
autopilot：帮我实现一个用户权限管理模块，支持 RBAC
```

---

### `ralph` - 持久执行直到完成

**触发词**：ralph、don't stop、must complete

**作用**：自我强化循环模式。任务失败不会停下，会持续分析、调整、重试，直到任务被验证完成为止。**包含 ultrawork 的并行能力**。

**最佳场景**：
- 复杂的多步骤重构任务
- 需要修复多个相关 bug
- 你明确要求"不要停，做完为止"的任务
- 涉及多文件改动的功能开发

**示例**：
```
ralph：把所有 any 类型替换为具体类型，修完所有 TypeScript 错误
```

---

### `ultrawork` (ulw) - 最大并行执行

**触发词**：ulw、ultrawork

**作用**：启动最大并行度的多代理执行引擎。将任务拆分给多个子代理同时处理，大幅提升吞吐量。

**最佳场景**：
- 大量独立的、可并行处理的子任务
- 批量重构（如统一命名规范、添加注释）
- 多模块同步开发

**示例**：
```
ulw：给所有 service 文件补充单元测试
```

> **注意**：`ralph` 已包含 `ultrawork`，不需要组合使用。

---

### `ultraqa` - QA 循环直到通过

**作用**：测试 → 发现问题 → 修复 → 再测试，循环直到所有质量标准满足为止。通常由 `autopilot` 自动激活。

**最佳场景**：
- 功能开发完成后的质量保障阶段
- CI 流水线中存在失败需要持续修复
- 需要达到特定测试覆盖率目标

## 二、规划与分析技能

### `plan` - 标准规划

**触发词**：plan、plan the、plan this

**作用**：制定详细实施计划，等待你审批后再执行。单个规划代理完成分析和方案设计。

**最佳场景**：
- 普通功能开发前的规划
- 需求明确、想看执行方案
- 改动范围适中（5 个文件以内）

**示例**：
```
plan：新增订单导出 Excel 功能
```

---

### `ralplan` - 共识规划（多代理辩论）

**触发词**：ralplan、consensus plan

**作用**：Planner（规划师）+ Architect（架构师）+ Critic（批评者）三个代理反复讨论、互相挑战，直到达成共识后才输出最终方案。

**最佳场景**：
- 架构级别的重要决策
- 有多种技术路线可选、需要权衡
- 高风险改动（核心业务逻辑、数据库结构）
- 你自己也没想清楚哪个方案更好

**示例**：
```
ralplan：重新设计消息队列的消费者架构，当前的方案存在积压问题
```

> **加 `--deliberate` 参数**：开启更严格的预演模式，增加故障预判和测试策略规划，适合生产级高风险改动。

---

### `deep-interview` - 苏格拉底式需求澄清

**作用**：在执行任何任务之前，通过反复追问澄清需求。内部用数学化的"歧义度"评分控制何时停止提问，只有当需求足够清晰时才开始执行。

**最佳场景**：
- 你脑子里还没完全想清楚要做什么
- 需求描述模糊，怕方向跑偏
- 重要功能，不能走弯路
- 和产品/业务对接后，需要转化为技术方案

**示例**：
```
/oh-my-claudecode:deep-interview
我想改进一下用户通知系统
```

---

### `analyze` - 调试与根因分析

**触发词**：analyze、debug、investigate

**作用**：调用 `debugger` 专业代理进行深度根因分析、回归隔离、堆栈追踪分析。

**最佳场景**：
- 线上 bug 排查
- 性能问题分析
- 复杂异常的堆栈追踪
- "为什么这个测试偶尔失败"类问题

**示例**：
```
analyze：WebSocket 连接在高并发下会随机断开，帮我找原因
```

---

### `sciomc` - 并行科学家代理分析

**触发词**：sciomc

**作用**：启动多个"科学家"代理并行进行综合分析，适合需要多角度审视的复杂研究性问题。支持 AUTO 模式自动决定代理数量。

**最佳场景**：
- 技术选型评估（比较多个方案）
- 性能基准测试分析
- 数据分析与统计研究
- 需要从多个维度同时分析同一问题

## 三、代码质量技能

### `code-review` - 综合代码审查

**触发词**：review code

**作用**：调用 `code-reviewer`（Opus 级别）进行全面代码审查，覆盖逻辑缺陷、API 合约、向后兼容性、可维护性。

**最佳场景**：
- PR 合并前的质量把关
- 重要功能开发完成后
- 引入新模式或依赖后

**示例**：
```
review code：审查刚完成的支付模块
```

---

### `security-review` - 安全审查

**触发词**：security review

**作用**：专注于安全漏洞检测，覆盖 OWASP Top 10、信任边界、认证授权、secrets 泄露等安全问题。

**最佳场景**：
- 涉及用户认证、权限控制的代码
- 处理外部输入（API 接口、文件上传）的代码
- 支付、财务相关功能
- 上线前的安全 checklist

**示例**：
```
security review：检查用户登录和 Token 刷新的相关代码
```

---

### `tdd` - 测试驱动开发

**触发词**：tdd、test first、red green

**作用**：强制执行"先写测试再写实现"的 TDD 流程。调用 `test-engineer` 代理规划测试策略，再写代码让测试通过。

**最佳场景**：
- 关键业务逻辑开发
- 需要高测试覆盖率的核心模块
- 你想培养 TDD 习惯

---

### `build-fix` - 构建错误修复

**触发词**：fix build、type errors

**作用**：专门处理构建失败和 TypeScript 类型错误，最小化改动范围，不改变架构。

**最佳场景**：
- `npm run build` 报错
- TypeScript 编译失败
- 升级依赖后出现类型不兼容
- CI 构建失败

**示例**：
```
build-fix：修复所有 TypeScript 类型错误
```

---

### `simplify` - 代码简化与优化

**作用**：审查最近修改的代码，寻找可复用的现有工具、消除冗余、提升清晰度和一致性，同时保留所有功能。

**最佳场景**：
- 功能实现后的代码整理
- 发现代码有重复逻辑时
- 想提升代码可读性

---

## 四、团队与并行技能

### `team` - N 个协调代理团队

**触发词**：team、coordinated team、team ralph

**作用**：创建 N 个协调工作的 Claude 代理，共享任务列表，按阶段路由（规划→执行→验证→修复）。`team ralph` 模式持续执行直到完成。

**最佳场景**：
- 大型功能开发（前后端同步）
- 需要多专业角色协作的复杂任务
- 想要最大化并行度的长期任务

**示例**：
```
team：实现完整的订单管理系统，包括前端页面、后端 API 和数据库设计
```

---

### `omc-teams` - tmux 多窗口 CLI 工作者

**触发词**：omc-teams、codex、gemini（单独使用时）

**作用**：在 tmux 窗格中启动 Claude/Codex/Gemini CLI 工作者，在后台并行处理任务。

**最佳场景**：
- 需要同时使用不同 AI 模型
- 长时间运行的后台任务
- 不想占用当前对话窗口

---

### `ccg` - Claude + Codex + Gemini 三模型编排

**触发词**：ccg、tri-model、claude codex gemini

**作用**：后端任务分发给 Codex，前端/UI 任务分发给 Gemini，Claude 综合两者结果。充分利用不同模型的专长。

**最佳场景**：
- 全栈功能开发
- 需要同时处理前后端的任务
- 想利用各模型特长

---

### `external-context` - 并行外部文档查询

**作用**：启动多个 `document-specialist` 代理并行搜索外部文档和网络资源，快速获取技术参考资料。

**最佳场景**：
- 需要查阅多个库的官方文档
- 调研某个技术领域的最佳实践
- 使用不熟悉的 SDK 或 API 前

---

## 五、配置与工具技能

### `omc-setup` - 初始化与配置

**作用**：OMC 的一键安装和配置向导。首次使用必须运行。

```
/oh-my-claudecode:omc-setup
```

---

### `mcp-setup` - MCP 服务器配置

**作用**：为 Claude Code 配置 MCP（Model Context Protocol）服务器，扩展工具能力（浏览器、数据库、文件系统等）。

**最佳场景**：想要让 Claude 直接操作浏览器、连接数据库、访问特定工具时。

---

### `configure-notifications` - 通知集成配置

**触发词**：configure discord、setup telegram、configure slack

**作用**：配置 Telegram、Discord、Slack 等通知集成，让长时间运行的任务完成后通知你。

**最佳场景**：运行 ralph 等长任务时，不用一直盯着终端。

---

### `configure-openclaw` - OpenClaw 网关配置

**作用**：配置 OpenClaw 网关集成，用于在 hook 事件触发时唤醒外部自动化流程和 AI 代理。

---

### `hud` - HUD 显示配置

**作用**：配置界面抬头显示（HUD）的布局、预设和显示元素。

---

### `keybindings-help` - 快捷键配置

**作用**：帮助自定义键盘快捷键，包括重新绑定、和弦快捷键、修改提交键等。修改 `~/.claude/keybindings.json`。

**示例**：
```
帮我把提交键改成 Ctrl+Enter
```

---

### `project-session-manager` - 项目会话管理

**作用**：使用 git worktree 和 tmux 会话管理隔离的开发环境，适合同时处理多个分支或任务。

---

## 六、辅助与记忆技能

### `note` - 记录笔记

**触发词**：note

**作用**：将重要信息保存到 `notepad.md`，在上下文压缩后仍然保留，防止关键信息丢失。

**最佳场景**：
- 长对话中发现重要结论需要保留
- 记录调试过程中的关键发现
- 保存当前任务的上下文状态

---

### `deep-interview` 配合 `writer-memory` - 写作记忆系统

**作用**：为写作项目提供持久化的角色、关系、场景、主题追踪系统。

**最佳场景**：长篇写作项目，需要跨会话保持人物和情节一致性。

---

### `learner` - 从对话中提取技能

**作用**：从当前对话中学习并提取新技能，保存为可复用的 OMC 技能。

**最佳场景**：当你在对话中建立了一套有价值的工作流程，想把它固化为技能时。

---

### `learn-about-omc` - OMC 使用分析

**作用**：分析你的 OMC 使用模式，识别哪些功能被忽略了，给出个性化的使用建议。

---

### `trace` - 代理流程追踪

**触发词**：trace

**作用**：显示代理执行的时间线和统计信息，包括 hook、技能激活、工具调用等事件。

**最佳场景**：想了解 OMC 在后台做了什么、性能瓶颈在哪里。

---

### `deepinit` - 深度代码库初始化

**触发词**：deepinit

**作用**：对代码库进行深度分析，生成分层次的 `AGENTS.md` 文档，帮助所有代理快速理解项目结构。

**最佳场景**：
- 加入新项目时
- 代码库结构发生重大变化后
- 想让 OMC 更准确地理解你的项目

---

### `ralph-init` - 初始化 PRD

**作用**：为 ralph 循环创建结构化的 PRD（产品需求文档），使持久执行更有方向性。

**最佳场景**：在启动复杂的 ralph 任务前，先定义清楚验收标准。

---

### `skill` - 技能管理

**作用**：管理本地技能，包括列出、添加、删除、搜索、编辑技能，以及运行设置向导。

---

### `omc-help` - 使用帮助

**作用**：显示 OMC 的完整使用指南。

---

### `omc-doctor` - 诊断与修复

**作用**：诊断并修复 OMC 安装问题。遇到 OMC 不正常时首先运行此命令。

---

### `cancel` - 取消当前模式

**触发词**：stop、cancel、abort

**作用**：智能停止当前运行的执行模式（autopilot、ralph、ultrawork 等）。

> `--force` 参数：强制清除所有 OMC 状态。

---

### `release` - 发布工作流

**作用**：oh-my-claudecode 自身的自动化发布工作流（OMC 开发者使用）。

---

### `notebooklm` - Google NotebookLM 查询

**作用**：直接从 Claude Code 查询 Google NotebookLM 笔记本，获得有文档引用支撑的回答，大幅减少幻觉。

---

### `claude-developer-platform` - Claude API 开发

**触发词**：代码中包含 `anthropic`/`@anthropic-ai/sdk`

**作用**：当你开发基于 Claude API 的应用时自动激活，提供 Anthropic SDK 使用指导和最佳实践。

## 七、快速选择指南

```
需求模糊，没想清楚
  → deep-interview（先澄清需求）

需求清晰，中等任务
  → plan → 审批 → 执行

需求清晰，重要架构决策
  → ralplan（三方辩论达成共识）

需求清晰，直接干
  → autopilot 或 ralph（不停直到完成）

大量并行任务
  → ultrawork / team

代码质量把关
  → code-review（综合）/ security-review（安全）/ tdd（测试先行）

构建/类型错误
  → build-fix

Bug 排查
  → analyze

典型完整流程：
  deep-interview → ralplan → ralph
  （澄清需求）→（共识方案）→（持续执行）
```
