# 创建插件

> 创建自定义插件，用技能、代理、hooks 和 MCP 服务器扩展 Claude Code。

插件让你可以用可在项目和团队间共享的自定义功能扩展 Claude Code。本指南涵盖使用技能、代理、hooks 和 MCP 服务器创建你自己的插件。

想要安装现有插件？见[发现和安装插件](/en/discover-plugins)。有关完整的技术规格，见[插件参考](/en/plugins-reference)。

---

## 何时使用插件 vs 独立配置

Claude Code 支持两种方式来添加自定义技能、代理和 hooks：

| 方式                                                | 技能名称           | 适用于                                                                                        |
| :-------------------------------------------------- | :----------------- | :-------------------------------------------------------------------------------------------- |
| **独立**（`.claude/` 目录）                         | `/hello`           | 个人工作流、项目特定定制、快速实验                                                            |
| **插件**（带 `.claude-plugin/plugin.json` 的目录）  | `/plugin-name:hello` | 与队友分享、分发给社区、版本化发布、跨项目复用                                                |

**使用独立配置当**：

- 你正在为单个项目定制 Claude Code
- 配置是个人的，不需要分享
- 你在打包之前正在实验技能或 hooks
- 你想要简短的技能名称如 `/hello` 或 `/deploy`

**使用插件当**：

- 你想要与团队或社区分享功能
- 你需要在多个项目中使用相同的技能/代理
- 你想要为你的扩展进行版本控制和轻松更新
- 你通过市场分发
- 你可以接受带命名空间的技能如 `/my-plugin:hello`（命名空间防止插件之间的冲突）

> **提示**：从 `.claude/` 中的独立配置开始进行快速迭代，然后当你准备分享时[转换为插件](#将现有配置转换为插件)。

## 快速入门

这个快速入门指导你创建一个带自定义技能的插件。你将创建一个清单（定义你的插件的配置文件）、添加一个技能，并使用 `--plugin-dir` 标志在本地测试它。

### 前提条件

- Claude Code [已安装和认证](/en/quickstart#step-1-install-claude-code)

> **注意**：如果你看不到 `/plugin` 命令，将 Claude Code 更新到最新版本。见[故障排除](/en/troubleshooting)获取升级说明。

### 创建第一个插件

**步骤 1：创建插件目录**

每个插件都生活在自己的目录中，包含一个清单和你的技能、代理或 hooks。现在创建一个：

```bash
mkdir my-first-plugin
```

**步骤 2：创建插件清单**

`.claude-plugin/plugin.json` 中的清单文件定义了插件的身份：名称、描述和版本。Claude Code 使用此元数据在插件管理器中显示你的插件。

在插件文件夹中创建 `.claude-plugin` 目录：

```bash
mkdir my-first-plugin/.claude-plugin
```

然后创建 `my-first-plugin/.claude-plugin/plugin.json`，内容如下：

```json
{
  "name": "my-first-plugin",
  "description": "A greeting plugin to learn the basics",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  }
}
```

| 字段          | 用途                                                                                              |
| :------------ | :------------------------------------------------------------------------------------------------ |
| `name`        | 唯一标识符和技能命名空间。技能以此作为前缀（例如 `/my-first-plugin:hello`）。                      |
| `description` | 在插件管理器中浏览或安装插件时显示。                                                               |
| `version`     | 使用[语义化版本](/en/plugins-reference#version-management)跟踪发布。                               |
| `author`      | 可选。有助于归属。                                                                                 |

有关 `homepage`、`repository` 和 `license` 等额外字段，见[完整清单 schema](/en/plugins-reference#plugin-manifest-schema)。

**步骤 3：添加技能**

技能生活在 `skills/` 目录中。每个技能是一个包含 `SKILL.md` 文件的文件夹。文件夹名称成为技能名称，带有插件的命名空间前缀（名为 `my-first-plugin` 的插件中的 `hello/` 创建 `/my-first-plugin:hello`）。

在插件文件夹中创建技能目录：

```bash
mkdir -p my-first-plugin/skills/hello
```

然后创建 `my-first-plugin/skills/hello/SKILL.md`，内容如下：

```markdown
---
description: Greet the user with a friendly message
disable-model-invocation: true
---

Greet the user warmly and ask how you can help them today.
```

**步骤 4：测试你的插件**

运行带 `--plugin-dir` 标志的 Claude Code 来加载你的插件：

```bash
claude --plugin-dir ./my-first-plugin
```

Claude Code 启动后，尝试你的新技能：

```shell
/my-first-plugin:hello
```

你会看到 Claude 响应问候。运行 `/help` 查看你的技能在插件命名空间下列出。

> **注意**：**为什么需要命名空间？** 插件技能始终带命名空间（如 `/my-first-plugin:hello`）以防止多个插件有同名技能时的冲突。
>
> 要更改命名空间前缀，更新 `plugin.json` 中的 `name` 字段。

**步骤 5：添加技能参数**

让你的技能动态化，接受用户输入。`$ARGUMENTS` 占位符捕获用户在技能名称后提供的任何文本。

更新你的 `SKILL.md` 文件：

```markdown
---
description: Greet the user with a personalized message
---

# Hello Skill

Greet the user named "$ARGUMENTS" warmly and ask how you can help them today. Make the greeting personal and encouraging.
```

运行 `/reload-plugins` 获取更改，然后用你的名字尝试技能：

```shell
/my-first-plugin:hello Alex
```

Claude 会以你的名字问候你。有关向技能传递参数的更多信息，见[技能](/en/skills#pass-arguments-to-skills)。

你已成功创建并测试了一个包含以下关键组件的插件：

- **插件清单**（`.claude-plugin/plugin.json`）：描述你的插件元数据
- **技能目录**（`skills/`）：包含你的自定义技能
- **技能参数**（`$ARGUMENTS`）：捕获用户输入以实现动态行为

> **提示**：`--plugin-dir` 标志适用于开发和测试。当你准备与他人分享你的插件时，见[创建和分发插件市场](/en/plugin-marketplaces)。

## 插件结构概述

你创建了一个带技能的插件，但插件可以包含更多内容：自定义代理、hooks、MCP 服务器和 LSP 服务器。

> **警告**：**常见错误**：不要将 `commands/`、`agents/`、`skills/` 或 `hooks/` 放在 `.claude-plugin/` 目录内。只有 `plugin.json` 放在 `.claude-plugin/` 内。所有其他目录必须在插件根级别。

| 目录              | 位置       | 用途                                                                           |
| :---------------- | :--------- | :----------------------------------------------------------------------------- |
| `.claude-plugin/` | 插件根目录 | 包含 `plugin.json` 清单（如果组件使用默认位置则是可选的）                       |
| `skills/`         | 插件根目录 | 技能作为 `<name>/SKILL.md` 目录                                                |
| `commands/`       | 插件根目录 | 扁平 Markdown 文件的技能。新插件使用 `skills/`                                 |
| `agents/`         | 插件根目录 | 自定义代理定义                                                                 |
| `hooks/`          | 插件根目录 | `hooks.json` 中的事件处理器                                                    |
| `.mcp.json`       | 插件根目录 | MCP 服务器配置                                                                 |
| `.lsp.json`       | 插件根目录 | 代码智能的 LSP 服务器配置                                                      |
| `bin/`            | 插件根目录 | 插件启用时添加到 Bash 工具 `PATH` 的可执行文件                                  |
| `settings.json`   | 插件根目录 | 插件启用时应用的默认[设置](/en/settings)                                        |

> **注意**：**下一步**：准备添加更多功能？跳到[开发更复杂的插件](#开发更复杂的插件)添加代理、hooks、MCP 服务器和 LSP 服务器。有关所有插件组件的完整技术规格，见[插件参考](/en/plugins-reference)。

## 开发更复杂的插件

一旦你熟悉了基本插件，你可以创建更复杂的扩展。

### 向插件添加技能

插件可以包含 [Agent 技能](/en/skills) 来扩展 Claude 的功能。技能由模型调用：Claude 根据任务上下文自动使用它们。

在插件根目录添加 `skills/` 目录，包含带 `SKILL.md` 文件的技能文件夹：

```text
my-plugin/
├── .claude-plugin/
│   └── plugin.json
└── skills/
    └── code-review/
        └── SKILL.md
```

每个 `SKILL.md` 需要带 `name` 和 `description` 字段的 frontmatter，后跟指令：

```yaml
---
name: code-review
description: Reviews code for best practices and potential issues. Use when reviewing code, checking PRs, or analyzing code quality.
---

When reviewing code, check for:
1. Code organization and structure
2. Error handling
3. Security concerns
4. Test coverage
```

安装插件后，运行 `/reload-plugins` 加载技能。有关技能编写的完整指南，包括渐进式披露和工具限制，见[Agent 技能](/en/skills)。

### 向插件添加 LSP 服务器

> **提示**：对于 TypeScript、Python 和 Rust 等常见语言，从官方市场安装预构建的 LSP 插件。仅在需要支持尚未覆盖的语言时创建自定义 LSP 插件。

LSP（语言服务器协议）插件给 Claude 实时代码智能。如果你需要支持没有官方 LSP 插件的语言，你可以通过向插件添加 `.lsp.json` 文件来创建自己的：

```json
{
  "go": {
    "command": "gopls",
    "args": ["serve"],
    "extensionToLanguage": {
      ".go": "go"
    }
  }
}
```

安装你插件的用户必须在其机器上安装了语言服务器二进制文件。

有关完整的 LSP 配置选项，见[LSP 服务器](/en/plugins-reference#lsp-servers)。

### 随插件附带默认设置

插件可以在插件根目录包含 `settings.json` 文件，以便在插件启用时应用默认配置。目前，仅支持 `agent` 键。

设置 `agent` 会激活插件的[自定义代理](/en/sub-agents)之一作为主线程，应用其系统提示、工具限制和模型。这让插件在启用时改变 Claude Code 的默认行为方式。

```json
{
  "agent": "security-reviewer"
}
```

这个示例激活插件 `agents/` 目录中定义的 `security-reviewer` 代理。`settings.json` 中的设置优先于 `plugin.json` 中声明的 `settings`。未知的键会被静默忽略。

### 组织复杂插件

对于有很多组件的插件，按功能组织你的目录结构。有关完整的目录布局和组织模式，见[插件目录结构](/en/plugins-reference#plugin-directory-structure)。

### 在本地测试你的插件

在开发期间使用 `--plugin-dir` 标志测试插件。这会直接加载你的插件，无需安装。

```bash
claude --plugin-dir ./my-plugin
```

当 `--plugin-dir` 插件与已安装的市场插件同名时，本地副本在该会话中优先。这让你可以测试已安装的插件的更改，而无需先卸载它。Managed 设置强制启用的市场插件是唯一的例外，不能被覆盖。

当你对插件进行更改时，运行 `/reload-plugins` 获取更新而无需重启。这会重新加载插件、技能、代理、hooks、插件 MCP 服务器和插件 LSP 服务器。测试你的插件组件：

- 用 `/plugin-name:skill-name` 尝试你的技能
- 检查代理是否出现在 `/agents` 中
- 验证 hooks 是否按预期工作

> **提示**：你可以通过多次指定标志来一次加载多个插件：
>
> ```bash
> claude --plugin-dir ./plugin-one --plugin-dir ./plugin-two
> ```

### 调试插件问题

如果你的插件没有按预期工作：

1. **检查结构**：确保你的目录在插件根目录，而不是在 `.claude-plugin/` 内
2. **单独测试组件**：分别检查每个技能、代理和 hook
3. **使用验证和调试工具**：见[调试和开发工具](/en/plugins-reference#debugging-and-development-tools)获取 CLI 命令和故障排除技术

### 分享你的插件

当你的插件准备好分享时：

1. **添加文档**：包含带安装和使用说明的 `README.md`
2. **版本化你的插件**：在 `plugin.json` 中使用[语义化版本](/en/plugins-reference#version-management)
3. **创建或使用市场**：通过[插件市场](/en/plugin-marketplaces)分发以供安装
4. **与他人测试**：在更广泛分发之前让团队成员测试插件

一旦你的插件进入市场，其他人可以使用[发现和安装插件](/en/discover-plugins)中的说明安装它。

### 向官方市场提交你的插件

要向 Anthropic 官方市场提交插件，使用应用内提交表单之一：

- **Claude.ai**：[claude.ai/settings/plugins/submit](https://claude.ai/settings/plugins/submit)
- **Console**：[platform.claude.com/plugins/submit](https://platform.claude.com/plugins/submit)

> **注意**：有关完整的技术规格、调试技巧和分发策略，见[插件参考](/en/plugins-reference)。

## 将现有配置转换为插件

如果你已经在 `.claude/` 目录中有技能或 hooks，你可以将它们转换为插件以便更容易地分享和分发。

### 迁移步骤

**步骤 1：创建插件结构**

创建新的插件目录：

```bash
mkdir -p my-plugin/.claude-plugin
```

在 `my-plugin/.claude-plugin/plugin.json` 创建清单文件：

```json
{
  "name": "my-plugin",
  "description": "Migrated from standalone configuration",
  "version": "1.0.0"
}
```

**步骤 2：复制现有文件**

将现有配置复制到插件目录：

```bash
# 复制 commands
cp -r .claude/commands my-plugin/

# 复制 agents（如果有）
cp -r .claude/agents my-plugin/

# 复制 skills（如果有）
cp -r .claude/skills my-plugin/
```

**步骤 3：迁移 hooks**

如果你在设置中有 hooks，创建 hooks 目录：

```bash
mkdir my-plugin/hooks
```

创建 `my-plugin/hooks/hooks.json` 并填入你的 hooks 配置。从 `.claude/settings.json` 或 `settings.local.json` 复制 `hooks` 对象，因为格式相同。命令通过 stdin 接收 hook 输入作为 JSON，因此使用 `jq` 提取文件路径：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{ "type": "command", "command": "jq -r '.tool_input.file_path' | xargs npm run lint:fix" }]
      }
    ]
  }
}
```

**步骤 4：测试迁移后的插件**

加载你的插件以验证一切正常：

```bash
claude --plugin-dir ./my-plugin
```

测试每个组件：运行你的命令、检查代理是否出现在 `/agents` 中、验证 hooks 是否正确触发。

### 迁移时的变化

| 独立（`.claude/`）              | 插件                                |
| :------------------------------ | :---------------------------------- |
| 仅在一个项目中可用              | 可通过市场分享                      |
| 文件在 `.claude/commands/` 中   | 文件在 `plugin-name/commands/` 中   |
| Hooks 在 `settings.json` 中     | Hooks 在 `hooks/hooks.json` 中      |
| 必须手动复制以分享              | 用 `/plugin install` 安装           |

> **注意**：迁移后，你可以从 `.claude/` 中删除原始文件以避免重复。插件版本在加载时将优先。
