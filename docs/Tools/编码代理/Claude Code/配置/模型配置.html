模型配置

了解 Claude Code 的模型配置，包括 opusplan 等模型别名。

可用模型​

在 Claude Code 的 model 设置中，你可以配置以下任一选项：

  • 模型别名
  • 模型名称
    • Anthropic API：完整的模型名称
    • Bedrock：推理配置文件 ARN
    • Foundry：部署名称
    • Vertex：版本名称

模型别名​

模型别名提供了一种便捷的方式来选择模型设置，而无需记住确切的版本号：

模型别名行为
default特殊值，清除任何模型覆盖并恢复到你账户类型推荐的模型。其本身不是模型别名
best使用可用的最强模型，目前等同于 opus
sonnet使用最新的 Sonnet 模型（目前为 Sonnet 4.6）进行日常编码任务
opus使用最新的 Opus 模型（目前为 Opus 4.6）进行复杂推理任务
haiku使用快速高效的 Haiku 模型进行简单任务
sonnet[1m]使用带 100 万 token 上下文窗口 的 Sonnet，适合长会话
opus[1m]使用带 100 万 token 上下文窗口 的 Opus，适合长会话
opusplan特殊模式，在计划模式中使用 opus，执行时自动切换到 sonnet

别名始终指向最新版本。要锁定到特定版本，使用完整模型名称（例如 claude-opus-4-6）或设置对应的环境变量如 ANTHROPIC_DEFAULT_OPUS_MODEL。

设置模型​

你可以通过以下方式配置模型，按优先级排序：

  1. 会话中 — 使用 /model <别名|名称> 在会话中切换模型
  2. 启动时 — 使用 claude --model <别名|名称> 启动
  3. 环境变量 — 设置 ANTHROPIC_MODEL=<别名|名称>
  4. 设置文件 — 在设置文件中使用 model 字段永久配置

使用示例：

# 以 Opus 启动
claude --model opus

# 在会话中切换到 Sonnet
/model sonnet

设置文件示例：

{
    "permissions": {
        ...
    },
    "model": "opus"
}

限制模型选择​

企业管理员可以在 managed 或策略设置中使用 availableModels 来限制用户可以选择的模型。

设置 availableModels 后，用户无法通过 /model、--model 标志、配置工具或 ANTHROPIC_MODEL 环境变量切换到列表之外的模型。

{
  "availableModels": ["sonnet", "haiku"]
}

Default 模型行为​

模型选择器中的 Default 选项不受 availableModels 影响。它始终可用，代表基于用户订阅层级的系统运行时默认模型。

即使设置了 availableModels: []，用户仍然可以使用其订阅层级的 Default 模型来使用 Claude Code。

控制用户运行的模型​

model 设置是初始选择，不是强制执行。它设置会话启动时激活的模型，但用户仍然可以打开 /model 并选择 Default，这会解析为其订阅层级的系统默认值，无论 model 设置为什么。

要完全控制模型体验，组合使用三个设置：

  • availableModels：限制用户可以切换到哪些命名的模型
  • model：设置会话启动时的初始模型选择
  • ANTHROPIC_DEFAULT_SONNET_MODEL / ANTHROPIC_DEFAULT_OPUS_MODEL / ANTHROPIC_DEFAULT_HAIKU_MODEL：控制 Default 选项以及 sonnet、opus 和 haiku 别名的解析目标

这个示例让用户从 Sonnet 4.5 开始，将选择器限制为 Sonnet 和 Haiku，并将 Default 锁定为解析到 Sonnet 4.5 而不是最新发布版：

{
  "model": "claude-sonnet-4-5",
  "availableModels": ["claude-sonnet-4-5", "haiku"],
  "env": {
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "claude-sonnet-4-5"
  }
}

如果没有 env 块，在选择器中选择 Default 的用户将获得最新的 Sonnet 版本，绕过 model 和 availableModels 中的版本锁定。

合并行为​

当 availableModels 在多个层级设置时（如 user 设置和 project 设置），数组会被合并和去重。要强制执行严格的白名单，在具有最高优先级的 managed 或策略设置中设置 availableModels。

Mantle 模型 ID​

当启用 Bedrock Mantle 端点 时，availableModels 中以 anthropic. 开头的条目会作为自定义选项添加到 /model 选择器中，并路由到 Mantle 端点。这是第三方部署锁定模型中描述的仅别名匹配规则的例外。该设置仍然将选择器限制为列出的条目，因此请在任何 Mantle ID 旁边包含标准别名。

特殊模型行为​

default 模型设置​

default 的行为取决于你的账户类型：

  • Max 和 Team Premium：默认为 Opus 4.6
  • Pro 和 Team Standard：默认为 Sonnet 4.6
  • Enterprise：Opus 4.6 可用但不是默认

当你达到 Opus 的使用阈值时，Claude Code 可能会自动回退到 Sonnet。

opusplan 模型设置​

opusplan 模型别名提供了一种自动化混合方法：

  • 在计划模式中 — 使用 opus 进行复杂推理和架构决策
  • 在执行模式中 — 自动切换到 sonnet 进行代码生成和实现

这让你兼得两者之长：Opus 的超强推理用于规划，Sonnet 的高效用于执行。

调整努力级别​

努力级别控制自适应推理，它根据任务复杂度动态分配思考。较低的努力级别更快、更便宜，适合直接的任务，而较高的努力级别为复杂问题提供更深入的推理。

三个级别跨会话持久化：low、medium 和 high。第四个级别 max 提供最深的推理，对 token 消耗没有约束，因此响应比 high 更慢、成本更高。max 仅在 Opus 4.6 上可用，不跨会话持久化，除非通过 CLAUDE_CODE_EFFORT_LEVEL 环境变量。

默认努力级别取决于你的计划。Pro 和 Max 订阅者默认为 medium 努力级别。所有其他用户默认为 high 努力级别：API 密钥、Team、Enterprise 和第三方提供商（Bedrock、Vertex AI、Foundry）用户。

你计划的默认值适合大多数编码任务。对于需要更深入推理的工作（如困难的调试问题或复杂的架构决策），请提高努力级别。较高的级别可能导致模型在常规工作上过度思考。

对于一次性深度推理而不改变会话设置，在提示中包含 "ultrathink" 以在该轮触发高努力级别。如果你的会话已经处于 high 或 max，这没有效果。

设置努力级别：

  • /effort：运行 /effort low、/effort medium、/effort high 或 /effort max 更改级别，或 /effort auto 重置为模型默认
  • 在 /model 中：选择模型时使用左右箭头键调整努力滑块
  • --effort 标志：传递 low、medium、high 或 max 在启动 Claude Code 时为单次会话设置级别
  • 环境变量：设置 CLAUDE_CODE_EFFORT_LEVEL 为 low、medium、high、max 或 auto
  • 设置文件：在设置文件中设置 effortLevel 为 "low"、"medium" 或 "high"
  • 技能和子代理 frontmatter：在技能或子代理 Markdown 文件中设置 effort 以在该技能或子代理运行时覆盖努力级别

环境变量优先于所有其他方法，然后是你配置的级别，最后是模型默认。Frontmatter 努力级别在该技能或子代理激活时应用，覆盖会话级别但不覆盖环境变量。

努力级别支持 Opus 4.6 和 Sonnet 4.6。选择支持的模型时，努力滑块会出现在 /model 中。当前努力级别也会显示在 logo 和旋转器旁边，例如 "with low effort"，因此你无需打开 /model 即可确认哪个级别处于激活状态。

要在 Opus 4.6 和 Sonnet 4.6 上禁用自适应推理并恢复之前的固定思考预算，设置 CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1。禁用时，这些模型使用由 MAX_THINKING_TOKENS 控制的固定预算。见环境变量。

扩展上下文​

Opus 4.6 和 Sonnet 4.6 支持 100 万 token 上下文窗口，用于包含大型代码库的长会话。

可用性因模型和计划而异。在 Max、Team 和 Enterprise 计划中，Opus 会自动升级到 100 万上下文，无需额外配置。这适用于 Team Standard 和 Team Premium 席位。

计划带 100 万上下文的 Opus 4.6带 100 万上下文的 Sonnet 4.6
Max、Team 和 Enterprise订阅包含需要额外用量
Pro需要额外用量需要额外用量
API 和按量付费完全访问完全访问

要完全禁用 100 万上下文，设置 CLAUDE_CODE_DISABLE_1M_CONTEXT=1。这会从模型选择器中移除 100 万模型变体。见环境变量。

100 万上下文窗口使用标准模型定价，超过 20 万 token 不收取额外费用。对于扩展上下文包含在订阅中的计划，用量仍然由订阅覆盖。对于通过额外用量访问扩展上下文的计划，token 按额外用量计费。

如果你的账户支持 100 万上下文，该选项会出现在最新版 Claude Code 的模型选择器（/model）中。如果没有看到，请尝试重启会话。

你也可以在模型别名或完整模型名称后使用 [1m] 后缀：

# 使用 opus[1m] 或 sonnet[1m] 别名
/model opus[1m]
/model sonnet[1m]

# 或在完整模型名称后添加 [1m]
/model claude-opus-4-6[1m]

检查当前模型​

你可以通过以下方式查看当前使用的模型：

  1. 在状态行中（如果已配置）
  2. 在 /status 中，还会显示你的账户信息

添加自定义模型选项​

使用 ANTHROPIC_CUSTOM_MODEL_OPTION 向 /model 选择器添加单个自定义条目，而不用替换内置别名。这对于 LLM 网关部署或测试 Claude Code 默认不列出的模型 ID 很有用。

这个示例设置所有三个变量，使网关路由的 Opus 部署可选：

export ANTHROPIC_CUSTOM_MODEL_OPTION="my-gateway/claude-opus-4-6"
export ANTHROPIC_CUSTOM_MODEL_OPTION_NAME="通过网关的 Opus"
export ANTHROPIC_CUSTOM_MODEL_OPTION_DESCRIPTION="通过内部 LLM 网关路由的自定义部署"

自定义条目出现在 /model 选择器底部。ANTHROPIC_CUSTOM_MODEL_OPTION_NAME 和 ANTHROPIC_CUSTOM_MODEL_OPTION_DESCRIPTION 是可选的。如果省略，模型 ID 将用作名称，描述默认为 Custom model (<model-id>)。

Claude Code 跳过对 ANTHROPIC_CUSTOM_MODEL_OPTION 中设置的模型 ID 的验证，因此你可以使用 API 端点接受的任何字符串。

环境变量​

你可以使用以下环境变量来控制别名映射到的模型名称。它们必须是完整的模型名称（或你的 API 提供商的等效名称）。

环境变量描述
ANTHROPIC_DEFAULT_OPUS_MODEL用于 opus 的模型，或计划模式激活时用于 opusplan 的模型
ANTHROPIC_DEFAULT_SONNET_MODEL用于 sonnet 的模型，或计划模式未激活时用于 opusplan 的模型
ANTHROPIC_DEFAULT_HAIKU_MODEL用于 haiku 的模型，或后台功能
CLAUDE_CODE_SUBAGENT_MODEL用于子代理的模型

注意：ANTHROPIC_SMALL_FAST_MODEL 已弃用，请使用 ANTHROPIC_DEFAULT_HAIKU_MODEL。

为第三方部署锁定模型​

当通过 Bedrock、Vertex AI 或 Foundry 部署 Claude Code 时，在向用户推出之前锁定模型版本。

如果不锁定，Claude Code 会使用解析到最新版本的模型别名（sonnet、opus、haiku）。当 Anthropic 发布尚未在用户账户中启用的新模型时，Bedrock 和 Vertex AI 用户会看到通知并在该会话回退到之前的版本，而 Foundry 用户会看到错误，因为 Foundry 没有等效的启动检查。

警告：将三个模型环境变量设置为特定版本 ID 作为初始设置的一部分。这让你可以控制用户何时迁移到新模型。

使用以下环境变量和提供商特定的版本 ID：

提供商示例
Bedrockexport ANTHROPIC_DEFAULT_OPUS_MODEL='us.anthropic.claude-opus-4-6-v1'
Vertex AIexport ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-6'
Foundryexport ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-6'

对 ANTHROPIC_DEFAULT_SONNET_MODEL 和 ANTHROPIC_DEFAULT_HAIKU_MODEL 应用相同的模式。关于所有提供商的当前和旧版模型 ID，见模型概览。要将用户升级到新模型版本，更新这些环境变量并重新部署。

要为锁定的模型启用扩展上下文，在 ANTHROPIC_DEFAULT_OPUS_MODEL 或 ANTHROPIC_DEFAULT_SONNET_MODEL 中的模型 ID 后附加 [1m]：

export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-6[1m]'

[1m] 后缀将 100 万上下文窗口应用于该别名的所有使用，包括 opusplan。Claude Code 在将模型 ID 发送到你的提供商之前会去掉此前缀。仅在底层模型支持 100 万上下文时附加 [1m]，如 Opus 4.6 或 Sonnet 4.6。

注意：使用第三方提供商时，settings.availableModels 白名单仍然适用。过滤匹配的是模型别名（opus、sonnet、haiku），而不是提供商特定的模型 ID。

自定义锁定模型的显示和功能​

当你在第三方提供商上锁定模型时，提供商特定的 ID 会原样出现在 /model 选择器中，Claude Code 可能无法识别模型支持哪些功能。你可以为每个锁定的模型使用配套环境变量来覆盖显示名称并声明功能。

这些变量仅在 Bedrock、Vertex AI 和 Foundry 等第三方提供商上生效。直接使用 Anthropic API 时无效。

环境变量描述
ANTHROPIC_DEFAULT_OPUS_MODEL_NAME/model 选择器中锁定 Opus 模型的显示名称。未设置时默认为模型 ID
ANTHROPIC_DEFAULT_OPUS_MODEL_DESCRIPTION/model 选择器中锁定 Opus 模型的显示描述。未设置时默认为 Custom Opus model
ANTHROPIC_DEFAULT_OPUS_MODEL_SUPPORTED_CAPABILITIES锁定的 Opus 模型支持的功能的逗号分隔列表

相同的 _NAME、_DESCRIPTION 和 _SUPPORTED_CAPABILITIES 后缀也可用于 ANTHROPIC_DEFAULT_SONNET_MODEL 和 ANTHROPIC_DEFAULT_HAIKU_MODEL。

Claude Code 通过将模型 ID 与已知模式匹配来启用努力级别和扩展思考等功能。提供商特定的 ID（如 Bedrock ARN 或自定义部署名称）通常不匹配这些模式，导致支持的功能被禁用。设置 _SUPPORTED_CAPABILITIES 来告诉 Claude Code 模型实际支持哪些功能：

功能值启用功能
effort努力级别和 /effort 命令
max_effortmax 努力级别
thinking扩展思考
adaptive_thinking根据任务复杂度动态分配思考的自适应推理
interleaved_thinking工具调用之间的思考

设置 _SUPPORTED_CAPABILITIES 后，列出的功能被启用，未列出的功能对匹配的锁定模型被禁用。未设置该变量时，Claude Code 回退到基于模型 ID 的内置检测。

这个示例将 Opus 锁定为 Bedrock 自定义模型 ARN，设置友好名称并声明其功能：

export ANTHROPIC_DEFAULT_OPUS_MODEL='arn:aws:bedrock:us-east-1:123456789012:custom-model/abc'
export ANTHROPIC_DEFAULT_OPUS_MODEL_NAME='通过 Bedrock 的 Opus'
export ANTHROPIC_DEFAULT_OPUS_MODEL_DESCRIPTION='通过 Bedrock 自定义端点路由的 Opus 4.6'
export ANTHROPIC_DEFAULT_OPUS_MODEL_SUPPORTED_CAPABILITIES='effort,max_effort,thinking,adaptive_thinking,interleaved_thinking'

按版本覆盖模型 ID​

上面的家族级环境变量为每个家族别名配置一个模型 ID。如果你需要在同一家族内将多个版本映射到不同的提供商 ID，请改用 modelOverrides 设置。

modelOverrides 将单个 Anthropic 模型 ID 映射到 Claude Code 发送到你的提供商 API 的提供商特定字符串。当用户在 /model 选择器中选择映射的模型时，Claude Code 使用你配置的值而不是内置默认值。

这让企业管理员可以将每个模型版本路由到特定的 Bedrock 推理配置文件 ARN、Vertex AI 版本名称或 Foundry 部署名称，用于治理、成本分配或区域路由。

在设置文件中设置 modelOverrides：

{
  "modelOverrides": {
    "claude-opus-4-6": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/opus-prod",
    "claude-opus-4-5-20251101": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/opus-45-prod",
    "claude-sonnet-4-6": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/sonnet-prod"
  }
}

键必须是模型概览中列出的 Anthropic 模型 ID。对于带日期后缀的模型 ID，请精确包含日期后缀。未知的键会被忽略。

覆盖替换了 /model 选择器中每个条目背后的内置模型 ID。在 Bedrock 上，覆盖优先于 Claude Code 在启动时自动发现的任何推理配置文件。你通过 ANTHROPIC_MODEL、--model 或 ANTHROPIC_DEFAULT_*_MODEL 环境变量直接提供的值会按原样传递给提供商，不会被 modelOverrides 转换。

modelOverrides 与 availableModels 协同工作。白名单是根据 Anthropic 模型 ID（而不是覆盖值）评估的，因此即使 Opus 版本被映射到 ARN，availableModels 中的 "opus" 条目仍然匹配。

提示缓存配置​

Claude Code 自动使用提示缓存来优化性能并降低成本。你可以全局或针对特定模型层级禁用提示缓存：

环境变量描述
DISABLE_PROMPT_CACHING设置为 1 以禁用所有模型的提示缓存（优先于每个模型的设置）
DISABLE_PROMPT_CACHING_HAIKU设置为 1 以仅禁用 Haiku 模型的提示缓存
DISABLE_PROMPT_CACHING_SONNET设置为 1 以仅禁用 Sonnet 模型的提示缓存
DISABLE_PROMPT_CACHING_OPUS设置为 1 以仅禁用 Opus 模型的提示缓存

这些环境变量让你精细控制提示缓存行为。全局 DISABLE_PROMPT_CACHING 设置优先于模型特定的设置，允许你在需要时快速禁用所有缓存。每个模型的设置用于选择性控制，例如调试特定模型或使用可能有不同缓存实现的云提供商时。
