创建自定义子代理

在 Claude Code 中创建和使用专门的 AI 子代理，实现特定任务的工作流和改进的上下文管理。

子代理是处理特定类型任务的专业 AI 助手。当一个边缘任务会在主对话中产生大量搜索结果、日志或文件内容（你不会再引用）时，让子代理在自己的上下文中完成这些工作并只返回摘要。当你反复创建相同类型的worker且使用相同的指令时，就该定义一个自定义子代理了。

每个子代理都在自己的上下文窗口中运行，带有自定义系统提示、特定工具访问和独立权限。当 Claude 遇到匹配子代理描述的任务时，它会委托给该子代理，子代理独立工作并返回结果。要查看实际的上下文节省效果，上下文窗口可视化演示了一个子代理在独立窗口中处理研究的会话。

注意：如果你需要多个代理并行工作并相互通信，请改用代理团队。子代理在单个会话内工作；代理团队跨独立会话协调。

子代理帮助你：

  • 保留上下文：让探索和实现不进入你的主对话
  • 执行约束：限制子代理可以使用的工具
  • 跨项目复用配置：通过 user 级别的子代理
  • 专业化行为：为特定领域提供聚焦的系统提示
  • 控制成本：将任务路由到更快、更便宜的模型如 Haiku

Claude 使用每个子代理的描述来决定何时委托任务。创建子代理时，编写清晰的描述让 Claude 知道何时使用它。

Claude Code 包含几个内置子代理如 Explore、Plan 和 general-purpose。你也可以创建自定义子代理来处理特定任务。本页涵盖内置子代理、如何创建自己的子代理、完整配置选项、与子代理工作的模式以及子代理示例。

内置子代理​

Claude Code 包含内置子代理，Claude 会在适当时自动使用。每个子代理都继承父对话的权限，并带有额外的工具限制。

Explore（探索）

为搜索和分析代码库优化的快速只读代理。

  • 模型：Haiku（快速、低延迟）
  • 工具：只读工具（拒绝 Write 和 Edit 工具）
  • 用途：文件发现、代码搜索、代码库探索

当 Claude 需要搜索或理解代码库而不做更改时，会委托给 Explore。这保持了探索结果不出现在你的主对话上下文中。

调用 Explore 时，Claude 指定一个彻底程度级别：quick 用于针对性查找，medium 用于平衡探索，或 very thorough 用于全面分析。

Plan（计划）

在计划模式期间用于在呈现计划之前收集上下文的研究代理。

  • 模型：继承自主对话
  • 工具：只读工具（拒绝 Write 和 Edit 工具）
  • 用途：计划用的代码库研究

当你在计划模式且 Claude 需要理解你的代码库时，它会将研究委托给 Plan 子代理。这防止了无限嵌套（子代理不能生成其他子代理），同时仍然收集必要的上下文。

General-purpose（通用）

用于需要探索和操作两者的复杂多步骤任务的强大代理。

  • 模型：继承自主对话
  • 工具：所有工具
  • 用途：复杂研究、多步骤操作、代码修改

当任务需要探索和修改、复杂推理来解释结果或多个依赖步骤时，Claude 会委托给 general-purpose。

其他

Claude Code 还包含用于特定任务的额外辅助代理。这些通常自动调用，因此你不需要直接使用它们。

代理模型Claude 何时使用它
statusline-setupSonnet当你运行 /statusline 来配置状态行时
Claude Code GuideHaiku当你询问关于 Claude Code 功能的问题时

除了这些内置子代理，你还可以创建带有自定义提示、工具限制、权限模式、hooks 和技能的自定义子代理。以下部分展示如何开始和自定义子代理。

快速入门：创建第一个子代理​

子代理在带 YAML frontmatter 的 Markdown 文件中定义。你可以手动创建或使用 /agents 命令。

这个演练指导你使用 /agents 命令创建一个 user 级别的子代理。该子代理审查代码并提出改进建议。

步骤 1：打开子代理界面

在 Claude Code 中运行：

/agents

步骤 2：选择位置

切换到 Library 选项卡，选择 Create new agent，然后选择 Personal。这会将子代理保存到 ~/.claude/agents/，使其在所有项目中可用。

步骤 3：使用 Claude 生成

选择 Generate with Claude。提示时描述子代理：

A code improvement agent that scans files and suggests improvements
for readability, performance, and best practices. It should explain
each issue, show the current code, and provide an improved version.

Claude 会为你生成标识符、描述和系统提示。

步骤 4：选择工具

对于只读审查者，取消选择除了 Read-only tools 之外的所有内容。如果保留所有工具，子代理将继承主对话可用的所有工具。

步骤 5：选择模型

选择子代理使用的模型。对于此示例代理，选择 Sonnet，它在分析代码模式时平衡了能力和速度。

步骤 6：选择颜色

为子代理选择背景色。这有助于你在 UI 中识别哪个子代理在运行。

步骤 7：配置记忆

选择 User scope 给子代理一个持久记忆目录在 ~/.claude/agent-memory/。子代理使用此目录跨对话积累见解，如代码库模式和重复出现的问题。如果不想让子代理持久化学习，选择 None。

步骤 8：保存并试用

查看配置摘要。按 s 或 Enter 保存，或按 e 保存并在编辑器中编辑文件。子代理立即可用。试用：

Use the code-improver agent to suggest improvements in this project

Claude 会委托给你的新子代理，它会扫描代码库并返回改进建议。

你现在有了一个可以在机器上任何项目中使用的子代理来分析代码库并提出改进建议。

你也可以手动将子代理创建为 Markdown 文件、通过 CLI 标志定义它们，或通过插件分发它们。以下部分涵盖所有配置选项。

配置子代理​

使用 /agents 命令​

/agents 命令打开一个标签页界面来管理子代理。Running 标签页显示活跃的子代理并允许你打开或停止它们。Library 标签页允许你：

  • 查看所有可用子代理（内置、user、project 和插件）
  • 创建带引导设置或 Claude 生成的新子代理
  • 编辑现有子代理配置和工具访问
  • 删除自定义子代理
  • 查看存在重复时哪些子代理处于激活状态

这是创建和管理子代理的推荐方式。对于手动创建或自动化，你也可以直接添加子代理文件。

要在不启动交互会话的情况下从命令行列出所有已配置的子代理，运行 claude agents。这会显示按源分组的代理并指出哪些被更高优先级的定义覆盖。

选择子代理作用域​

子代理是带 YAML frontmatter 的 Markdown 文件。根据作用域将它们存储在不同的位置。当多个子代理共享相同名称时，更高优先级的位置获胜。

位置作用域优先级如何创建
Managed 设置组织范围1（最高）通过 managed 设置 部署
--agents CLI 标志当前会话2启动 Claude Code 时传递 JSON
.claude/agents/当前项目3交互式或手动
~/.claude/agents/你的所有项目4交互式或手动
插件的 agents/ 目录启用插件的位置5（最低）随 插件 安装

Project 子代理（.claude/agents/）适合特定于代码库的子代理。将它们检入版本控制，让你的团队可以协作使用和改进它们。

Project 子代理通过从当前工作目录向上遍历来发现。使用 --add-dir 添加的目录仅授予文件访问权限，不会扫描子代理。要在项目间共享子代理，使用 ~/.claude/agents/ 或插件。

User 子代理（~/.claude/agents/）是在你所有项目中可用的个人子代理。

CLI 定义的子代理在启动 Claude Code 时作为 JSON 传递。它们仅存在于该会话中且不保存到磁盘，适用于快速测试或自动化脚本。你可以在单个 --agents 调用中定义多个子代理：

claude --agents '{
  "code-reviewer": {
    "description": "Expert code reviewer. Use proactively after code changes.",
    "prompt": "You are a senior code reviewer. Focus on code quality, security, and best practices.",
    "tools": ["Read", "Grep", "Glob", "Bash"],
    "model": "sonnet"
  },
  "debugger": {
    "description": "Debugging specialist for errors and test failures.",
    "prompt": "You are an expert debugger. Analyze errors, identify root causes, and provide fixes."
  }
}'

--agents 标志接受与基于文件的子代理相同的 frontmatter 字段：description、prompt、tools、disallowedTools、model、permissionMode、mcpServers、hooks、maxTurns、skills、initialPrompt、memory、effort、background、isolation 和 color。使用 prompt 作为系统提示，等同于基于文件的子代理中的 markdown body。

Managed 子代理由组织管理员部署。将 markdown 文件放在 managed 设置目录中的 .claude/agents/ 内，使用与 project 和 user 子代理相同的 frontmatter 格式。Managed 定义优先于同名的 project 和 user 子代理。

插件子代理来自你已安装的插件。它们与你的自定义子代理一起出现在 /agents 中。有关创建插件子代理的详细信息，见插件组件参考。

注意：出于安全原因，插件子代理不支持 hooks、mcpServers 或 permissionMode frontmatter 字段。从插件加载代理时这些字段被忽略。如果你需要它们，将代理文件复制到 .claude/agents/ 或 ~/.claude/agents/。你也可以在 settings.json 或 settings.local.json 的 permissions.allow 中添加规则，但这些规则适用于整个会话，而不仅是插件子代理。

这些作用域中的子代理定义也可用于代理团队：生成队友时，你可以引用子代理类型，队友使用其 tools 和 model，定义的 body 作为附加指令附加到队友的系统提示中。见代理团队了解该路径上适用的 frontmatter 字段。

编写子代理文件​

子代理文件使用 YAML frontmatter 进行配置，后面是 Markdown 格式的系统提示：

注意：子代理在会话启动时加载。如果你通过手动添加文件创建子代理，请重启会话或使用 /agents 立即加载它。

---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: sonnet
---

You are a code reviewer. When invoked, analyze the code and provide
specific, actionable feedback on quality, security, and best practices.

Frontmatter 定义了子代理的元数据和配置。Body 成为指导子代理行为的系统提示。子代理只接收此系统提示（加上基本环境详情如工作目录），而不是完整的 Claude Code 系统提示。

子代理在主对话的当前工作目录中启动。在子代理中，cd 命令在 Bash 或 PowerShell 工具调用之间不会持久化，也不会影响主对话的工作目录。要给子代理一个隔离的仓库副本，改为设置 isolation: worktree。

支持的 frontmatter 字段​

以下字段可用于 YAML frontmatter。只有 name 和 description 是必需的。

字段必需描述
name是唯一标识符，使用小写字母和连字符
description是Claude 何时委托给此子代理
tools否子代理可以使用的工具。如果省略则继承所有工具
disallowedTools否要拒绝的工具，从继承或指定的列表中移除
model否使用的模型：sonnet、opus、haiku、完整模型 ID（例如 claude-opus-4-6）或 inherit。默认为 inherit
permissionMode否权限模式：default、acceptEdits、auto、dontAsk、bypassPermissions 或 plan
maxTurns否子代理停止前的最大代理轮次数量
skills否在启动时加载到子代理上下文中的技能。注入完整的技能内容，而不仅是使其可调用。子代理不从父对话继承技能
mcpServers否此子代理可用的 MCP 服务器。每个条目要么是引用已配置服务器的服务器名称（如 "slack"），要么是内联定义，以服务器名称为键，完整的 MCP 服务器配置 为值
hooks否作用域到此子代理的[生命周期 hooks](#为子代理定义 hooks)
memory否持久记忆作用域：user、project 或 local。启用跨对话学习
background否设置为 true 始终将此子代理作为后台任务运行。默认：false
effort否此子代理激活时的努力级别。覆盖会话努力级别。默认：继承自会话。选项：low、medium、high、max（仅 Opus 4.6）
isolation否设置为 worktree 在临时 git worktree 中运行子代理，给它一个隔离的仓库副本。如果子代理未做任何更改，worktree 会自动清理
color否任务列表和转录中子代理的显示颜色。接受 red、blue、green、yellow、purple、orange、pink 或 cyan
initialPrompt否当此代理作为主会话代理运行时（通过 --agent 或 agent 设置），自动作为第一个用户轮次提交。命令和技能会被处理。 prepend 到任何用户提供的提示前

选择模型​

model 字段控制子代理使用的 AI 模型：

  • 模型别名：使用可用别名之一：sonnet、opus 或 haiku
  • 完整模型 ID：使用完整模型 ID 如 claude-opus-4-6 或 claude-sonnet-4-6。接受与 --model 标志相同的值
  • inherit：使用与主对话相同的模型
  • 省略：如果未指定，默认为 inherit（使用与主对话相同的模型）

当 Claude 调用子代理时，它还可以为该特定调用传递 model 参数。Claude Code 按以下顺序解析子代理的模型：

  1. CLAUDE_CODE_SUBAGENT_MODEL 环境变量（如果已设置）
  2. 每次调用的 model 参数
  3. 子代理定义的 model frontmatter
  4. 主对话的模型

控制子代理能力​

你可以通过工具访问、权限模式和条件规则来控制子代理能做什么。

可用工具​

子代理可以使用 Claude Code 的任何内部工具。默认情况下，子代理继承主对话的所有工具，包括 MCP 工具。

要限制工具，使用 tools 字段（白名单）或 disallowedTools 字段（黑名单）。这个示例使用 tools 仅允许 Read、Grep、Glob 和 Bash。子代理不能编辑文件、写入文件或使用任何 MCP 工具：

---
name: safe-researcher
description: Research agent with restricted capabilities
tools: Read, Grep, Glob, Bash
---

这个示例使用 disallowedTools 继承主对话的每个工具除了 Write 和 Edit。子代理保留 Bash、MCP 工具和所有其他工具：

---
name: no-writes
description: Inherits every tool except file writes
disallowedTools: Write, Edit
---

如果两者都设置，先应用 disallowedTools，然后 tools 针对剩余池解析。两者中都列出的工具会被移除。

限制可以生成的子代理​

当代理作为主线程以 claude --agent 运行时，它可以使用 Agent 工具生成子代理。要限制它可以生成哪些子代理类型，在 tools 字段中使用 Agent(agent_type) 语法。

在 2.1.63 版本中，Task 工具重命名为 Agent。设置和代理定义中的现有 Task(...) 引用仍然作为别名工作。

---
name: coordinator
description: Coordinates work across specialized agents
tools: Agent(worker, researcher), Read, Bash
---

这是白名单：只能生成 worker 和 researcher 子代理。如果代理尝试生成任何其他类型，请求会失败，代理在其提示中只看到允许的类型。要在允许所有其他代理的同时阻止特定代理，改用 permissions.deny。

要允许生成任何子代理而不受限制，使用不带括号的 Agent：

tools: Agent, Read, Bash

如果 Agent 完全从 tools 列表中省略，代理不能生成任何子代理。此限制仅适用于以 claude --agent 作为主线程运行的代理。子代理不能生成其他子代理，因此 Agent(agent_type) 在子代理定义中不起作用。

为子代理限定 MCP 服务器范围​

使用 mcpServers 字段让子代理访问主对话中不可用的 MCP 服务器。此处定义的内联服务器在子代理启动时连接，完成时断开。字符串引用共享父会话的连接。

列表中的每个条目要么是内联服务器定义，要么是引用会话中已配置的 MCP 服务器的字符串：

---
name: browser-tester
description: Tests features in a real browser using Playwright
mcpServers:
  # 内联定义：仅限此子代理
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@playwright/mcp@latest"]
  # 按名称引用：复用已配置的服务器
  - github
---

Use the Playwright tools to navigate, screenshot, and interact with pages.

内联定义使用与 .mcp.json 服务器条目相同的 schema（stdio、http、sse、ws），以服务器名称为键。

要将 MCP 服务器完全排除在主对话之外并避免其工具描述在那里消耗上下文，在此处内联定义它而不是在 .mcp.json 中。子代理获得工具；父对话没有。

权限模式​

permissionMode 字段控制子代理如何处理权限提示。子代理从主对话继承权限上下文，可以覆盖模式，除非父模式如下所述优先。

模式行为
default标准权限检查，带提示
acceptEdits自动接受工作目录或 additionalDirectories 中路径的文件修改和常见文件系统命令
auto自动模式：后台分类器审查命令和受保护目录写入
dontAsk自动拒绝权限提示（明确允许的工具仍然有效）
bypassPermissions跳过权限提示
plan计划模式（只读探索）

警告：谨慎使用 bypassPermissions。它跳过权限提示，允许子代理未经批准执行操作。写入 .git、.claude、.vscode、.idea 和 .husky 目录仍然需要确认提示，除了 .claude/commands、.claude/agents 和 .claude/skills。详见权限模式。

如果父对话使用 bypassPermissions，这优先且不能被覆盖。如果父对话使用自动模式，子代理继承自动模式，其 frontmatter 中的任何 permissionMode 被忽略：分类器使用与父会话相同的阻止和允许规则评估子代理的工具调用。

预加载技能到子代理​

使用 skills 字段在启动时将技能内容注入子代理的上下文。这让子代理获得领域知识，而无需在执行期间发现和加载技能。

---
name: api-developer
description: Implement API endpoints following team conventions
skills:
  - api-conventions
  - error-handling-patterns
---

Implement API endpoints. Follow the conventions and patterns from the preloaded skills.

每个技能的完整内容被注入到子代理的上下文中，而不仅是使其可调用。子代理不从父对话继承技能；你必须显式列出它们。

注意：这与在子代理中运行技能相反。在子代理中使用 skills 时，子代理控制系统提示并加载技能内容。在技能中使用 context: fork 时，技能内容被注入到你指定的代理中。两者使用相同的底层系统。

启用持久记忆​

memory 字段给子代理一个跨对话持久化的目录。子代理使用此目录随时间积累知识，如代码库模式、调试见解和架构决策。

---
name: code-reviewer
description: Reviews code for quality and best practices
memory: user
---

You are a code reviewer. As you review code, update your agent memory with
patterns, conventions, and recurring issues you discover.

根据记忆应多广泛地应用来选择作用域：

作用域位置何时使用
user~/.claude/agent-memory/<name-of-agent>/子代理应在所有项目中记住学习
project.claude/agent-memory/<name-of-agent>/子代理的知识是项目特定的且可通过版本控制共享
local.claude/agent-memory-local/<name-of-agent>/子代理的知识是项目特定的但不应该检入版本控制

启用记忆时：

  • 子代理的系统提示包含读取和写入记忆目录的指令。
  • 子代理的系统提示还包括记忆目录中 MEMORY.md 的前 200 行或 25KB（以先到者为准），以及如果超过该限制则管理 MEMORY.md 的指令。
  • Read、Write 和 Edit 工具自动启用，以便子代理管理其记忆文件。
持久记忆技巧​

  • project 是推荐的默认作用域。它使子代理的知识可通过版本控制共享。当子代理的知识在所有项目中广泛适用时使用 user，或当知识不应检入版本控制时使用 local。

  • 要求子代理在开始工作前查阅其记忆："审查此 PR，并检查你的记忆中是否有之前见过的模式。"

  • 要求子代理在完成任务后更新其记忆："现在你完成了，将学到的东西保存到你的记忆中。" 随着时间推移，这会建立知识库，使子代理更有效。

  • 将记忆指令直接包含在子代理的 markdown 文件中，使其主动维护自己的知识库：

    Update your agent memory as you discover codepaths, patterns, library
    locations, and key architectural decisions. This builds up institutional
    knowledge across conversations. Write concise notes about what you found
    and where.

带 hooks 的条件规则​

要对工具使用进行更动态的控制，使用 PreToolUse hooks 在操作执行之前验证它们。这在你需要允许工具的某些操作同时阻止其他操作时很有用。

这个示例创建一个只允许只读数据库查询的子代理。PreToolUse hook 在每个 Bash 命令执行之前运行指定的脚本：

---
name: db-reader
description: Execute read-only database queries
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly-query.sh"
---

Claude Code 通过 stdin 将 hook 输入作为 JSON 传递给 hook 命令。验证脚本读取此 JSON，提取 Bash 命令，并以退出码 2 退出来阻止写操作：

#!/bin/bash
# ./scripts/validate-readonly-query.sh

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# Block SQL write operations (case-insensitive)
if echo "$COMMAND" | grep -iE '\b(INSERT|UPDATE|DELETE|DROP|CREATE|ALTER|TRUNCATE)\b' > /dev/null; then
  echo "Blocked: Only SELECT queries are allowed" >&2
  exit 2
fi

exit 0

见 Hook 输入获取完整的输入 schema，退出码了解退出码如何影响行为。

禁用特定子代理​

你可以通过将特定子代理添加到设置中的 deny 数组来阻止 Claude 使用它们。使用格式 Agent(subagent-name)，其中 subagent-name 匹配子代理的 name 字段。

{
  "permissions": {
    "deny": ["Agent(Explore)", "Agent(my-custom-agent)"]
  }
}

这适用于内置和自定义子代理。你也可以使用 --disallowedTools CLI 标志：

claude --disallowedTools "Agent(Explore)"

有关权限规则的更多详情，见权限文档。

为子代理定义 hooks​

子代理可以定义在其生命周期内运行的 hooks。有两种配置 hooks 的方式：

  1. 在子代理的 frontmatter 中：定义仅在该子代理激活时运行的 hooks
  2. 在 settings.json 中：定义在主会话中子代理启动或停止时运行的 hooks

子代理 frontmatter 中的 hooks​

直接在子代理的 markdown 文件中定义 hooks。这些 hooks 仅在该特定子代理激活时运行，完成后被清理。

支持所有 hook 事件。子代理最常见的事件是：

事件匹配器输入触发时机
PreToolUse工具名称子代理使用工具之前
PostToolUse工具名称子代理使用工具之后
Stop（无）子代理完成时（运行时转换为 SubagentStop）

这个示例使用 PreToolUse hook 验证 Bash 命令，使用 PostToolUse 在文件编辑后运行 linter：

---
name: code-reviewer
description: Review code changes with automatic linting
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-command.sh $TOOL_INPUT"
  PostToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "./scripts/run-linter.sh"
---

Frontmatter 中的 Stop hooks 会自动转换为 SubagentStop 事件。

子代理事件的 project 级别 hooks​

在 settings.json 中配置 hooks 以响应主会话中的子代理生命周期事件。

事件匹配器输入触发时机
SubagentStart代理类型名称子代理开始执行时
SubagentStop代理类型名称子代理完成时

两个事件都支持匹配器以按名称定位特定代理类型。这个示例仅在 db-agent 子代理启动时运行设置脚本，在任何子代理停止时运行清理脚本：

{
  "hooks": {
    "SubagentStart": [
      {
        "matcher": "db-agent",
        "hooks": [
          { "type": "command", "command": "./scripts/setup-db-connection.sh" }
        ]
      }
    ],
    "SubagentStop": [
      {
        "hooks": [
          { "type": "command", "command": "./scripts/cleanup-db-connection.sh" }
        ]
      }
    ]
  }
}

见 Hooks 获取完整的 hook 配置格式。

与子代理协作​

理解自动委托​

Claude 根据你的请求中的任务描述、子代理配置中的 description 字段和当前上下文自动委托任务。要鼓励主动委托，在子代理的描述字段中包含"use proactively"等短语。

显式调用子代理​

当自动委托不够用时，你可以自己请求子代理。三种模式从一次性建议到会话范围默认逐步升级：

  • 自然语言：在提示中命名子代理；Claude 决定是否委托
  • @-提及：确保子代理为一个任务运行
  • 会话范围：整个会话使用该子代理的系统提示、工具限制和模型，通过 --agent 标志或 agent 设置

对于自然语言，没有特殊语法。命名子代理，Claude 通常会委托：

Use the test-runner subagent to fix failing tests
Have the code-reviewer subagent look at my recent changes

@-提及子代理。 输入 @ 并从自动补全中选择子代理，与 @-提及文件的方式相同。这确保特定的子代理运行而不是让 Claude 选择：

@"code-reviewer (agent)" look at the auth changes

你的完整消息仍然发送给 Claude，它根据你要求的内容编写子代理的任务提示。@-提及控制 Claude 调用哪个子代理，而不是它接收什么提示。

已启用插件提供的子代理在自动补全中显示为 <plugin-name>:<agent-name>。当前在会话中运行的命名后台子代理也出现在自动补全中，在名称旁边显示其状态。你也可以不用选择器手动输入提及：本地子代理使用 @agent-<name>，插件子代理使用 @agent-<plugin-name>:<agent-name>。

将整个会话作为子代理运行。 传递 --agent <name> 启动一个会话，其中主线程本身采用该子代理的系统提示、工具限制和模型：

claude --agent code-reviewer

子代理的系统提示完全替换默认的 Claude Code 系统提示，与 --system-prompt 的方式相同。CLAUDE.md 文件和项目记忆仍然通过正常的消息流程加载。代理名称在启动标题中显示为 @<name>，以便你确认它处于激活状态。

这适用于内置和自定义子代理，并且在恢复会话时选择会持久化。

对于插件提供的子代理，传递作用域名称：claude --agent <plugin-name>:<agent-name>。

要使其成为项目中每个会话的默认值，在 .claude/settings.json 中设置 agent：

{
  "agent": "code-reviewer"
}

如果两者都存在，CLI 标志覆盖设置。

前台或后台运行子代理​

子代理可以在前台（阻塞）或后台（并发）运行：

  • 前台子代理阻塞主对话直到完成。权限提示和澄清问题（如 AskUserQuestion）会传递给你。
  • 后台子代理在你继续工作时并发运行。启动之前，Claude Code 会提示子代理需要的任何工具权限，确保它 upfront 获得必要的批准。运行后，子代理继承这些权限并自动拒绝任何未预批准的内容。如果后台子代理需要问澄清问题，该工具调用会失败但子代理继续运行。

如果后台子代理因缺少权限而失败，你可以启动一个具有相同任务的前台子代理来重试并带交互提示。

Claude 根据任务决定前台或后台运行子代理。你也可以：

  • 要求 Claude"在后台运行这个"
  • 按 Ctrl+B 将运行的任务放到后台

要禁用所有后台任务功能，将 CLAUDE_CODE_DISABLE_BACKGROUND_TASKS 环境变量设置为 1。见环境变量。

常见模式​

隔离高输出操作​

子代理最有效的用途之一是隔离产生大量输出的操作。运行测试、获取文档或处理日志文件可能消耗大量上下文。通过将这些委托给子代理，详细输出保留在子代理的上下文中，只有相关摘要返回到你的主对话。

Use a subagent to run the test suite and report only the failing tests with their error messages

运行并行研究​

对于独立调查，生成多个子代理同时工作：

Research the authentication, database, and API modules in parallel using separate subagents

每个子代理独立探索其领域，然后 Claude 综合发现。这在研究路径相互不依赖时效果最好。

警告：当子代理完成时，它们的结果返回到你的主对话。运行多个各自返回详细结果的子代理可能消耗大量上下文。

对于需要持续并行或超出上下文窗口的任务，代理团队给每个 worker 自己的独立上下文。

链式子代理​

对于多步骤工作流，要求 Claude 按顺序使用子代理。每个子代理完成其任务并将结果返回给 Claude，然后 Claude 将相关上下文传递给下一个子代理。

Use the code-reviewer subagent to find performance issues, then use the optimizer subagent to fix them

在子代理和主对话之间选择​

使用主对话当：

  • 任务需要频繁的来回交互或迭代优化
  • 多个阶段共享大量上下文（规划 → 实现 → 测试）
  • 你在进行快速、有针对性的更改
  • 延迟很重要。子代理从头开始，可能需要时间收集上下文

使用子代理当：

  • 任务产生你不需要在主上下文中的冗长输出
  • 你想执行特定的工具限制或权限
  • 工作是自包含的，可以返回摘要

当你想在主对话上下文中运行可复用的提示或工作流而不是隔离的子代理上下文时，考虑改用技能。

对于已经在你对话中的内容的快速问题，使用 /btw 而不是子代理。它看到你的完整上下文但没有工具访问，答案被丢弃而不是添加到历史中。

注意：子代理不能生成其他子代理。如果你的工作流需要嵌套委托，使用技能或从主对话链式子代理。

管理子代理上下文​

恢复子代理​

每次子代理调用都创建一个带全新上下文的实例。要继续现有子代理的工作而不是重新开始，要求 Claude 恢复它。

恢复的子代理保留其完整的对话历史，包括所有之前的工具调用、结果和推理。子代理从停止的地方继续而不是从头开始。

当子代理完成时，Claude 接收其代理 ID。Claude 使用 SendMessage 工具，将代理的 ID 作为 to 字段来恢复它。SendMessage 工具仅在通过 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 启用代理团队时可用。

要恢复子代理，要求 Claude 继续之前的工作：

Use the code-reviewer subagent to review the authentication module
[Agent completes]

Continue that code review and now analyze the authorization logic
[Claude resumes the subagent with full context from previous conversation]

如果已停止的子代理收到 SendMessage，它会在后台自动恢复，无需新的 Agent 调用。

如果你想显式引用代理 ID，也可以向 Claude 询问，或在 ~/.claude/projects/{project}/{sessionId}/subagents/ 的转录文件中找到 ID。每个转录存储为 agent-{agentId}.jsonl。

子代理转录独立于主对话持久化：

  • 主对话压缩：当主对话压缩时，子代理转录不受影响。它们存储在单独的文件中。
  • 会话持久化：子代理转录在其会话中持久化。你可以通过恢复同一会话在重启 Claude Code 后恢复子代理。
  • 自动清理：根据 cleanupPeriodDays 设置（默认：30 天）清理转录。

自动压缩​

子代理支持与主对话相同的逻辑自动压缩。默认情况下，自动压缩在约 95% 容量时触发。要更早触发压缩，将 CLAUDE_AUTOCOMPACT_PCT_OVERRIDE 设置为更低的百分比（例如 50）。详情见环境变量。

压缩事件记录在子代理转录文件中：

{
  "type": "system",
  "subtype": "compact_boundary",
  "compactMetadata": {
    "trigger": "auto",
    "preTokens": 167189
  }
}

preTokens 值显示压缩发生之前使用了多少 token。

子代理示例​

这些示例展示了构建子代理的有效模式。将它们作为起点，或使用 Claude 生成自定义版本。

最佳实践：

  • 设计专注的子代理： 每个子代理应擅长一个特定任务
  • 编写详细描述： Claude 使用描述来决定何时委托
  • 限制工具访问： 仅授予必要权限以确保安全和专注
  • 检入版本控制： 与团队共享 project 子代理

代码审查者​

一个不修改代码的只读子代理。这个示例展示了如何设计一个带有限工具访问（没有 Edit 或 Write）和详细提示的专注子代理，指定查找什么以及如何格式化输出。

---
name: code-reviewer
description: Expert code review specialist. Proactively reviews code for quality, security, and maintainability. Use immediately after writing or modifying code.
tools: Read, Grep, Glob, Bash
model: inherit
---

You are a senior code reviewer ensuring high standards of code quality and security.

When invoked:
1. Run git diff to see recent changes
2. Focus on modified files
3. Begin review immediately

Review checklist:
- Code is clear and readable
- Functions and variables are well-named
- No duplicated code
- Proper error handling
- No exposed secrets or API keys
- Input validation implemented
- Good test coverage
- Performance considerations addressed

Provide feedback organized by priority:
- Critical issues (must fix)
- Warnings (should fix)
- Suggestions (consider improving)

Include specific examples of how to fix issues.

调试器​

一个可以分析和修复问题的子代理。与代码审查者不同，这个包含 Edit 因为修复 bug 需要修改代码。提示提供了从诊断到验证的清晰工作流。

---
name: debugger
description: Debugging specialist for errors, test failures, and unexpected behavior. Use proactively when encountering any issues.
tools: Read, Edit, Bash, Grep, Glob
---

You are an expert debugger specializing in root cause analysis.

When invoked:
1. Capture error message and stack trace
2. Identify reproduction steps
3. Isolate the failure location
4. Implement minimal fix
5. Verify solution works

Debugging process:
- Analyze error messages and logs
- Check recent code changes
- Form and test hypotheses
- Add strategic debug logging
- Inspect variable states

For each issue, provides:
- Root cause explanation
- Evidence supporting the diagnosis
- Specific code fix
- Testing approach
- Prevention recommendations

Focus on fixing the underlying issue, not the symptoms.

数据科学家​

用于数据分析工作的领域特定子代理。这个示例展示了如何为典型编码任务之外的专门工作流创建子代理。它显式设置 model: sonnet 用于更强大的分析。

---
name: data-scientist
description: Data analysis expert for SQL queries, BigQuery operations, and data insights. Use proactively for data analysis tasks and queries.
tools: Bash, Read, Write
model: sonnet
---

You are a data scientist specializing in SQL and BigQuery analysis.

When invoked:
1. Understand the data analysis requirement
2. Write efficient SQL queries
3. Use BigQuery command line tools (bq) when appropriate
4. Analyze and summarize results
5. Present findings clearly

Key practices:
- Write optimized SQL queries with proper filters
- Use appropriate aggregations and joins
- Include comments explaining complex logic
- Format results for readability
- Provide data-driven recommendations

For each analysis:
- Explain the query approach
- Document any assumptions
- Highlight key findings
- Suggest next steps based on data

Always ensure queries are efficient and cost-effective.

数据库查询验证器​

一个允许 Bash 访问但验证命令以仅允许只读 SQL 查询的子代理。这个示例展示了如何在需要比 tools 字段更精细的控制时使用 PreToolUse hooks 进行条件验证。

---
name: db-reader
description: Execute read-only database queries. Use when analyzing data or generating reports.
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly-query.sh"
---

You are a database analyst with read-only access. Execute SELECT queries to answer questions about the data.

When asked to analyze data:
1. Identify which tables contain the relevant data
2. Write efficient SELECT queries with appropriate filters
3. Present results clearly with context

You cannot modify data. If asked to INSERT, UPDATE, DELETE, or modifying schema, explain that you only have read access.

Claude Code 通过 stdin 将 hook 输入作为 JSON 传递给 hook 命令。验证脚本读取此 JSON，提取正在执行的命令，并针对 SQL 写操作列表检查它。如果检测到写操作，脚本以退出码 2 退出来阻止执行并通过 stderr 将错误消息返回给 Claude。

在你的项目中任何位置创建验证脚本。路径必须匹配 hook 配置中的 command 字段：

#!/bin/bash
# Blocks SQL write operations, allows SELECT queries

# Read JSON input from stdin
INPUT=$(cat)

# Extract the command field from tool_input using jq
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

if [ -z "$COMMAND" ]; then
  exit 0
fi

# Block write operations (case-insensitive)
if echo "$COMMAND" | grep -iE '\b(INSERT|UPDATE|DELETE|DROP|CREATE|ALTER|TRUNCATE|REPLACE|MERGE)\b' > /dev/null; then
  echo "Blocked: Write operations not allowed. Use SELECT queries only." >&2
  exit 2
fi

exit 0

使脚本可执行：

chmod +x ./scripts/validate-readonly-query.sh

Hook 通过 stdin 接收 JSON，Bash 命令在 tool_input.command 中。退出码 2 阻止操作并将错误消息反馈给 Claude。有关退出码的详细信息，见 Hooks，有关完整输入 schema，见 Hook 输入。
