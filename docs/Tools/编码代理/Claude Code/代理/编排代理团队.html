编排 Claude Code 会话团队

协调多个 Claude Code 实例作为团队协同工作，包含共享任务、代理间消息传递和集中管理。

警告：代理团队是实验性功能，默认禁用。通过在 settings.json 或环境中添加 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS 来启用它们。代理团队在会话恢复、任务协调和关闭行为方面有已知限制。

代理团队让你可以协调多个 Claude Code 实例协同工作。一个会话充当团队负责人（team lead），协调工作、分配任务并综合结果。队友独立工作，各自拥有独立的上下文窗口，并且可以直接相互通信。

与子代理（在单个会话内运行，只能向主代理报告）不同，你也可以直接与个别队友交互，无需经过负责人。

注意：代理团队需要 Claude Code v2.1.32 或更高版本。使用 claude --version 检查你的版本。

本页涵盖：

  • 何时使用代理团队，包括最佳用例以及与子代理的比较
  • 启动团队
  • 控制队友，包括显示模式、任务分配和委托
  • 并行工作的最佳实践

何时使用代理团队​

代理团队在并行探索能带来真正价值的任务中最为有效。完整场景见用例示例。最强的用例包括：

  • 研究和审查：多个队友可以同时调查问题的不同方面，然后分享并挑战彼此的发现
  • 新模块或功能：队友可以各自负责独立的部分而互不干扰
  • 带竞争假设的调试：队友并行测试不同理论，更快收敛到答案
  • 跨层协调：涉及前端、后端和测试的更改，每部分由不同的队友负责

代理团队会增加协调开销并比单个会话消耗显著更多的 token。它们在队友可以独立操作时效果最好。对于顺序任务、同一文件编辑或有很多依赖的工作，单个会话或子代理更有效。

与子代理比较​

代理团队和子代理都可以并行化工作，但它们的运作方式不同。根据你的 worker 是否需要相互通信来选择：

架构对比

子代理只将结果报告给主代理，从不相互交谈。在代理团队中，队友共享任务列表，认领工作，并直接相互通信。

子代理代理团队
上下文自己的上下文窗口；结果返回给调用方自己的上下文窗口；完全独立
通信仅向主代理报告结果队友直接相互消息
协调主代理管理所有工作共享任务列表，自协调
适用于只需结果的专注任务需要讨论和协作的复杂工作
Token 成本较低：结果摘要返回到主上下文较高：每个队友是独立的 Claude 实例

当你需要快速、专注的 worker 并只需结果时使用子代理。当队友需要分享发现、相互挑战并自行协调时使用代理团队。

启用代理团队​

代理团队默认禁用。通过在 shell 环境或 settings.json 中将 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS 环境变量设置为 1 来启用：

{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

启动第一个代理团队​

启用代理团队后，用自然语言告诉 Claude 创建代理团队并描述你想要的任务和团队结构。Claude 会创建团队、生成队友并根据你的提示协调工作。

这个示例效果很好，因为三个角色是独立的，可以无需等待彼此就能探索问题：

I'm designing a CLI tool that helps developers track TODO comments across
their codebase. Create an agent team to explore this from different angles: one
teammate on UX, one on technical architecture, one playing devil's advocate.

从这里开始，Claude 创建一个带共享任务列表的团队，为每个视角生成队友，让他们探索问题，综合发现，并在完成后尝试清理团队。

负责人的终端列出所有队友以及他们正在做什么。使用 Shift+Down 循环浏览队友并直接给他们发消息。在最后一个队友之后，Shift+Down 会绕回到负责人。

如果你希望每个队友在独立的分屏面板中，见选择显示模式。

控制你的代理团队​

用自然语言告诉负责人你想要什么。它会根据你的指令处理团队协调、任务分配和委托。

选择显示模式​

代理团队支持两种显示模式：

  • 进程内（In-process）：所有队友在你的主终端内运行。使用 Shift+Down 循环浏览队友并输入消息。适用于任何终端，无需额外设置。
  • 分屏（Split panes）：每个队友获得独立的面板。你可以同时看到所有人的输出并点击面板直接交互。需要 tmux 或 iTerm2。

注意：tmux 在某些操作系统上有已知限制，传统上在 macOS 上效果最好。在 iTerm2 中使用 tmux -CC 是推荐的 tmux 入口。

默认是 "auto"，如果你已经在 tmux 会话中运行则使用分屏，否则使用进程内。"tmux" 设置启用分屏模式并根据你的终端自动检测使用 tmux 还是 iTerm2。要覆盖，在 ~/.claude.json 的全局配置中设置 teammateMode：

{
  "teammateMode": "in-process"
}

要为单个会话强制进程内模式，作为标志传递：

claude --teammate-mode in-process

分屏模式需要 tmux 或带 it2 CLI 的 iTerm2。手动安装：

  • tmux：通过系统的包管理器安装。见 tmux wiki 获取平台特定说明。
  • iTerm2：安装 it2 CLI，然后在 iTerm2 → Settings → General → Magic → Enable Python API 中启用 Python API。

指定队友和模型​

Claude 根据你的任务决定生成多少队友，或者你可以精确指定你想要的：

Create a team with 4 teammates to refactor these modules in parallel.
Use Sonnet for each teammate.

要求队友计划审批​

对于复杂或高风险任务，你可以要求队友在实现之前先做计划。队友在只读计划模式下工作，直到负责人批准他们的方法：

Spawn an architect teammate to refactor the authentication module.
Require plan approval before they make any changes.

当队友完成计划时，它会向负责人发送计划审批请求。负责人审查计划并批准或附带反馈拒绝。如果被拒绝，队友保持在计划模式下，根据反馈修改并重新提交。一旦被批准，队友退出计划模式并开始实现。

负责人自主做出审批决定。要影响负责人的判断，在你的提示中给它标准，如"只批准包含测试覆盖的计划"或"拒绝修改数据库 schema 的计划"。

直接与队友对话​

每个队友都是完整、独立的 Claude Code 会话。你可以直接给任何队友发消息来提供额外指令、追问或重定向他们的方法。

  • 进程内模式：使用 Shift+Down 循环浏览队友，然后输入消息给他们。按 Enter 查看队友的会话，然后按 Escape 中断他们当前的轮次。按 Ctrl+T 切换任务列表。
  • 分屏模式：点击队友的面板直接与他们的会话交互。每个队友都有自己的终端完整视图。

分配和认领任务​

共享任务列表协调团队的工作。负责人创建任务，队友处理它们。任务有三种状态：pending（待处理）、in progress（进行中）和 completed（已完成）。任务也可以依赖其他任务：有未解决依赖的待处理任务在这些依赖完成之前不能被认领。

负责人可以显式分配任务，或者队友可以自行认领：

  • 负责人分配：告诉负责人将哪个任务给哪个队友
  • 自行认领：完成任务后，队友自行选择下一个未分配、未阻塞的任务

任务认领使用文件锁来防止多个队友同时尝试认领同一任务时的竞态条件。

关闭队友​

要优雅地结束队友的会话：

Ask the researcher teammate to shut down

负责人发送关闭请求。队友可以批准并优雅退出，或附带解释拒绝。

清理团队​

当你完成时，要求负责人清理：

Clean up the team

这会移除共享的团队资源。当负责人运行清理时，它会检查活跃的队友，如果有任何仍在运行则失败，所以先关闭他们。

警告：始终使用负责人来清理。队友不应该运行清理，因为他们的团队上下文可能无法正确解析，可能导致资源处于不一致状态。

使用 hooks 执行质量门控​

使用 hooks 在队友完成工作或任务被创建或完成时强制执行规则：

  • TeammateIdle：当队友即将进入空闲时运行。以退出码 2 退出以发送反馈并让队友继续工作。
  • TaskCreated：当任务被创建时运行。以退出码 2 退出以阻止创建并发送反馈。
  • TaskCompleted：当任务被标记完成时运行。以退出码 2 退出以阻止完成并发送反馈。

代理团队如何工作​

本节涵盖代理团队背后的架构和机制。如果你想开始使用它们，见上方的控制你的代理团队。

Claude 如何启动代理团队​

代理团队有两种方式启动：

  • 你请求团队：给 Claude 一个受益于并行工作的任务并显式要求代理团队。Claude 会根据你的指令创建一个。
  • Claude 提议团队：如果 Claude 确定你的任务受益于并行工作，它可能会建议创建团队。你在它继续之前确认。

在这两种情况下，你都保持控制。未经你批准，Claude 不会创建团队。

架构​

代理团队由以下组件组成：

组件角色
负责人创建团队、生成队友并协调工作的主 Claude Code 会话
队友各自负责分配任务的独立 Claude Code 实例
任务列表队友认领和完成的共享工作项列表
邮箱代理间通信的消息系统

见选择显示模式了解显示配置选项。队友消息自动到达负责人。

系统自动管理任务依赖。当队友完成其他任务依赖的任务时，被阻塞的任务会自动解除阻塞，无需手动干预。

团队和任务存储在本地：

  • 团队配置：~/.claude/teams/{team-name}/config.json
  • 任务列表：~/.claude/tasks/{team-name}/

当你创建团队时，Claude Code 自动生成这两者，并随着队友加入、进入空闲或离开而更新它们。团队配置包含会话 ID 和 tmux 面板 ID 等运行时状态，因此不要手动编辑它或预先编写：你的更改会在下次状态更新时被覆盖。

要定义可复用的队友角色，改用子代理定义。

团队配置包含 members 数组，其中有每个队友的名称、代理 ID 和代理类型。队友可以读取此文件来发现其他团队成员。

没有等效于项目级别的团队配置。你项目目录中的 .claude/teams/teams.json 不会被识别为配置；Claude 将其视为普通文件。

使用子代理定义作为队友​

在生成队友时，你可以引用任何子代理作用域（project、user、插件或 CLI 定义）中的子代理类型。这让你可以定义一次角色（如 security-reviewer 或 test-runner），并将其既用作委托的子代理，又用作代理团队队友。

要使用子代理定义，在要求 Claude 生成队友时按名称提及它：

Spawn a teammate using the security-reviewer agent type to audit the auth module.

队友会尊重该定义的 tools 白名单和 model，定义的 body 会作为附加指令附加到队友的系统提示中，而不是替换它。即使 tools 限制了其他工具，SendMessage 和任务管理工具等团队协调工具对队友始终可用。

注意：子代理定义中的 skills 和 mcpServers frontmatter 字段在该定义作为队友运行时不应用。队友从你的 project 和 user 设置中加载技能和 MCP 服务器，与常规会话相同。

权限​

队友以负责人的权限设置启动。如果负责人以 --dangerously-skip-permissions 运行，所有队友也是如此。生成后，你可以更改个别队友的模式，但不能在生成时设置每个队友的模式。

上下文和通信​

每个队友都有自己的上下文窗口。生成时，队友加载与常规会话相同的项目上下文：CLAUDE.md、MCP 服务器和技能。它还接收来自负责人的生成提示。负责人的对话历史不会传递。

队友如何共享信息：

  • 自动消息传递：当队友发送消息时，它们会自动传递给接收方。负责人无需轮询更新。
  • 空闲通知：当队友完成并停止时，他们会自动通知负责人。
  • 共享任务列表：所有代理都可以看到任务状态并认领可用工作。

队友消息：

  • message：向特定队友发送消息
  • broadcast：同时向所有队友发送。谨慎使用，因为成本随团队规模扩展。

负责人在生成每个队友时给他们分配一个名称，任何队友都可以通过该名称向任何其他队友发消息。要获得你可以在后续提示中引用的可预测名称，在你的生成指令中告诉负责人如何称呼每个队友。

Token 用量​

代理团队比单个会话消耗显著更多的 token。每个队友都有自己的上下文窗口，token 用量随活跃队友数量扩展。对于研究、审查和新功能工作，额外的 token 通常是值得的。对于常规任务，单个会话更具成本效益。使用指导见代理团队 token 成本。

用例示例​

这些示例展示了代理团队如何处理并行探索能带来价值的任务。

运行并行代码审查​

单个审查者往往倾向于一次关注一类问题。将审查标准拆分为独立领域意味着安全、性能和测试覆盖可以同时获得彻底关注。提示为每个队友分配不同的视角，这样他们不会重叠：

Create an agent team to review PR #142. Spawn three reviewers:
- One focused on security implications
- One checking performance impact
- One validating test coverage
Have them each review and report findings.

每个审查者从同一个 PR 工作，但应用不同的过滤器。负责人在他们完成后综合所有三个的发现。

用竞争假设调查​

当根本原因不清楚时，单个代理倾向于找到一个合理的解释就停止查找。提示通过让队友显式对抗来对抗这一点：每个队友的工作不仅是调查自己的理论，还要挑战其他人的理论。

Users report the app exits after one message instead of staying connected.
Spawn 5 agent teammates to investigate different hypotheses. Have them talk to
each other to try to disprove each other's theories, like a scientific
debate. Update the findings doc with whatever consensus emerges.

辩论结构是关键机制。顺序调查 suffers from anchoring：一旦探索了一个理论，后续调查就会偏向它。

通过多个独立调查员积极试图相互反驳，存活下来的理论更可能是真正的根本原因。

最佳实践​

给队友足够的上下文​

队友自动加载项目上下文，包括 CLAUDE.md、MCP 服务器和技能，但他们不继承负责人的对话历史。详情见上下文和通信。在生成提示中包含特定于任务的细节：

Spawn a security reviewer teammate with the prompt: "Review the authentication module
at src/auth/ for security vulnerabilities. Focus on token handling, session
management, and input validation. The app uses JWT tokens stored in
httpOnly cookies. Report any issues with severity ratings."

选择适当的团队规模​

队友数量没有硬性限制，但有实际约束：

  • Token 成本线性扩展：每个队友有自己的上下文窗口并独立消耗 token。详情见代理团队 token 成本。
  • 协调开销增加：更多队友意味着更多通信、任务协调和潜在冲突
  • 收益递减：超过某个点，额外的队友不会按比例加速工作

大多数工作流从 3-5 个队友开始。这在并行工作和可管理的协调之间取得平衡。本指南中的示例使用 3-5 个队友，因为这个范围在不同任务类型中效果都很好。

每个队友 5-6 个任务让每个人都保持高效，不会过多的上下文切换。如果你有 15 个独立任务，3 个队友是一个好的起点。

仅在工作真正受益于队友同时工作时才扩大规模。三个专注的队友通常优于五个分散的队友。

适当 sized 任务​

  • 太小：协调开销超过收益
  • 太大：队友工作太久没有检查点，增加浪费 effort 的风险
  • 刚好：自包含的单元，产生清晰的交付物，如函数、测试文件或审查

提示：负责人自动将工作拆分为任务并分配给队友。如果它没有创建足够的任务，要求它将工作拆分为更小的部分。每个队友 5-6 个任务让每个人都保持高效，并让负责人在某人卡住时重新分配工作。

等待队友完成​

有时负责人开始自己实现任务而不是等待队友。如果你注意到这个：

Wait for your teammates to complete their tasks before proceeding

从研究和审查开始​

如果你是代理团队的新手，从有清晰边界且不需要编写代码的任务开始：审查 PR、研究库或调查 bug。这些任务展示了并行探索的价值，而不会带来并行实现带来的协调挑战。

避免文件冲突​

两个队友编辑同一个文件会导致覆盖。拆分工作，让每个队友负责不同的文件集。

监控和引导​

检查队友的进展，重定向不起作用的方法，并在发现进来时综合它们。让团队无人看管运行太久会增加浪费 effort 的风险。

故障排除​

队友不出现​

如果你要求 Claude 创建团队后队友没有出现：

  • 在进程内模式中，队友可能已经在运行但不可见。按 Shift+Down 循环浏览活跃的队友。
  • 检查你给 Claude 的任务是否复杂到需要团队。Claude 根据任务决定是否生成队友。
  • 如果你显式请求了分屏，确保 tmux 已安装并在 PATH 中可用： 
    which tmux
  • 对于 iTerm2，验证 it2 CLI 已安装且 iTerm2 偏好设置中启用了 Python API。

太多权限提示​

队友的权限请求会冒泡到负责人，这会产生摩擦。在生成队友之前，在权限设置中预批准常见操作以减少中断。

队友在错误时停止​

队友可能在遇到错误后停止而不是恢复。在进程内模式中使用 Shift+Down 或在分屏模式中点击面板检查他们的输出，然后：

  • 直接给他们额外指令
  • 生成替代队友继续工作

负责人在工作完成前关闭​

负责人可能在所有任务实际完成之前决定团队已完成。如果发生这种情况，告诉它继续。如果它开始做工作而不是委托，你也可以告诉负责人在继续之前等待队友完成。

孤立的 tmux 会话​

如果团队结束后 tmux 会话仍然存在，它可能没有被完全清理。列出会话并杀死团队创建的会话：

tmux ls
tmux kill-session -t <session-name>

限制​

代理团队是实验性的。当前需要注意的限制：

  • 进程内队友不支持会话恢复：/resume 和 /rewind 不会恢复进程内队友。恢复会话后，负责人可能会尝试向不再存在的队友发消息。如果发生这种情况，告诉负责人生成新队友。
  • 任务状态可能滞后：队友有时未能将任务标记为完成，这会阻塞依赖任务。如果任务看起来卡住了，检查工作是否实际完成并手动更新任务状态或告诉负责人催促队友。
  • 关闭可能很慢：队友在完成当前请求或工具调用之前才会关闭，这可能需要时间。
  • 每会话一个团队：负责人一次只能管理一个团队。在开始新团队之前清理当前团队。
  • 没有嵌套团队：队友不能生成自己的团队或队友。只有负责人可以管理团队。
  • 负责人是固定的：创建团队的会话在其生命周期内是负责人。你不能将队友提升为负责人或转移领导权。
  • 权限在生成时设置：所有队友以负责人的权限模式启动。你可以在生成后更改个别队友模式，但不能在生成时设置每个队友的模式。
  • 分屏需要 tmux 或 iTerm2：默认进程内模式适用于任何终端。分屏模式在 VS Code 集成终端、Windows Terminal 或 Ghostty 中不受支持。

提示：CLAUDE.md 正常工作：队友从他们的工作目录读取 CLAUDE.md 文件。使用它向所有队友提供项目特定的指导。

下一步​

探索并行工作和委托的相关方法：

  • 轻量级委托：子代理在你的会话内生成研究或验证的辅助代理，更适合不需要代理间协调的任务
  • 手动并行会话：Git worktrees 让你自己运行多个 Claude Code 会话，无需自动化团队协调
  • 比较方法：见子代理 vs 代理团队比较获取并排分解
