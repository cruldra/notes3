自托管教程

实战部署一份 Daytona OSS 到 Ubuntu 服务器, 配 HTTPS 域名, 接入 frp 反代. 期间踩了 7 个坑, 全部记录.

前置条件​

  • 一台 Linux 服务器 (4 核 / 8G 内存 / 50G 磁盘以上, 内网即可)
  • Docker + Docker Compose v2 (docker compose version 能跑)
  • 一个公网域名 (本文用 example.com 占位)
  • frp + Caddy (或类似 公网入口) —— 不强制, 仅内网用可省

整体步骤​

1. clone repo, 拷 docker/ 到部署目录
2. 改 docker-compose.yaml: 换密钥 + 改 OIDC URL 到真实域名
3. 改 dex/config.yaml: issuer + redirect URIs 同步
4. (国内) runner 配 docker mirror 避免 docker.io 超时
5. (国内) image 换内置 registry 避免 inspect 走外网
6. docker compose up -d, 等所有容器 healthy
7. 配 HTTPS (Caddy 或类似)
8. 浏览器登 dashboard 创建 API key
9. SDK 测试创建 sandbox

步骤 1: 拉源码​

git clone --depth 1 https://github.com/daytonaio/daytona.git /tmp/daytona-src
mkdir -p /opt/daytona
cp -rT /tmp/daytona-src/docker/ /opt/daytona/
cd /opt/daytona
ls
# Caddyfile? 没有 (这是 Daytona 自己的 docker-compose)
# docker-compose.yaml ✓
# dex/  otel/  pgadmin4/  README.md ✓

步骤 2: 换密钥 + 改 OIDC URL​

默认密钥不能用 —— supersecretkey / pass / secret_api_token 这种到处可见.

写一个 .env 文件 (跟 docker-compose.yaml 同目录):

cat > .env <<EOF
ENCRYPTION_KEY=$(openssl rand -hex 32)
ENCRYPTION_SALT=$(openssl rand -hex 16)
DB_PASSWORD=$(openssl rand -hex 16)
PROXY_API_KEY=$(openssl rand -hex 32)
DEFAULT_RUNNER_API_KEY=$(openssl rand -hex 32)
SSH_GATEWAY_API_KEY=$(openssl rand -hex 32)
OTEL_COLLECTOR_API_KEY=$(openssl rand -hex 16)
HEALTH_CHECK_API_KEY=$(openssl rand -hex 16)
EOF
chmod 600 .env

把 docker-compose.yaml 里的硬编码密钥改成 ${VAR} 引用:

sed -i 's|ENCRYPTION_KEY=supersecretkey|ENCRYPTION_KEY=${ENCRYPTION_KEY}|g' docker-compose.yaml
sed -i 's|ENCRYPTION_SALT=supersecretsalt|ENCRYPTION_SALT=${ENCRYPTION_SALT}|g' docker-compose.yaml
sed -i 's|DB_PASSWORD=pass|DB_PASSWORD=${DB_PASSWORD}|g' docker-compose.yaml
sed -i 's|POSTGRES_PASSWORD=pass|POSTGRES_PASSWORD=${DB_PASSWORD}|g' docker-compose.yaml
sed -i 's|PROXY_API_KEY=super_secret_key|PROXY_API_KEY=${PROXY_API_KEY}|g' docker-compose.yaml
sed -i 's|DEFAULT_RUNNER_API_KEY=secret_api_token|DEFAULT_RUNNER_API_KEY=${DEFAULT_RUNNER_API_KEY}|g' docker-compose.yaml
sed -i 's|DAYTONA_RUNNER_TOKEN=secret_api_token|DAYTONA_RUNNER_TOKEN=${DEFAULT_RUNNER_API_KEY}|g' docker-compose.yaml
sed -i 's|SSH_GATEWAY_API_KEY=ssh_secret_api_token|SSH_GATEWAY_API_KEY=${SSH_GATEWAY_API_KEY}|g' docker-compose.yaml
sed -i 's|API_KEY=ssh_secret_api_token|API_KEY=${SSH_GATEWAY_API_KEY}|g' docker-compose.yaml
sed -i 's|OTEL_COLLECTOR_API_KEY=otel_collector_api_key|OTEL_COLLECTOR_API_KEY=${OTEL_COLLECTOR_API_KEY}|g' docker-compose.yaml
sed -i 's|HEALTH_CHECK_API_KEY=supersecretkey|HEALTH_CHECK_API_KEY=${HEALTH_CHECK_API_KEY}|g' docker-compose.yaml

改 OIDC URL (4 处) 和 DASHBOARD URL (2 处) 到真实域名:

# 假设你的域名是: daytona-api.example.com (api), daytona-dex.example.com (dex)
sed -i 's|OIDC_ISSUER_BASE_URL=http://dex:5556/dex|OIDC_ISSUER_BASE_URL=https://daytona-dex.example.com/dex|' docker-compose.yaml
sed -i 's|PUBLIC_OIDC_DOMAIN=http://localhost:5556/dex|PUBLIC_OIDC_DOMAIN=https://daytona-dex.example.com/dex|' docker-compose.yaml
sed -i 's|OIDC_DOMAIN=http://dex:5556/dex|OIDC_DOMAIN=https://daytona-dex.example.com/dex|' docker-compose.yaml
sed -i 's|OIDC_PUBLIC_DOMAIN=http://localhost:5556/dex|OIDC_PUBLIC_DOMAIN=https://daytona-dex.example.com/dex|' docker-compose.yaml
sed -i 's|DASHBOARD_URL=http://localhost:3000/dashboard|DASHBOARD_URL=https://daytona-api.example.com/dashboard|' docker-compose.yaml
sed -i 's|DASHBOARD_BASE_API_URL=http://localhost:3000|DASHBOARD_BASE_API_URL=https://daytona-api.example.com|' docker-compose.yaml
sed -i 's|PROXY_DOMAIN=proxy.localhost:4000|PROXY_DOMAIN=daytona-proxy.example.com:443|' docker-compose.yaml
sed -i 's|PROXY_TEMPLATE_URL=http://{{PORT}}-{{sandboxId}}.proxy.localhost:4000|PROXY_TEMPLATE_URL=https://{{PORT}}-{{sandboxId}}.daytona-proxy.example.com|' docker-compose.yaml

步骤 3: 改 dex/config.yaml​

dex/config.yaml 里 issuer 和 redirectURIs 也要改:

issuer: https://daytona-dex.example.com/dex   # ← 改这里

staticClients:
  - id: daytona
    redirectURIs:
      - 'https://daytona-api.example.com'                              # ← 改
      - 'https://daytona-api.example.com/api/oauth2-redirect.html'    # ← 改
      - 'https://daytona-api.example.com/callback'                    # ← 改
      - 'https://daytona-proxy.example.com/callback'                  # ← 改
    name: 'Daytona'
    public: true

关键: dex 自己宣告的 issuer 必须完全等于 OIDC_ISSUER_BASE_URL (容器层 fetch 时的 URL). token 验证靠这个字符串严格匹配.

步骤 4: 国内 docker mirror (跳过此步骤如果你在境外)​

症状: runner 启动时尝试 inspect sandbox image 报:

Get "https://registry-1.docker.io/v2/": net/http: request canceled while waiting for connection

原因: runner 内部跑了独立 docker daemon (DinD), 默认配置直连 docker.io, 国内访问慢/被墙.

修法: 给 runner 内部 daemon 配 mirror.

cat > runner-daemon.json <<'EOF'
{
  "registry-mirrors": [
    "https://docker.mirrors.ustc.edu.cn",
    "https://docker.m.daocloud.io",
    "https://docker.1panel.live"
  ],
  "insecure-registries": ["registry:6000"],
  "max-concurrent-downloads": 10
}
EOF

cat > docker-compose.override.yaml <<'EOF'
services:
  runner:
    volumes:
      - ./runner-daemon.json:/etc/docker/daemon.json:ro
EOF

docker-compose.override.yaml 自动跟主 yaml merge, 给 runner 容器额外 mount 一个 daemon.json.

步骤 5: 用 internal registry 做 sandbox image (国内必做)​

坑: 即使配了 mirror, runner 的 docker manifest inspect 命令不走 mirror, 仍直连 docker.io, 还是会超时. 这是 docker daemon 的 bug/设计.

最稳的办法: 把 sandbox image push 到 daytona 内置的 registry (registry:6000), DEFAULT_SNAPSHOT 改用这个内部 image.

# 在 runner 容器内 tag + push (mirror 已能 pull)
docker exec daytona-runner-1 sh -c "
  docker tag daytonaio/sandbox:0.5.0-slim registry:6000/daytona/sandbox:0.5.0-slim &&
  docker push registry:6000/daytona/sandbox:0.5.0-slim
"

# 验证 push 成功 (registry 容器里看 repository 树)
docker exec daytona-registry-1 ls /var/lib/registry/docker/registry/v2/repositories/

# 改 docker-compose.yaml 让 default snapshot 用 internal image
sed -i 's|DEFAULT_SNAPSHOT=daytonaio/sandbox:0.5.0-slim|DEFAULT_SNAPSHOT=registry:6000/daytona/sandbox:0.5.0-slim|' docker-compose.yaml

# 删旧 snapshot row 让 api 重新 init (如果之前已经跑过)
docker exec daytona-db-1 psql -U user daytona -c "DELETE FROM snapshot;"

# recreate api
docker compose up -d

之后 inspect 走容器内网 (registry:6000), 秒成.

步骤 6: 启动​

docker compose up -d

等 1-2 分钟所有容器 healthy:

docker compose ps
# 期望: 所有 14 个容器都是 Up + (healthy)

测 API 健康:

curl http://localhost:3030/api/health
# 期望: {"status":"ok"}

注意: 如果你要把容器端口绑到特定 IP (而不是 0.0.0.0), 要改 yaml 的 ports: 字段, 比如 192.168.1.4:3030:3000. 但确保 IP 是宿主机真实拥有的网卡 IP —— 写错了会报 cannot assign requested address.

步骤 7: 配 HTTPS (用 Caddy)​

最简单的 Caddy 配置, 给 api 和 dex 各一个反代:

daytona-api.example.com {
    reverse_proxy 192.168.1.4:3030 {
        header_up X-Forwarded-Proto https
    }
}

daytona-dex.example.com {
    reverse_proxy 192.168.1.4:5556 {
        header_up X-Forwarded-Proto https
    }
}

Caddy 自动走 ACME HTTP-01 申请 Let's Encrypt 证书.

通过 frp 反代时, frpc.ini 加:

[daytona-api]
type = http
local_ip = 192.168.1.4   # 必须显式, 否则 frpc 默认连 127.0.0.1
local_port = 3030
custom_domains = daytona-api.example.com
header_X-Forwarded-Proto = https

[daytona-dex]
type = http
local_ip = 192.168.1.4
local_port = 5556
custom_domains = daytona-dex.example.com
header_X-Forwarded-Proto = https

local_ip = 192.168.1.4 这一行很重要 —— 如果省略, frpc 默认连 127.0.0.1:port, 同主机如果还跑了别的服务用同样端口 (比如 langfuse 也在 3030), 流量就会路由到错的地方.

步骤 8: 登录 + 创建 API key​

打开 https://daytona-api.example.com/dashboard, 用默认账号:

  • 邮箱: dev@daytona.io
  • 密码: password

(密码定义在 dex/config.yaml 的 staticPasswords 里, 生产部署应该改掉)

登录进去 → 左侧 Settings → Keys → Create New API Key:

  • 避坑: 当前版本 Full Access 选项有 bug, 会发送 server 不识别的 permission 字串
  • 走 Restricted 模式, 手动勾选错误信息列出来的 14 个权限:
    • write:registries / delete:registries
    • write:snapshots / delete:snapshots
    • write:sandboxes / delete:sandboxes
    • read:volumes / write:volumes / delete:volumes
    • write:regions / delete:regions
    • read:runners / write:runners / delete:runners
    • read:audit_logs

创建后会显示一次 dtn_xxx..., 存好 —— 关掉就再也看不到了.

步骤 9: 给 organization 绑 default region​

OSS 部署后 organization 默认没有 default region, sandbox 创建会报 This organization does not have a default region.

最简办法: 在 dashboard Settings 找 region 设置, 把 us region 设为 default.

如果 dashboard 没暴露这个 UI, 用 API 设 (但 API key 没权限调 organization 接口, 需要先在 dashboard 拿 OIDC token):

# 在浏览器 DevTools → Network → 任一 /api/* 请求 → Authorization header
# 拿到形如 ey... 的 OIDC access token
TOKEN="ey..."
ORG_ID="$(从 dashboard URL 或 /api/users/me 拿到)"

curl -X PATCH \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"regionId":"us"}' \
  "https://daytona-api.example.com/api/organizations/$ORG_ID/default-region"

步骤 10: SDK 测试​

from daytona import Daytona, DaytonaConfig

cfg = DaytonaConfig(
    api_url="https://daytona-api.example.com/api",   # 注意带 /api 后缀
    api_key="dtn_xxx",
)
client = Daytona(cfg)

sb = client.create()
print(sb.id, sb.state)

r = sb.process.exec("echo hello && uname -a")
print(r.exit_code, r.result)

sb.delete()

期望输出:

4f00ccaf-fc98-... SandboxState.STARTED
0
hello
Linux 4f00ccaf-fc98-... 6.8.0-... x86_64 GNU/Linux

7 个常见踩坑​

症状原因修法
cannot assign requested addressdocker-compose ports: 绑了非本机 IP用真实网卡 IP 或 0.0.0.0
frpc 路由到错的容器langfuse 等占用同端口在 127.0.0.1frpc.ini 加 local_ip = 192.168.1.x
Caddy ACME 拒绝签证书域名没在 Caddyfile site list加进 site list 然后 docker restart
浏览器 Crypto.subtle is available only in secure contexts用 IP/HTTP 访问 dashboard必须 HTTPS, 配真实域名 + Caddy 自动签
Snapshot inspect timeoutrunner 内 docker daemon 直连 docker.io改用 internal registry image
Full Access API Key 创建报 permission 不识别dashboard 跟 server 版本不匹配用 Restricted 模式手选
No available runnerssnapshot 没绑到 runnerDELETE snapshot row 让 api 重新 init

资源消耗参考​

跑空 (无 sandbox):

内存: ~2.5 GB
磁盘: ~3 GB (镜像 + db)
CPU: 3-5% idle

每个 sandbox: 默认 2 GB 内存 / 1 CPU / 3 GB 磁盘. 想限流可改 DEFAULT_ORG_QUOTA_* 系列环境变量.

备份 / 灾难恢复​

最少要备份的:

  • db_data volume (PostgreSQL 数据库) ← 用户/sandbox/snapshot 元数据
  • minio_data volume ← 用户应用数据 + 日志
  • registry volume ← 内部镜像
  • dex_db volume ← OIDC 用户

定期 docker run --rm -v daytona_db_data:/data -v $(pwd):/backup busybox tar czf /backup/db-$(date +%F).tar.gz /data 之类的备份脚本.
