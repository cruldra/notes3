核心组件

一个 Daytona OSS docker-compose 启动后会跑 14 个容器, 看起来很多, 但它们各自职责清晰. 这篇按"数据流方向"把它们分成 5 层讲.

一张架构图​

┌─────────────────────────────────────────────────────────────────────┐
│                           用户 / SDK                                  │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ HTTPS + Bearer token
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  ① 接入层                                                             │
│  ├─ api          NestJS, REST 总入口, port 3000                       │
│  ├─ proxy        sandbox preview URL 路由器, port 4000                │
│  ├─ ssh-gateway  允许 ssh 进 sandbox, port 2222                       │
│  └─ dex          OIDC 登录提供者 (浏览器登 dashboard 用), port 5556    │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ 内部调度
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  ② 调度层                                                             │
│  ├─ runner       真正起/停 sandbox 容器的服务 (DinD), port 3003       │
│  └─ snapshot 体系 (在 api 内, 不是独立容器)                            │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  ③ 存储层                                                             │
│  ├─ db (postgres)  agent / sandbox / snapshot 元数据                  │
│  ├─ redis          job 队列 + 缓存                                     │
│  ├─ registry       内置 docker registry, 存 sandbox image            │
│  └─ minio          S3 兼容对象存储, 存 volume / log                    │
└─────────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────────┐
│  ④ 可观测层                                                           │
│  ├─ otel-collector  收 traces                                         │
│  ├─ jaeger          看 traces                                          │
│  └─ pgadmin         看 DB                                              │
│  ⑤ 邮件                                                               │
│  └─ maildev         dev 期间收发邮件 (邀请/通知)                       │
└─────────────────────────────────────────────────────────────────────┘

下面逐个讲.

① 接入层​

api (NestJS, port 3000)​

所有 REST 请求都从这里进来. 路径前缀统一 /api/.... 关键功能:

  • 用户 / 组织 / API Key 管理
  • Sandbox 生命周期 (create / start / stop / delete)
  • Snapshot 管理 (image 注册 + 状态机)
  • Runner 注册 + 调度
  • 认证: 同时支持 OIDC token (浏览器登录 → Dex 颁发) 和 API Key (dtn_xxx, 程序调用用)

启动时会:

  1. 检查/创建 default snapshot (daytonaio/sandbox:0.5.0-slim)
  2. 检查/创建 default runner (注册名为 "default")
  3. 跑 alembic 类似的 migration (RUN_MIGRATIONS=true)

proxy (port 4000)​

专门给 sandbox 预览 URL 用的反向代理. 工作原理:

LLM 在 sandbox 里跑 vite dev 监听 :5173. SDK 调 workspace.preview_url(5173) 返回:

http://5173-<sandbox-id>.proxy.example.com:4000

浏览器访问这个 URL → DNS 解析到 proxy 容器 → proxy 按 host header 解出 <sandbox-id> 和 port → 转发到 runner 内对应 sandbox 的 5173 端口.

PROXY_TEMPLATE_URL 这个 env 决定 URL 模板, 想用真实域名要改这个变量 + 配 wildcard DNS.

ssh-gateway (port 2222)​

让用户能 ssh -p 2222 <token>@host 直接进到自己的 sandbox 里. 不是必需 —— Code Agent 场景 LLM 走 process.exec() SDK 接口, 不需要 ssh. 但调试用得上.

dex (port 5556)​

OIDC Provider, 支持:

  • 静态用户 (dev@daytona.io / password 内置)
  • 外接 LDAP / Google / GitHub / 自定义 OIDC

dashboard 浏览器登录走 Dex; API key 不走 Dex, 直接 api 内置 ApiKeyStrategy 验.

② 调度层​

runner (port 3003, privileged)​

最复杂的一个. 它内部跑了一个完整的 docker daemon (DinD), 用来真的启动 sandbox 容器.

工作流:

  1. 注册自己到 api (启动时心跳 POST /api/runners/healthcheck)
  2. 持续 poll api 的 job 队列 (GET /api/jobs/poll)
  3. 收到 INSPECT_SNAPSHOT_IN_REGISTRY job → 用 docker manifest inspect 拿镜像元数据
  4. 收到 PULL_SNAPSHOT job → docker pull 镜像
  5. 收到 CREATE_SANDBOX job → docker run 启动一个 sandbox 容器
  6. 收到 EXEC job → docker exec 在 sandbox 内跑命令
  7. 收到 DELETE_SANDBOX job → docker rm -f

为什么 privileged: 内嵌 docker daemon 需要 cgroup / namespace 操作权限. 这是安全敏感点, 生产部署应该用 gVisor / kata-containers 加固.

关键 env:

  • DAYTONA_API_URL=http://api:3000/api —— runner 跟 api 怎么通信
  • DAYTONA_RUNNER_TOKEN=secret —— 内部认证 token
  • RESOURCE_LIMITS_DISABLED=true —— OSS 默认禁了 cgroup 限制 (DinD 限制), 生产应该开

Snapshot 体系 (api 内)​

Daytona 的 image 不直接叫 image, 叫 Snapshot. 一个 Snapshot 经历的状态:

pending → (runner inspect 成功) → active
   ↓
   → (inspect/pull 失败) → error

Snapshot vs SnapshotRunner 两层概念:

  • Snapshot: image 元数据 (org-scoped, 一份)
  • SnapshotRunner: 某个 runner 上有这个 snapshot 的副本 (per-runner, N 份)

新 sandbox 要创建时, api 找已经 READY 状态的 SnapshotRunner, 把任务派给那个 runner.

③ 存储层​

db (postgres:18)​

最重要的表:

  • organization —— 多租户的组织
  • user / api_key —— 用户和 API key
  • sandbox —— 当前活着的 sandbox
  • snapshot —— image 元数据
  • snapshot_runner —— snapshot ↔ runner 的副本表
  • job —— 任务队列 (跟 redis 配合)
  • region —— 部署区域 (OSS 默认就一个 us)

redis​

  • Job 队列 (BRPOP 长轮询)
  • 速率限制计数器
  • WebSocket 用户在线状态 (Notification Gateway)

registry (docker registry:2.8.2)​

内置 docker registry, 端口 6000. 三个用途:

  1. transient registry: 用户用 sandbox 跑 build 后的产物 image, 临时存这里
  2. internal registry: Daytona 自己的 image (sandbox base / runner ...) cache 这里
  3. 给 runner 拉 image 时优先走这里 (避开公网 docker.io 的网络问题)

minio (S3 兼容)​

  • 用户 volume 数据
  • sandbox 日志 (telemetry/logs)
  • snapshot 备份

④ 可观测层​

不是必需, 但默认 docker-compose 都跑了:

  • otel-collector: OTel collector, 收所有服务的 traces/metrics
  • jaeger: 看 trace UI (port 16686)
  • pgadmin: 看 DB UI (port 5050, dev@daytona.io / pgadmin)

生产部署可以全删, 接到自家的 SigNoz / Datadog / Grafana 就行.

⑤ 邮件​

maildev (port 1080)​

dev 期间假装一个 SMTP server, 把 daytona 发的邀请/重置密码邮件收到内存里, UI 上看. 生产要换成真 SMTP.

容器之间的网络拓扑​

                 daytona-network (bridge)
   ┌──────────────────────────────────────────────────┐
   │                                                   │
   │  api   ←→ db                                      │
   │   ↓ ↑    ↑                                        │
   │   ↓ │    └──── pgadmin                            │
   │   ↓ │                                             │
   │   ↓ ←→ redis ←──── proxy ←──── ssh-gateway        │
   │   ↓                                               │
   │   ↓ ←→ runner ←→ registry                         │
   │   ↓                                               │
   │   ↓ ←→ dex (OIDC)                                 │
   │   ↓                                               │
   │   ↓ ←→ minio (S3)                                 │
   │   ↓                                               │
   │   ↓ ←→ otel-collector ←─── jaeger                 │
   │                                                   │
   └──────────────────────────────────────────────────┘
                           ↑
                           │ docker DNS (服务名)
                           │
              所有容器互相走服务名通信:
              http://api:3000 / http://dex:5556 / etc

关键约束: 所有容器不要写死 localhost 或 IP, 都用服务名. 唯一例外是 OIDC URL —— 因为 OIDC issuer 字符串需要在 token 里跟外部访问的 URL 一致, 所以 OIDC_ISSUER_BASE_URL / PUBLIC_OIDC_DOMAIN 都要写外部能访问的真实域名 (例如 https://daytona-dex.example.com/dex).

资源占用 (跑空)​

跑起来不开 sandbox 的占用大约:

资源占用
内存~2-3 GB (14 容器加起来)
磁盘~2-4 GB (镜像 + 初始 db)
CPUidle ~3-5%

每开一个 sandbox 默认 2 GB 内存 / 1 CPU, 所以跑 N 个 sandbox 大概占 (基线) + N×2GB 内存.
