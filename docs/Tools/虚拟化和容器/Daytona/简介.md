> 一句话: **Daytona 是给 AI Agent 用的"远程一次性 Linux 工作站"** —— LLM 可以在里面写代码、跑 npm/pip、起 dev server、用完即销毁, 而不需要你的宿主机承担任何风险.

## 它解决一个具体问题

当你做"代码生成 Agent"(类似 Bolt.new / Lovable / Devin) 时, LLM 写出来的代码必须在某个**隔离的 Linux 环境**里:

1. **真的能跑** —— 不是 sandbox 玩具, 要能 `npm install` / `cargo build` / `pip install` 真东西
2. **互相隔离** —— 用户 A 装的恶意 npm package 不能影响用户 B
3. **能给预览 URL** —— LLM 起了 `vite dev` 在 :5173, 怎么变成一个公网可访问的地址给前端 iframe?
4. **能保留状态** —— 用户走开 30 分钟回来, 代码还在, dev server 还在, 不需要重头装依赖
5. **能秒级启停** —— 用户活跃时启, 走开 hibernate 不计费, 回来 wake up 几百 ms

如果不用 Daytona, 你要自己造这套基础设施: docker / firecracker / gVisor 选型 + reverse proxy + volume 持久化 + 容器生命周期管理 + 资源配额 + 网络隔离 ... 通常需要一个工程师全职做 2-3 个月.

Daytona 把这 6 件事**打包成一个 docker-compose stack**, 接 SDK 一行调用就能创建一个独立沙箱.

## 跟 Docker / Codespaces / E2B 的区别

| | Docker (裸用) | GitHub Codespaces | E2B | **Daytona** |
|---|---|---|---|---|
| **谁是终端用户** | 工程师 | 工程师 | LLM Agent | **LLM Agent** |
| **隔离粒度** | 容器 | 容器 (per repo) | 容器 (per session) | **容器 (per session)** |
| **预览 URL** | 自己起 nginx | 自带 *.github.dev | SDK 给 | **SDK 一行 `workspace.preview_url(port)`** |
| **持久化** | 自己管 volume | 7 天不用回收 | 默认无 | **hibernate + 状态保留** |
| **可自托管** | ✅ | ❌ | ❌ (云只) | **✅ (开源 docker-compose)** |
| **AI 编排接入** | 自己造 | 不为此设计 | Python/JS SDK | **Python/JS SDK + REST 完整** |
| **许可** | Apache 2.0 | 闭源 | 闭源 | **AGPL-3.0 (使用方不受传染, 改源码并部署才传染)** |

简单说: **Codespaces** 给人用 / **E2B** 给 AI 用但只云上 / **Daytona** = 给 AI 用 + 可自托管.

## 3 个典型场景

### 1. Code Agent (本系列主用例)

LLM 写一个完整的 React/Vue/Next 应用 → 在 Daytona 里 `npm run build` → 拿 dist/ 推到生产 → 给小白用户一个 24h 在线 URL.

```
用户一句话 "做个咖啡店官网"
   ↓
LLM (在 Daytona sandbox 内): 写 src/App.tsx, 调 npm install, 调 npm run build
   ↓
拿到 dist/ → 推到静态托管 → 给用户 cafe-x7k.example.com
```

### 2. AI 编码助手的"试验沙箱"

Cursor / Cline 之类的 IDE 助手让 AI 跑测试 / 装包 / 改代码, 但**怕它把你本地搞坏** —— 把这些操作都路由到一个临时 Daytona workspace, 玩坏了 `delete()` 一下没事.

### 3. RAG / Tool Use 中的"代码执行工具"

你给 LLM 一个 `python_exec(code)` 工具, 让它在 Daytona sandbox 里跑 Python 算数据分析 / 画图 / 调 API. 一次调用一个 sandbox, 跑完就销毁, 不用担心 LLM 写出 `os.system("rm -rf /")` 这种东西打到你的服务器.

## 何时不要用

- **只是跑普通 Web 后端** —— 杀鸡用牛刀, 直接 docker-compose 就行
- **静态托管** —— Daytona 的 sandbox 是开发态的, 生产 URL 应该走专门的静态托管 (Nginx / Caddy / Cloudflare)
- **需要 GPU 推理** —— Daytona 默认 sandbox 是 CPU, 想跑 GPU 要特殊配置 image (有但不是首选场景)
- **极致冷启动延迟** —— 沙箱启动 ~几秒, 如果你需要 < 100ms 响应, 用预热池或换轻量方案 (Cloudflare Workers / WASM)

## 关于"自托管 vs 云"

Daytona 提供两种部署:

- **Daytona Cloud** (SaaS): 注册即用, 按用量付费, 零运维
- **Daytona OSS** (自托管): 一份 docker-compose 自己跑, 长期省钱 + 数据不出门

OSS 部署文档明确写了 "**not safe to use in production**" —— 是给开发用的参考栈, 想用作生产要做加固 (替密钥/加备份/限权限/上 HTTPS). MVP 阶段直接 OSS 跑通方向, 量起来后再考虑迁 Cloud 或彻底加固自托管.
