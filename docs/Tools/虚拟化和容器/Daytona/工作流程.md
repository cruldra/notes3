> 跟着一次 SDK 调用走一遍 —— 从 `client.create()` 到 sandbox 起来跑 `echo hello`, 中间到底发生了什么.

## 用户视角的 SDK 调用

```python
from daytona import Daytona, DaytonaConfig

cfg = DaytonaConfig(
    api_url="https://daytona-api.example.com/api",
    api_key="dtn_xxx",
)
client = Daytona(cfg)

sb = client.create()                      # ① 创建 sandbox
print(sb.id, sb.state)                    # → SandboxState.STARTED

r = sb.process.exec("echo hello")         # ② 跑命令
print(r.exit_code, r.result)              # → 0, "hello\n"

url = sb.preview_url(5173)                # ③ 拿预览 URL
print(url)                                # → http://5173-<id>.proxy.example.com:4000

sb.delete()                               # ④ 销毁
```

下面看背后的 4 个阶段.

## 阶段 ①: 创建 sandbox

```
SDK                    api               db              runner          docker
 │                      │                 │                │              │
 │ POST /api/sandbox    │                 │                │              │
 ├─────────────────────►│                 │                │              │
 │                      │ 验证 api_key    │                │              │
 │                      │ 找 default region│                │              │
 │                      │ 找 active snapshot│               │              │
 │                      │ INSERT sandbox  │                │              │
 │                      ├────────────────►│                │              │
 │                      │ INSERT job (CREATE_SANDBOX)       │              │
 │                      ├────────────────►│                │              │
 │ 200 {id, state:CREATING}              │                │              │
 │◄─────────────────────│                 │                │              │
 │                                        │                │              │
 │                                        │ runner BRPOP   │              │
 │                                        │◄───────────────┤              │
 │                                        │                │              │
 │                                        │                │ docker run   │
 │                                        │                ├─────────────►│
 │                                        │                │              │
 │                                        │                │ 容器 ID, 端口 │
 │                                        │                │◄─────────────┤
 │                                        │                │              │
 │                                        │ POST /jobs/done │              │
 │                                        │◄───────────────┤              │
 │                                        │ UPDATE sandbox state=STARTED   │
 │                                        │                │              │
 │ (SDK 轮询 /sandbox/:id)                │                │              │
 │ GET /api/sandbox/:id                   │                │              │
 ├─────────────────────►│                 │                │              │
 │ 200 {state: STARTED, ip: ...}          │                │              │
 │◄─────────────────────│                 │                │              │
```

**关键点**:

1. **api 不直接动 docker**, 走"job 队列 + runner 异步处理"模型. 这样 api 可以无状态, runner 可以横向扩 N 个.
2. **state machine**: `CREATING → STARTED → STOPPED → DELETED`. SDK 客户端通过轮询拿到 STARTED 才返回.
3. **没有 default region 会失败** —— OSS 部署后 organization 必须先绑定一个 region (默认有个 `us` shared region, 但要手动 PATCH 到 organization).

## 阶段 ②: 在 sandbox 内跑命令

`sb.process.exec("echo hello")` 流程:

```
SDK                api                  runner             sandbox container
 │                  │                    │                   │
 │ POST /sandbox/:id/exec                │                   │
 │ {cmd: "echo hello"}                   │                   │
 ├─────────────────►│                    │                   │
 │                  │ INSERT job (EXEC)  │                   │
 │                  │                    │                   │
 │                  │ runner poll picks  │                   │
 │                  │◄───────────────────┤                   │
 │                  │                    │ docker exec       │
 │                  │                    ├──────────────────►│
 │                  │                    │ stdout/stderr     │
 │                  │                    │◄──────────────────┤
 │                  │                    │ POST /jobs/done   │
 │                  │◄───────────────────┤                   │
 │ 200 {exit_code, stdout, stderr}       │                   │
 │◄─────────────────│                    │                   │
```

每次 `exec` 都过一遍这个链路. **同步阻塞**到命令跑完. 长时间命令 (像 `npm install`) 会跑几十秒, SDK 期间持有 HTTP 连接等着.

## 阶段 ③: 预览 URL (开发期看效果)

```
LLM 在 sandbox 内: nohup npm run dev &
                    (vite dev 监听 :5173)
        ↓
SDK 调 preview_url(5173)
        ↓
api 返回: http://5173-{sandbox_id}.proxy.example.com:4000
        ↓
浏览器访问这个 URL
        ↓
DNS 解析 → proxy 容器 :4000
        ↓
proxy 解析 host header:
  "5173-{sandbox_id}.proxy.example.com" 
  → port=5173, sandbox_id=...
        ↓
proxy 找到 sandbox 在哪个 runner
        ↓
proxy 转发请求到 sandbox 容器的 :5173 端口
        ↓
vite 返回 HTML/JS
        ↓
浏览器渲染 + WebSocket (HMR) 也走这条链路
```

**关键限制**:
- preview URL 只在 sandbox running 时有效, hibernate 后 URL 失效
- WebSocket 也支持 (vite HMR 必需)
- HTTPS 需要在 proxy 前面再套 Caddy/Nginx 终结 TLS (proxy 自己只听 HTTP)
- 想要 wildcard 通配证书 (`*.proxy.example.com`) 需要 ACME DNS-01 challenge

## 阶段 ④: 销毁

```
sb.delete()
   ↓
DELETE /api/sandbox/:id
   ↓
api → INSERT job (DELETE_SANDBOX)
   ↓
runner picks → docker rm -f <container_id>
   ↓
api → UPDATE sandbox state=DELETED, deletedAt=now()
   ↓
(可选) 异步清理 volume / 释放端口
```

软删除 —— 数据库行还在但 state=DELETED. 真正 GC 由 cron job 定期跑.

## hibernate / wake 流程

跟 delete 不同, hibernate **保留容器状态**:

```
sb.hibernate()
   ↓
api → INSERT job (HIBERNATE)
   ↓
runner: docker stop <id>  (不是 rm)
   ↓
sandbox state=STOPPED
   ↓
......
   ↓
sb.start()
   ↓
api → INSERT job (START)
   ↓
runner: docker start <id>
   ↓
sandbox state=STARTED, 文件系统 + 内存 (虽然内存其实没保留, 但 fs 保留) 都还在
```

这是给"用户走开 30 分钟回来"用的, 比 delete + recreate 快几个数量级.

## Snapshot 怎么变成 sandbox

新部署的 daytona 会自动创建 default snapshot, 走这个状态机:

```
docker-compose.yaml 设置: DEFAULT_SNAPSHOT=daytonaio/sandbox:0.5.0-slim
   ↓
api 启动时检查 db
   ↓ (db 里没有这个 snapshot)
api: INSERT snapshot {state: pending}
   ↓
api: INSERT job (INSPECT_SNAPSHOT_IN_REGISTRY)
   ↓
runner: docker manifest inspect daytonaio/sandbox:0.5.0-slim
   ↓ (成功 → 拿到 manifest digest)
api: INSERT job (PULL_SNAPSHOT)
   ↓
runner: docker pull daytonaio/sandbox:0.5.0-slim
   ↓ (成功)
api: UPDATE snapshot state=active, ref=daytonaio/sandbox:0.5.0-slim@<digest>
api: INSERT snapshot_runner {snapshotId, runnerId, state: READY}
```

之后所有新 sandbox 都从 active 的 snapshot 启动, 不用每次都 pull.

**坑**: 如果 inspect 失败 (网络问题 / 镜像不存在), snapshot 会停留在 `error` 状态, 永远不会自动重试. 必须人为 DELETE 这条 snapshot row 让 api 启动时重新 init, 或者改成 internal registry 的 image.

## 一次完整的 Code Agent 调用

把上面所有阶段串起来:

```python
# t=0s
sb = client.create()
# t=2-5s (snapshot 已 ready 时)

sb.process.exec("git clone https://github.com/foo/bar.git /workspace")
# t=10s

sb.process.exec("cd /workspace && npm install")
# t=60-120s (慢)

sb.process.exec("cd /workspace && nohup npm run dev > /tmp/dev.log 2>&1 &")
# t=125s

dev_url = sb.preview_url(5173)
print(f"Visit: {dev_url}")
# 用户浏览器看效果

# 用户走开
sb.hibernate()
# 此时不计费

# 用户回来
sb.start()
# t=resume + 0.5s, dev server 还在

sb.process.exec("cd /workspace && npm run build")
# 把 dist/ 推到生产

sb.delete()
# 完事
```
