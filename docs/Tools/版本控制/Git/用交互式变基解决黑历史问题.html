用交互式变基解决黑历史问题

问题描述

场景痛点​

在日常开发中，我们经常会遇到这样的尴尬场景：

  1. 自己手抖，或者使用了 AI 自动提交工具，连续产生了几个 Commit Message 完全不符合团队规范的“垃圾提交”。
  2. 更要命的是，这些“垃圾提交”并没有集中在最顶部，而是夹杂在其他正常的、优秀的提交记录中间。
  3. 我们想要删掉这些提交记录，但必须保留里面写的代码，并且不能影响前后其他的正常提交。

面对这种情况，很多可视化工具（如 Git Graph）的右键菜单并没有一键解决的方案：

  • ❌ 点击 Drop：连提交带代码一起彻底删除了，当场白干。
  • ❌ 使用 Reset (Soft/Mixed) 到之前的节点：虽然代码保住了，但会把你夹在中间的“好提交”也一起撤销，导致所有代码混成一团。

最完美、最干净的解决方案是使用命令行的 交互式变基（Interactive Rebase），配合 reword 和 fixup 命令。

核心解法：Interactive Rebase + Fixup​

Step 1: 启动交互式变基​

首先，找到你想整理的那堆提交之前的一个正常节点。你可以通过计算需要往前回溯的步数，在终端输入：

# 比如往前回溯 15 个提交
git rebase -i HEAD~15

或者直接指定那个正常提交的 Commit Hash：

git rebase -i <正常提交的Hash值>

Step 2: 神奇的 r 和 f 组合技​

执行命令后，终端会进入 Vim 编辑器，按时间顺序列出你最近的提交。默认前面都是 pick（保留该提交）。

很多教程会教你把不需要的提交改成 squash (或 s)。但 squash 有一个致命缺点：它会把所有合并的提交信息堆叠在一起弹出来，你需要手动一行行删掉那些废话，非常痛苦。

终极偷懒且干净的方案是使用 reword (r) 和 fixup (f)：

  • reword (r)：保留提交的代码，但要求重新编写 Commit Message。
  • fixup (f)：将该提交的代码合并到上一个提交中，并且直接丢弃它的 Commit Message。

操作示例： 按 i 键进入编辑模式，将你的列表改成类似这样：

pick d32b151 ✨ feat(auth): 正常的提交，保持不变

# --- 下面是需要合并重写的一组“黑历史” ---
r b94da12 docs: 糟糕的提交1 (设为 r，等下在这里重写总注释)
f 03e1c37 docs: 糟糕的提交2 (设为 f，代码合并，注释直接丢弃)
f 237b781 feat: 糟糕的提交3 (设为 f，丢弃注释)

# --- 中间的正常提交，不要动 ---
pick 89b01b0 ✨ feat(kf): 另一个正常的提交，保持不变

修改完成后，按 Esc 键退出编辑模式，输入 :wq 保存并退出。

Step 3: 重写干净的 Commit Message​

因为我们刚才设置了一个 r，Git 在处理到那里时，会自动弹出一个新的 Vim 窗口。

这个窗口里非常干净，只有那个 r 对应的旧注释，没有其他杂乱的信息。你只需要：

  1. 按 i 进入编辑模式。
  2. 删掉旧注释，写上一段完美的、符合规范的总结（例如：✨ feat(module): 完整实现 XXX 基础功能与数据模型）。
  3. 按 Esc，输入 :wq 保存退出。

如果你的列表中有多组 r 和 f 的组合，Git 就会弹框多次，你依次重写即可。全程清清爽爽，代码完美合并。

收尾工作：安全地推送到远程仓库​

由于我们使用了 Rebase，本地的 Git 历史已经被改写。如果这些“黑历史”之前已经 push 到过远程仓库（如 GitHub/GitLab），现在普通的 git push 会报错。

我们需要使用强制推送来覆盖远程历史。但是，强烈建议不要用暴力的 -f，而是使用带保险的强制推送：

git push --force-with-lease origin <你的分支名>

💡 为什么要用 --force-with-lease？ 它在覆盖远程代码前，会先检查一下远程仓库在你上次拉取之后，有没有其他人推送了新代码。如果有，它会拦截你的操作，防止你误删同事的心血。绝对是多人协作时的护身符！

总结： 遇到零散的垃圾提交不用慌，git rebase -i 配合 r 和 f，最后 --force-with-lease 兜底，你就可以永远在同事面前保持一份极其专业、干净的 Git 提交记录。
