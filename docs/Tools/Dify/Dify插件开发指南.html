Dify插件开发指南

选择你要开发的插件类型​

在开始之前，你需要了解插件类型的基本知识。插件支持在Dify中扩展以下能力：

  • 工具（Tool）：工具提供者，如Google搜索、Stable Diffusion等，它可以用来执行特定任务。
  • 模型（Model）：模型提供者，如OpenAI、Anthropic等，你可以使用他们的模型来增强AI能力。
  • 端点（Endpoint）：类似于Dify中的服务API和Kubernetes中的Ingress，你可以将http服务扩展为端点，并使用自己的代码控制其逻辑。

基于你要扩展的能力，我们将插件分为三种类型：工具（Tool）、模型（Model）和扩展（Extension）。

  • 工具（Tool）：它是一个工具提供者，但不仅限于工具，你也可以在那里实现端点，例如，如果你正在构建Discord机器人，你需要"发送消息"和"接收消息"，工具和端点都是必需的。
  • 模型（Model）：只是一个模型提供者，不允许扩展其他功能。
  • 扩展（Extension）：其他时候，你可能只需要一个简单的http服务来扩展功能，扩展是你的正确选择。

我相信你在创建插件时已经选择了正确的类型，如果没有，你可以稍后通过修改manifest.yaml文件来更改它。

清单文件（Manifest）​

现在你可以编辑manifest.yaml文件来描述你的插件，以下是它的基本结构：

  • version（版本，必需）：插件的版本
  • type（类型，必需）：插件的类型，目前只支持plugin，未来支持bundle
  • author（字符串，必需）：作者，这是市场中的组织名称，也应该等于仓库的所有者
  • label（标签，必需）：多语言名称
  • created_at（RFC3339，必需）：创建时间，市场要求创建时间必须小于当前时间
  • icon（资产，必需）：图标路径
  • resource（对象）：要申请的资源
    • memory（int64）：最大内存使用量，主要与SaaS上无服务器资源申请相关，单位字节
    • permission（对象）：权限申请
      • tool（对象）：反向调用工具权限
        • enabled（bool）
      • model（对象）：反向调用模型权限
        • enabled（bool）
        • llm（bool）
        • text_embedding（bool）
        • rerank（bool）
        • tts（bool）
        • speech2text（bool）
        • moderation（bool）
      • node（对象）：反向调用节点权限
        • enabled（bool）
      • endpoint（对象）：允许注册端点权限
        • enabled（bool）
      • app（对象）：反向调用应用权限
        • enabled（bool）
      • storage（对象）：申请持久存储权限
        • enabled（bool）
        • size（int64）：允许的最大持久内存，单位字节
  • plugins（对象，必需）：插件扩展特定能力yaml文件列表，插件包中的绝对路径，如果需要扩展模型，需要定义一个像openai.yaml这样的文件，并在这里填写路径，路径上的文件必须存在，否则打包将失败。
    • 格式
      • tools（list[string]）：扩展工具供应商，详细格式请参考工具指南
      • models（list[string]）：扩展模型供应商，详细格式请参考模型指南
      • endpoints（list[string]）：扩展端点供应商，详细格式请参考端点指南
    • 限制
      • 不允许同时扩展工具和模型
      • 不允许没有扩展
      • 不允许同时扩展模型和端点
      • 目前每种扩展类型只支持最多一个供应商
  • meta（对象）
    • version（版本，必需）：manifest格式版本，初始版本0.0.1
    • arch（list[string]，必需）：支持的架构，目前只支持amd64 arm64
    • runner（对象，必需）：运行时配置
      • language（字符串）：目前只支持python
      • version（字符串）：语言版本，目前只支持3.12
      • entrypoint（字符串）：程序入口，在python中应该是main

安装依赖​

  • 首先，你需要一个Python 3.11+环境，因为我们的SDK需要这个版本。
  • 然后，安装依赖： 
    pip install -r requirements.txt
  • 如果你想添加更多依赖，可以将它们添加到requirements.txt文件中，一旦你在manifest.yaml文件中将运行器设置为python，requirements.txt将自动生成并用于打包和部署。

实现插件​

现在你可以开始实现你的插件，通过以下示例，你可以快速了解如何实现自己的插件：

  • OpenAI：模型提供者的最佳实践
  • Google Search：工具提供者的简单示例
  • Neko：端点组的有趣示例

测试和调试插件​

你可能已经注意到插件根目录中有一个.env.example文件，只需将其复制到.env并填写相应的值，如果你想在本地调试插件，需要设置一些环境变量。

  • INSTALL_METHOD：将此设置为remote，你的插件将通过网络连接到Dify实例。
  • REMOTE_INSTALL_URL：来自Dify实例的plugin-daemon服务的调试主机和端口URL，例如debug.dify.ai:5003。可以使用Dify SaaS或自托管Dify实例。
  • REMOTE_INSTALL_KEY：你应该从使用的Dify实例获取调试密钥，在插件管理页面的右上角，你可以看到一个带有debug图标的按钮，点击它就会获得密钥。

运行以下命令启动你的插件：

python -m main

刷新你的Dify实例页面，你现在应该能够在列表中看到你的插件，但它会被标记为debugging，你可以正常使用它，但不建议用于生产环境。

发布和更新插件​

为了简化你的插件更新工作流程，你可以配置GitHub Actions，在创建发布版本时自动创建PR到Dify插件仓库。

前置条件​

  • 你的插件源仓库
  • Dify-plugins仓库的fork
  • 在你的fork中正确的插件目录结构

配置GitHub Action​

  1. 创建一个具有写权限的Personal Access Token到你的fork仓库
  2. 将其作为名为PLUGIN_ACTION的secret添加到你的源仓库设置中
  3. 在.github/workflows/plugin-publish.yml创建工作流文件

使用方法​

  1. 更新你的代码和manifest.yaml中的版本
  2. 在你的源仓库中创建一个发布版本
  3. 该action会自动打包你的插件并创建PR到你的fork仓库

优势​

  • 消除手动打包和PR创建步骤
  • 确保发布过程的一致性
  • 在频繁更新期间节省时间

详细的设置说明和示例配置，请访问：GitHub Actions工作流文档

打包插件​

最后，通过运行以下命令打包你的插件：

dify-plugin plugin package ./ROOT_DIRECTORY_OF_YOUR_PLUGIN

你将得到一个plugin.difypkg文件，就是这样，你现在可以将它提交到市场了，期待你的插件被列出！
