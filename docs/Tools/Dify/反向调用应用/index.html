反向调用应用

本文档详细介绍了插件如何反向调用Dify平台内的应用服务。它涵盖了三种类型的接口：聊天接口（用于聊天机器人/智能体/聊天流应用）、工作流接口和文本生成接口，为每种接口提供入口点、调用规范和实际代码示例。

反向调用应用意味着插件可以访问Dify内应用的数据。此模块支持流式和非流式应用调用。如果你对反向调用的基本概念不熟悉，请先阅读Dify服务反向调用。

接口类型​

对于聊天机器人/智能体/聊天流类型的应用，它们都是基于聊天的应用，因此共享相同的输入和输出参数类型。因此，它们可以统一视为聊天接口。

对于工作流应用，它们占用单独的工作流接口。

对于文本生成应用，它们占用单独的文本生成接口。

请注意，插件只允许访问插件所在工作空间内的应用。

调用聊天接口​

入口点​

self.session.app.chat

接口规范​

def invoke(
    self,
    app_id: str,
    inputs: dict,
    response_mode: Literal["streaming", "blocking"],
    conversation_id: str,
    files: list,
) -> Generator[dict, None, None] | dict:
    pass

当response_mode为streaming时，此接口将直接返回Generator[dict]。否则，它返回dict。具体的接口字段，请参考ServiceApi的返回结果。

使用案例​

我们可以在Endpoint中调用聊天类型应用并直接返回结果。

import json
from typing import Mapping
from werkzeug import Request, Response
from dify_plugin import Endpoint

class Duck(Endpoint):
    def _invoke(self, r: Request, values: Mapping, settings: Mapping) -> Response:
        """
        使用给定的请求调用端点。
        """
        app_id = values["app_id"]

        def generator():
            # 注意：原始示例错误地调用了self.session.app.workflow.invoke
            # 对于聊天应用，应该调用self.session.app.chat.invoke
            # 基于节标题，这里假设是聊天应用
            response = self.session.app.chat.invoke(
                app_id=app_id, 
                inputs={}, # 根据需要提供实际输入
                response_mode="streaming", 
                conversation_id="some-conversation-id", # 如果需要，提供会话ID
                files=[]
            )

            for data in response:
                yield f"{json.dumps(data)} <br>"

        return Response(generator(), status=200, content_type="text/html")
