Godot 的发布策略在不断演变中。以下的描述提供了大致的预期，但实际情况取决于核心贡献者的选择以及特定时期社区的需求。

## Godot 版本控制

Godot 大致遵循 [语义化版本控制 (Semantic Versioning)](https://semver.org/)，采用 `major.minor.patch`（主版本.次版本.补丁版本）的版本系统，但对每个术语的解释根据游戏引擎的复杂性进行了调整：

*   **`major` (主版本)**：当发生重大兼容性破坏，意味着将项目从一个主版本迁移到另一个主版本需要大量的移植工作时，增加此版本号。
    
    例如，将 Godot 项目从 Godot 3.x 移植到 Godot 4.x 需要通过转换工具运行项目，然后手动执行一些工具无法自动完成的进一步调整。
    
*   **`minor` (次版本)**：用于发布不破坏主要兼容性的新功能时增加。在次版本中 _可能_ 会发生非常特定区域的轻微兼容性破坏，但绝大多数项目应该不受影响或不需要大量的移植工作。
    
    这是因为 Godot 作为游戏引擎，涵盖了渲染、物理和脚本等许多领域。修复某个领域的错误或实现新功能有时可能需要更改特性的行为或修改类的接口，即使引擎 API 的其余部分保持向后兼容。
    
    > **提示**
    > 
    > 建议所有用户升级到新的次版本，但必须进行一些测试以确您的项目仍然按预期运行。

*   **`patch` (补丁版本)**：用于发布维护版本时增加，重点是修复错误和安全问题、实现新的平台支持要求以及向后移植安全的可用性增强。补丁版本是向后兼容的。
    
    补丁版本可能包含不影响现有 API 的小型新功能，因此没有影响现有项目的风险。
    
    > **提示**
    > 
    > 因此，更新到新的补丁版本被认为是安全的，并强烈建议给定稳定分支的所有用户进行更新。

我们将 `major.minor` 组合称为 **稳定分支 (stable branches)**。每个稳定分支都以一个 `major.minor` 发布开始（没有 `patch` 的 `0`），并在同名的 Git 分支中进一步开发维护版本（例如，4.0 稳定分支的补丁更新在 `4.0` Git 分支中开发）。

## 发布支持时间表

稳定分支 _至少_ 支持到下一个稳定分支发布并收到其第一个补丁更新为止。实际上，只要有活跃用户需要维护更新，我们就会以 _尽力而为_ 的基础支持稳定分支。

每当发布新的主版本时，我们将前一个稳定分支作为长期支持版本，并尽力为那些无法将复杂项目移植到新主版本的该分支用户提供问题修复。2.1 分支就是这种情况，3.x 分支也是如此。

在给定的次版本系列中，只有最新的补丁版本才会获得支持。如果您在使用旧的补丁版本时遇到问题，请升级到该系列的最新补丁版本并再次测试，然后再在 GitHub 上报告问题。

## 新项目应该使用哪个版本？

我们建议新项目使用 Godot 4.x，因为在未来 3.x 停止接收更新很久之后，Godot 4.x 系列仍将受到支持。一个需要注意的是，许多第三方文档尚未针对 Godot 4.x 进行更新。如果你必须遵循为 Godot 3.x 设计的教程，我们建议在一个单独的标签页中打开 [从 Godot 3 升级到 Godot 4](https://docs.godotengine.org/en/stable/tutorials/migrating/upgrading_to_godot_4.html#doc-upgrading-to-godot-4)，以便在尝试使用在 Godot 4.x 中已重命名的特定节点或方法时出现脚本错误时，检查哪些方法已被重命名。

如果你的项目需要 4.x 中缺失的功能（例如 GLES2/WebGL 1.0），那么你应该为新项目使用 Godot 3.x。

## 我应该升级我的项目以使用新的引擎版本吗？

> **注意**
> 
> 在开发项目时升级软件本身就有风险，因此在尝试升级之前，请考虑这对你的项目是否是个好主意。此外，请备份你的项目或使用版本控制，以防升级出错导致数据丢失。

即便如此，我们也会尽最大努力保持次要版本，尤其是补丁版本与现有项目的兼容性。

一般建议是将你的项目升级以跟随新的 **补丁** 版本，例如从 4.0.2 升级到 4.0.3。这确保你获得错误修复、安全更新和平台支持更新（这对移动平台尤为重要）。你还可以获得持续的支持，因为在官方社区平台上只有最新的补丁版本才会获得支持。

对于 **次要** 版本，你应该根据具体情况决定是否升级。我们在使升级过程尽可能无缝方面付出了很大努力，但在次要版本中可能会存在一些破坏性更改，以及更大的回归风险。次要版本中包含的一些修复也可能会为了修复某些错误而改变类的预期行为。这在文档中标记为 _实验性 (experimental)_ 的类中尤为常见。

**主要** 版本带来了许多新功能，但也删除了以前存在的功能，并可能提高硬件要求。与次要版本相比，升级到主要版本也需要更多的工作。因此，如果你对项目当前的运行方式感到满意，我们建议坚持使用你开始项目时使用的主要版本。例如，如果你的项目是使用 3.5 开始的，我们建议升级到 3.5.2 以及未来的 3.6，但不要升级到 4.0+，除非你的项目确实需要 4.0+ 带来的新功能。

## 下一个版本什么时候发布？

虽然 Godot 贡献者没有在任何截止日期下工作，但我们努力相对频繁地发布次要版本。

特别是，在 4.0 经历了很长的发布周期后，我们正转向快节奏的开发工作流程，4.1 在 4.0 发布 4 个月后发布，4.2 在 4.1 发布 4 个月后发布。

频繁的次要版本发布将使我们要么更快地发布新功能（可能作为实验性功能），快速获得用户反馈，并迭代以改进这些功能及其可用性。同样，总体用户体验将通过更快的路径到达最终用户而得到更稳定的改善。

维护（补丁）版本将根据需要发布，开发周期可能非常短，以便为当前稳定分支的用户提供满足其生产需求的最新错误修复。

目前没有下一个 3.x 次要版本 3.7 的计划发布日期。当前的稳定版本 3.6 可能是 Godot 3.x 的最后一个稳定分支。只要贡献者继续维护，Godot 3.x 就会在尽力而为的基础上得到支持。

## 跨引擎版本的兼容性标准是什么？

> **注意**
> 
> 本节旨在供贡献者确定哪些更改对于给定版本是安全的。该列表并非详尽无遗；它仅概述了 Godot 开发过程中遇到的最常见情况。

以下更改在 **补丁版本** 中是可以接受的：

*   以对大多数项目没有重大负面影响的方式修复错误，例如视觉或物理错误。Godot 的物理引擎不是确定性的，因此物理错误修复不被视为破坏兼容性。如果修复错误产生的负面影响可能会影响大量项目，则应将其设为可选（例如使用项目设置或单独的方法）。
    
*   向方法添加新的可选参数。
    
*   小规模的编辑器可用性调整。

请注意，我们倾向于对后续每个补丁版本中允许的修复更加保守。例如，4.0.1 可能会收到比 4.0.4 影响更大的修复。

以下更改在 **次要版本** 中是可以接受的，但在补丁版本中则不行：

*   重要的新功能。
    
*   重命名方法参数。在 C# 中，方法参数可以按名称传递（但在 GDScript 中不行）。因此，这可能会破坏一些使用 C# 的项目。
    
*   弃用方法、成员变量或类。这是通过向其类引用添加已弃用标志来完成的，该标志将显示在编辑器中。当方法被标记为已弃用时，它计划在下一个 _主要_ 版本中被移除。
    
*   影响默认项目主题视觉效果的更改。
    
*   显著改变行为或输出的错误修复，旨在更好地满足用户期望。相比之下，在补丁版本中，我们可能倾向于保留错误的行为，这样我们就不会破坏可能已经依赖该错误或使用变通方法的现有项目。
    
*   导致视觉变化的性能优化。

以下更改被视为 **破坏兼容性**，只能在新的 **主要版本** 中执行：

*   重命名或移除方法、成员变量或类。
    
*   通过使其继承自不同的类来修改节点的继承树。
    
*   以影响现有项目的方式更改项目设置值的默认值。若要仅影响新项目，项目管理器应改为编写修改后的 `project.godot`。

由于 Godot 5.0 尚未分支，我们目前不鼓励进行此类破坏兼容性的更改。

> **注意**
> 
> 以任何方式修改方法的签名（包括添加可选参数）时，必须创建一个 GDExtension 兼容性方法。这确保了现有的 GDExtensions 在补丁和次要版本中继续工作，以便用户不必重新编译它们。有关更多信息，请参阅 [处理兼容性破坏](https://docs.godotengine.org/en/stable/engine_details/development/handling_compatibility_breakages.html#doc-handling-compatibility-breakages)。
