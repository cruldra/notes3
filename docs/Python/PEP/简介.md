**PEP** 是 **Python Enhancement Proposal**（Python 增强提案）的缩写，是 Python 社区用来提议新特性、规范设计决策、记录信息的正式文档。任何对 Python 语言、CPython 实现、标准库、社区流程的重大变更，都需要经过一个 PEP 的撰写、讨论与审议流程才能落地。

PEP 的灵感来自 IETF 的 [RFC](https://www.rfc-editor.org/) 体系，但更侧重一个语言生态内部的设计协同。

## 官方资源

- 官方索引站点：[https://peps.python.org/](https://peps.python.org/)
- 索引仓库（PEP 0）：[https://peps.python.org/pep-0000/](https://peps.python.org/pep-0000/)
- 流程定义 PEP 1：[https://peps.python.org/pep-0001/](https://peps.python.org/pep-0001/)（介绍 PEP 的目的、流程、角色）
- 源码仓库：[https://github.com/python/peps](https://github.com/python/peps)

每一个 PEP 都有固定的编号，例如著名的 [PEP 8](https://peps.python.org/pep-0008/) 代码风格指南、[PEP 20](https://peps.python.org/pep-0020/) The Zen of Python。

## PEP 的三种类型

PEP 1 将 PEP 划分为三种类型：

| 类型 | 用途 | 例子 |
|------|------|------|
| **Standards Track** | 提议对语言、实现或标准库的新特性 | PEP 695（类型参数语法）、PEP 703（去 GIL） |
| **Informational** | 描述设计、风格、社区共识，无强制性 | PEP 8（代码风格）、PEP 20（Python 之禅） |
| **Process** | 规范 Python 项目自身的开发与决策流程 | PEP 1（PEP 流程）、PEP 13（治理结构） |

## PEP 的状态流转

一个 PEP 在生命周期中会经历以下主要状态：

```
Draft → Accepted → Final
       ↘ Rejected
       ↘ Withdrawn
       ↘ Deferred
       ↘ Superseded
```

- **Draft**：草案阶段，作者正在撰写与征求反馈
- **Accepted**：已被指导委员会（Steering Council）或 BDFL-Delegate 接受
- **Final**：相关代码已合入并在某个 Python 版本中发布
- **Rejected / Withdrawn / Deferred**：被拒、撤回、暂缓
- **Superseded**：被另一个 PEP 取代

## 为什么需要关注 PEP

- **理解新特性的来龙去脉**：任何 `match`、`async`、`type` 关键字背后都对应着一个 PEP，PEP 提供了完整的设计动机、备选方案、争议点
- **跟随语言的演进节奏**：Python 每年 10 月发布一个新版本，新增的语法、语义都通过 PEP 公开
- **参与社区**：PEP 是普通开发者向 Python 社区贡献设计想法的正规渠道
