PEP-735-依赖组

PEP 735 在 pyproject.toml 中引入了 [dependency-groups] 表，用来声明"项目维护者会用，但不该写进发行包元数据"的依赖集合（测试、文档、类型检查、linter、本地调试工具等）。它在 2024 年 10 月被接受（Status: Final），pip 25.1、uv、hatch、pdm、Poetry 等主流工具陆续提供 --group / --all-groups 支持。

设计目标：把过去散落在 requirements-dev.txt、被滥用的 [project.optional-dependencies]、Poetry tool.poetry.group.*、PDM tool.pdm.dev-dependencies 等位置的"开发期依赖"，收敛成一个工具无关、PEP 标准化的位置。

它解决了什么​

PEP 735 之前，"开发依赖"有几种主流写法，但都各有缺陷：

写法问题
[project.optional-dependencies]（PEP 621 extras）这是面向使用者的可选特性（pip install mypkg[plotting]），把 pytest、mypy 写进去会污染包元数据，使用者 pip show 时也能看到不该暴露的工具。
requirements-dev.txt 等独立文件游离在 pyproject.toml 之外，缺少跨工具一致性，也无法与 [project] 段共享 PEP 508 解析。
tool.poetry.group.* / tool.pdm.dev-dependencies / Hatch envs各家自定义键名，换工具就得重写依赖声明，无法在不同包管理器之间互通。

PEP 735 给出第三选项：[dependency-groups] 表本身就是 PEP 标准化的字段，且不写入构建产物的 METADATA——消费者执行 pip install mypkg 不会看到它。

基本语法​

[dependency-groups]
test = ["pytest>=8", "pytest-cov", "hypothesis"]
docs = ["sphinx", "myst-parser", "furo"]
typing = ["mypy", "types-requests"]
lint = ["ruff", "pre-commit"]

每个键是组名（按 PEP 508 normalize：连字符/下划线/点等价、大小写不敏感），值是 PEP 508 依赖字符串数组。组之间彼此独立，不会被自动暴露给安装者。

组合：include-group​

PEP 735 允许一个组引用另一个组，形成层叠：

[dependency-groups]
test = ["pytest", "hypothesis"]
typing = ["mypy"]

# dev 组聚合所有日常开发需要的内容
dev = [
    {include-group = "test"},
    {include-group = "typing"},
    "ipython",
    "pdbpp",
]

规则：

  • 禁止循环引用：A → B → A 会被工具拒绝。
  • 是展开而不是引用，resolver 最终看到的是去重后的扁平 PEP 508 列表。
  • 只能引用本文件内的组，不能跨项目。

与 [project.optional-dependencies] 的对比​

维度[project.optional-dependencies]（extras）[dependency-groups]（PEP 735）
谁会用到包的最终用户：pip install pkg[plotting]包的维护者/贡献者：pip install --group dev
是否进入 wheel 元数据是（Provides-Extra 字段）否
适合声明真正的可选特性（数据库驱动、可视化后端、加速器）测试、文档、类型检查、CI、本地调试工具
是否要求 [project] 段是（必须有项目元数据）否（也可用于非项目的 pyproject.toml）
是否支持引用其它集合不支持支持 include-group

判断口径很简单：如果发布到 PyPI 的用户安装时会用到这个依赖，写到 extras；否则写到 dependency-groups。

工具支持​

# pip ≥ 25.1
pip install --group dev
pip install --group test --group docs

# uv
uv sync --group dev          # 安装项目 + dev 组
uv sync --all-groups         # 安装项目 + 所有组
uv add --group test pytest   # 把 pytest 写入 test 组

# pdm / hatch / Poetry 也已陆续对接（具体命令参见各家文档）

非项目模式也是合法用法（PEP 735 明确支持）：

# 整个 pyproject.toml 中不写 [project] 段也 OK
[dependency-groups]
linters = ["ruff", "black"]
scripts = ["typer", "rich"]

适合 monorepo 仓库根、纯文档仓库、脚本集合等"不打包但仍要锁依赖"的场景。

三类依赖对照​

[project]
name = "mylib"
dependencies = [
    "httpx>=0.27",          # ① 运行时强依赖：所有用户都装
]

[project.optional-dependencies]
postgres = ["psycopg[binary]"]   # ② 可选特性：用户主动 pip install mylib[postgres]

[dependency-groups]
test = ["pytest", "pytest-asyncio"]
docs = ["sphinx"]
dev = [                          # ③ 开发组：仅维护者本地或 CI 使用
    {include-group = "test"},
    {include-group = "docs"},
    "ipython",
]
字段进入 METADATA用户能否安装维护者能否安装
dependencies✅✅（默认）✅
optional-dependencies✅（Provides-Extra）✅（带 extras）✅
dependency-groups❌❌✅（--group 系列命令）

边界与限制​

  • 不支持组级 environment markers：标记只能写在每条 PEP 508 字符串里（如 "pywin32; platform_system == 'Windows'"），不能在组上加 markers = "..."。
  • 不能与 extras 互转：include-group 不能引用 extras，extras 也不能引用 group——它们是两条平行的概念。
  • 没有"默认组"概念：必须由安装命令显式指定 --group，否则不会被装上（这是有意为之，避免 pip install 时意外拉一堆 dev 依赖）。

小结​

  • [dependency-groups] 是工具无关的"开发依赖"标准位置，替代各家私有的 dev-deps 写法。
  • 与 extras 的核心差异：不进入发行元数据，仅供维护侧用。
  • include-group 提供层叠组合，便于用 dev 聚合 test + typing + docs。
  • 主流安装器（pip 25.1+、uv、pdm、hatch）通过 --group 选项使用，迁移成本低。
