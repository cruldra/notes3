PEP-487-类自定义简化

PEP 487 在 Python 3.6 中引入了两个新的"类钩子"——__init_subclass__ 与 __set_name__，目的是让"在类定义阶段做点事情"这种需求不再被迫写元类（metaclass）。它把过去 90% 需要 metaclass 才能完成的工作（注册子类、校验子类约定、给描述符注入它自己的名字）下放成普通的类方法，写起来像普通继承，调试起来像普通函数。

设计动机：metaclass 解决了真问题，但代价过高——它影响 MRO、与多继承中的元类冲突、让阅读者必须先理解"类的类"这一层抽象。PEP 487 的判断是：绝大多数 metaclass 用法只是想在子类定义那一刻插一段代码，那就直接给"子类被定义"这一时机一个普通的钩子即可。

两个新钩子​

__init_subclass__(cls, **kwargs)​

定义在父类上，每当一个子类被创建时被调用，cls 是新建的子类本身。它是隐式的 classmethod，不需要写 @classmethod 装饰器。

class PluginBase:
    plugins: list[type] = []

    def __init_subclass__(cls, **kwargs) -> None:
        super().__init_subclass__(**kwargs)
        PluginBase.plugins.append(cls)

class JsonPlugin(PluginBase): ...
class YamlPlugin(PluginBase): ...

print(PluginBase.plugins)
# [<class '__main__.JsonPlugin'>, <class '__main__.YamlPlugin'>]

注意三点：

  • super().__init_subclass__(**kwargs) 一般要调用，否则会破坏多继承下的协作链。
  • 父类自身不会触发它——只在更下游的子类定义时触发。
  • 它能接收类定义时的关键字参数：
class APIView:
    def __init_subclass__(cls, *, route: str, **kwargs) -> None:
        super().__init_subclass__(**kwargs)
        cls.route = route

class UserView(APIView, route="/users"):
    ...

print(UserView.route)  # /users

这套语法过去只有 metaclass 才能做到（class UserView(APIView, metaclass=...) 通过 __init_subclass__ 之外的渠道传参）。

__set_name__(owner, name)​

定义在描述符（descriptor）上，类被创建时，Python 会对类体里每个实现了 __set_name__ 的属性调用一次，把它所属的类和自己被赋给的属性名告诉它。

class Field:
    def __set_name__(self, owner: type, name: str) -> None:
        self.owner = owner
        self.name = name
        self.private_name = f"_{name}"

    def __get__(self, instance, owner=None):
        if instance is None:
            return self
        return getattr(instance, self.private_name)

    def __set__(self, instance, value) -> None:
        setattr(instance, self.private_name, value)

class User:
    username = Field()
    email = Field()

print(User.username.name)  # 'username'
print(User.email.name)     # 'email'

在 PEP 487 之前，描述符想知道自己叫什么名字，要么靠用户手动传字符串（username = Field("username")，违反 DRY），要么靠 metaclass 在类创建后扫一遍 __dict__ 回填。__set_name__ 把这件事内建化了。

替代 metaclass 的典型场景​

下面是 PEP 487 之前必须写 metaclass、之后只用 __init_subclass__ 就够了的几类需求：

需求metaclass 方案PEP 487 方案
自动注册子类（插件、ORM 模型）class Meta(type): def __init__(cls, ...): registry.append(cls)父类上写 __init_subclass__
校验子类必须实现某接口metaclass __init__ 中检查 cls.__dict____init_subclass__ 中 assert
通过类定义关键字传参metaclass __new__(mcs, name, bases, ns, **kw)__init_subclass__(cls, **kw)
给类属性注入"自己的名字"metaclass 扫描 __dict__ 调 attr.set_name = k描述符上写 __set_name__

什么时候仍然需要 metaclass：

  • 改变类自身的实例化行为（即"调用类得到实例"这一步要做手脚）。
  • 干预 isinstance / issubclass（虚基类那一层）。
  • 控制类命名空间的构造（用 __prepare__ 返回 OrderedDict 之类）。

绝大多数业务代码遇到的"我希望子类一被定义就……"，PEP 487 已经覆盖。

调用顺序的细节​

类创建时，Python 大致按这个顺序执行：

  1. 执行类体（class body）→ 生成命名空间 ns。
  2. 调用元类 type(name, bases, ns) 创建类对象 cls。
  3. 对 ns 中每个有 __set_name__ 的值，调用 value.__set_name__(cls, attr_name)。
  4. 调用最近的 super().__init_subclass__(cls, **kwargs)。

这意味着：

  • __set_name__ 先于 __init_subclass__——__init_subclass__ 里能直接看到所有描述符已经"知道自己叫什么"。
  • 关键字参数只交给 __init_subclass__，不会透传到 __set_name__。

与 dataclasses / typing 的关系​

理解 PEP 487 后再回头看几个标准库特性，它们的实现就不再神秘：

  • dataclasses.dataclass 的"类装饰器"版本本质上做了一件事：在类定义之后扫 __annotations__ 生成 __init__ / __repr__。它没有用 metaclass，因为 PEP 487 之后的钩子已经够用。
  • typing.Generic 历史上用 metaclass，3.7 之后大量改造为基于 __init_subclass__。
  • enum.Enum 仍然是 metaclass，因为它需要拦截"用类成员的方式定义实例"这种构造行为，超出了 PEP 487 的能力范围。

判断标准：只要不需要改变"调用类 = 创建实例"这件事本身，就用 __init_subclass__，不要碰 metaclass。

易错点​

  1. 忘记调 super().__init_subclass__()。多继承时第二层父类的钩子会被无声跳过。
  2. 在 __init_subclass__ 里访问类属性时把"父类调用 / 子类调用"搞混。cls 始终是触发钩子的那个子类，不是定义钩子的父类。
  3. 把 __init_subclass__ 写成普通方法。它隐式是 classmethod，不要再加 @classmethod，否则在某些 mypy 配置下会重复装饰。
  4. __set_name__ 抛异常会让整个类创建失败。描述符里做 I/O、做网络调用是高危行为——name 阶段就崩，整张类都用不了。
