Python 远程调试（debugpy + Docker）

像 Java 的 JDWP 一样，把本地 IDE 接到远程 / 容器里跑着的 Python 进程上下断点。debugpy 是微软维护的 DAP（Debug Adapter Protocol）实现，VSCode 原生支持，PyCharm 2024.2+ 也通过 "Attach to DAP" 支持。

整体思路​

本地 IDE (DAP 客户端)  ──TCP──▶  容器里的 debugpy server
  • 容器内 debugpy.listen((host, port)) 监听端口
  • 宿主机用 Docker 映射端口到外网
  • 本地 IDE 通过 IP:Port 连接，走 DAP 协议握手 → 下断点 → 命中

一、容器侧改造​

1. Dockerfile 安装 debugpy​

镜像里装就行，本地 IDE 不需要装 —— IDE 自带 DAP 客户端。

RUN uv pip install debugpy
# 或者用 pip install debugpy

2. 共享的引导模块​

按环境变量决定是否启动监听，平时不影响生产：

# core/debugpy_bootstrap.py
"""远程调试引导。

环境变量:
    DEBUGPY_ENABLE: "1" 开启监听
    DEBUGPY_HOST:   监听地址，默认 0.0.0.0
    DEBUGPY_PORT:   监听端口
    DEBUGPY_WAIT:   "1" 阻塞等待 IDE 连接（用于调启动期问题）
"""
import os


def enable_debugpy(default_port: int = 5678) -> None:
    if os.getenv("DEBUGPY_ENABLE") != "1":
        return

    try:
        import debugpy
    except ImportError:
        print("[debugpy] DEBUGPY_ENABLE=1 但 debugpy 未安装，跳过", flush=True)
        return

    host = os.getenv("DEBUGPY_HOST", "0.0.0.0")
    port = int(os.getenv("DEBUGPY_PORT", str(default_port)))

    try:
        debugpy.listen((host, port))
        print(f"[debugpy] 监听 {host}:{port}", flush=True)
    except RuntimeError as exc:
        # 多 worker / reload 场景下重复 listen 会抛错，忽略即可
        print(f"[debugpy] listen 失败（可能已在监听）: {exc}", flush=True)
        return

    if os.getenv("DEBUGPY_WAIT") == "1":
        print("[debugpy] 等待 IDE 连接…", flush=True)
        debugpy.wait_for_client()
        print("[debugpy] IDE 已连接", flush=True)

3. 在最早入口调用​

# main.py (FastAPI)
from luffy_agent.core.debugpy_bootstrap import enable_debugpy
enable_debugpy(default_port=5678)  # 必须在其它代码之前

# celery_app.py
import os
from luffy_agent.core.debugpy_bootstrap import enable_debugpy
_role = os.getenv("CELERY_ROLE", "worker")
enable_debugpy(default_port=5680 if _role == "beat" else 5679)

不同角色用不同端口，避免同一主机多角色冲突。

4. docker-compose 暴露端口​

services:
  api:
    image: my-backend:latest
    ports:
      - "8000:8000"
      - "5678:5678"          # 调试端口
    environment:
      DEBUGPY_ENABLE: ${DEBUGPY_ENABLE:-0}
      DEBUGPY_WAIT:   ${DEBUGPY_WAIT:-0}
      DEBUGPY_PORT:   "5678"

  worker:
    image: my-backend:latest
    ports:
      - "5679:5679"
    environment:
      CELERY_ROLE: worker
      DEBUGPY_ENABLE: ${DEBUGPY_ENABLE_WORKER:-0}
      DEBUGPY_PORT: "5679"

5. 启动调试​

DEBUGPY_ENABLE=1 docker compose up -d api
docker logs api 2>&1 | grep debugpy
# [debugpy] 监听 0.0.0.0:5678

二、VSCode 客户端​

仓库根放 .vscode/launch.json：

{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Attach Remote API",
      "type": "debugpy",
      "request": "attach",
      "connect": { "host": "192.168.1.100", "port": 5678 },
      "pathMappings": [
        {
          "localRoot": "${workspaceFolder}/backend",
          "remoteRoot": "/app"
        },
        {
          "localRoot": "${workspaceFolder}/lib",
          "remoteRoot": "/lib"
        }
      ],
      "justMyCode": false
    }
  ]
}

pathMappings 必须对齐 —— Dockerfile 里 COPY backend/src/ ./src/ 会让容器内路径变成 /app/src/...，本地 backend/src/...，前缀映射是 ${workspaceFolder}/backend ↔ /app。

F5 → 选配置 → 秒连。

三、PyCharm 客户端（坑较多）​

1. 开启 Registry 标志​

PyCharm 默认隐藏了 DAP 支持，必须先打开：

  • Help → Find Action → 搜 Registry → 打开
  • 勾上 debugpy.dap.is.enable
  • 可选：debugpy.dap.debug.port 大于1才能连
  • 重启 PyCharm

否则新建 Run/Debug Configuration 时根本看不到 Attach to DAP 项，或者按钮一直是灰的。

官方支持文章：Attach to DAP configuration is greyed out in PyCharm

2. 新建 Attach to DAP 配置​

字段值
Name自定义
Remote address192.168.1.100:5678
Local project path/path/to/repo/backend
Remote project path/app

3. 项目必须配置 Python interpreter​

DAP 按钮灰的另一个常见原因：项目没有有效 interpreter。打开 Project Structure → Python Interpreter 配一个本地 venv 即可，不要求版本和远程一致。

4. 已知缺陷​

PyCharm 的 DAP 还处于 preview：

  • 连接稳定性差，断开后经常重连不上 → 重启远端容器才能恢复
  • 不支持 DataFrame / 数组可视化等部分数据检查功能
  • Debug adapter initialization timeout 偶发，原因常是上次连接没正常释放 → 重启容器

建议日常调试用 VSCode，PyCharm 当备用。

四、网络与安全​

公网暴露不安全​

debugpy 没有任何鉴权，直接开 5678 到公网等于把 Python 解释器交给陌生人。强烈建议走 SSH 隧道：

ssh -L 5678:localhost:5678 user@remote-server

然后 IDE 连 localhost:5678 即可。

多 worker / reload 场景​

  • uvicorn 多 worker（--workers N）会 fork 多个进程，每个都执行 debugpy.listen() → 端口冲突。生产慎用，调试时建议单 worker。
  • --reload 模式下进程会重启，listen() 重复调用要捕获 RuntimeError（上面引导模块已处理）。

五、几个易踩坑点速查​

现象原因
容器日志无 [debugpy] 监听...DEBUGPY_ENABLE=1 没传进容器，检查 compose env
TCP 连得上但断点不命中path mapping 没对齐，容器内真实路径和本地仓库路径前缀映射错
断点显示 unverified 且行号偏移本地代码版本和镜像 commit 不一致，先 git checkout 到镜像 tag
Celery worker 启动期断点不停没设 DEBUGPY_WAIT=1，进程已经跑过初始化代码了
address already in use上次 IDE 没断干净；重启容器即可
PyCharm 调试按钮灰色Registry debugpy.dap.is.enable 没开 / interpreter 未配置
PyCharm Initialization timeout大概率上次连接残留，重启容器
