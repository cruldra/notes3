> 开始使用 Agent Client Protocol (ACP)

Agent Client Protocol (ACP) 标准化了代码编辑器/IDE 与编码代理（Coding Agents）之间的通信，适用于本地和远程场景。

## 为什么选择 ACP？

AI 编码代理和编辑器紧密耦合，但互操作性并非默认具备。每个编辑器必须为其想要支持的每个代理构建自定义集成，而代理必须实现特定于编辑器的 API 才能触达用户。
这产生了一些问题：

* 集成开销：每个新的代理-编辑器组合都需要定制开发工作
* 兼容性有限：代理只能在部分可用编辑器中工作
* 开发者锁定：选择一个代理通常意味着接受其可用的界面

ACP 通过为代理-编辑器通信提供标准化协议来解决这个问题，类似于 [Language Server Protocol (LSP)](https://microsoft.github.io/language-server-protocol/) 标准化语言服务器集成的方式。

实现 ACP 的代理可以在任何兼容的编辑器中工作。支持 ACP 的编辑器可以访问整个 ACP 兼容代理生态系统。
这种解耦允许双方独立创新，同时让开发者可以自由选择最适合其工作流的工具。

## 概览

ACP 假设用户主要在编辑器中，并希望通过调用和使用代理来协助他们完成特定任务。

ACP 适用于本地和远程场景：

* **本地代理** 作为代码编辑器的子进程运行，通过 stdio 上的 JSON-RPC 进行通信。
* **远程代理** 可以托管在云端或独立的基础设施上，通过 HTTP 或 WebSocket 进行通信。

> **注意：** 对远程代理的全面支持正在进行中。我们正在与代理平台积极合作，以确保协议满足云托管和远程部署场景的特定需求。

该协议尽可能重用 MCP 中使用的 JSON 表示，但也包含用于有用的代理编码 UX 元素（如显示 diff）的自定义类型。

用户可读文本的默认格式是 Markdown，它具有足够的灵活性来表示丰富的格式，而无需代码编辑器具备渲染 HTML 的能力。
