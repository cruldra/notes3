数据感知工程 (Data-aware Engineering)

1. 一句话理解​

写代码处理数据之前，先看清楚数据长什么样。

数据感知工程是一种工程哲学：把"对数据特性的真实理解"作为代码设计的前置条件，而不是写完代码再让数据"有什么算什么"。

2. 通俗对比​

数据无感 (data-blind) 的写法​

# 写一个清洗器：去掉所有空白字符
def clean(text: str) -> str:
    return text.replace(" ", "").replace("\t", "").replace("\n", "")

for doc in corpus:
    cleaned = clean(doc["text"])
    save(cleaned)

潜在问题：

  • 你怎么知道语料里只有这三种空白？
  • 万一有全角空格 　？
  • 万一有零宽空格 ​？
  • 万一有 BOM ﻿？

代码看似正确，但因为没有"看见数据"，遗留了一堆静默 bug。

数据感知 (data-aware) 的写法​

# 第一步：先看数据
import unicodedata
from collections import Counter

char_classes = Counter()
for doc in corpus:
    for ch in doc["text"]:
        char_classes[unicodedata.category(ch)] += 1
print(char_classes.most_common(20))
# 输出揭示：除了常规空白还有 0x3000 全角空格、0x200b 零宽空格...

# 第二步：基于真实分布写清洗
WHITESPACE_CATEGORIES = {"Zs", "Zl", "Zp", "Cc"}
def clean(text: str) -> str:
    return "".join(
        ch for ch in text
        if unicodedata.category(ch) not in WHITESPACE_CATEGORIES
    )

核心动作：先观察分布、再写处理代码。

3. 核心原则​

原则含义
数据是一等公民数据像代码一样需要版本、审视、文档（数据卡片）、回归保护
观察先于动作任何对数据的处理动作都基于"已知分布"，不基于"假设分布"
异常显式记录发现的异常样本不静默过滤，而是单独保留 + 记录原因
分布即合约上下游组件之间的"契约"包括 schema 之外的统计性质（如长度分布、标签比例）
可重现观察探查脚本与代码同样入库，让"我看到的分布"可被复现

4. 与 Data-centric AI 的区别​

两个概念常被混淆。

维度Data-centric AIData-aware Engineering
提出者Andrew Ng (2021)业界共识，无单一来源
核心主张提升模型效果靠改数据，不靠改模型写处理数据的代码前要先理解数据
适用阶段训练 + 数据准备整个 MLOps 生命周期
目标模型指标工程系统的稳定性与正确性

可以这样理解：Data-centric AI 是 Data-aware Engineering 的一个特例，专门聚焦在"为提升模型效果而迭代数据"上。

5. 在 MLOps 生命周期中的位置​

完整 MLOps 生命周期 + 数据感知活动地图：

#阶段数据感知做什么
1问题定义评估数据可获得性 / 标注成本 / 隐私合规
2数据采集记录来源、许可、采集时间；为每个数据源写"数据卡片"
3数据探查 ⭐EDA：长度分布、字符类别、缺失率、标签分布、离群样本
4数据清洗基于探查结果写过滤规则（不是凭直觉）
5数据切分train/val/test 按文档级避免泄漏；切完再 EDA 一次确认分布一致
6特征工程每个特征的分布、与目标的相关性、缺失策略
7模型训练数据感知采样（balanced batch、reweight）、loss 按数据切片监控
8评估切片评估（slice metrics）：模型在不同数据子群上表现如何
9部署输入数据 schema 校验、distribution shift 检测
10监控 ⭐实时监控数据漂移、特征分布、标签飘移；触发再训练
11持续迭代每次模型更新前回到 #3 重新探查

最密集的两个阶段：阶段 3（探查）和阶段 10（监控） —— 这两个是"对数据持续打开雷达"的关键节点。

数据感知不是某一个阶段独占的活动，而是贯穿 11 个阶段的横向工程纪律。

6. 反例：data-blind 工程的代价​

经典案例（部分来自论文 Hidden Technical Debt in Machine Learning Systems, Sculley et al., NeurIPS 2015）：

现象真实原因
训练 acc 99%，线上一塌糊涂训练时数据被规范化成小写，部署时输入直接 utf8 → 大写词全错
模型悄悄变差但没人察觉用 mean=0/std=1 标准化训练，部署后输入分布漂移，没人监控
评估指标虚高训练数据按时间顺序，没洗牌就切 train/val → val 是未来，train 是过去
模型学到噪声标注质量参差，但训练时全数据等权
训练 loss 不降数据里 50% 是同一段文本（重复未去），模型只学这一段就够了

每一条都不是"模型问题"，而是"对数据缺乏感知"导致的工程问题。

7. 在 LLM 项目中的应用​

LLM 训练的全程都需要数据感知：

阶段数据感知关注点
预训练前语料长度分布 → 决定 max_seq_len；字符类别 → 决定 tokenizer 词表大小
预训练中loss 按数据来源分桶监控（哪类数据 loss 不降代表数据质量问题）
SFT指令数据的长度分布、回答风格分布、loss mask 是否对齐
DPO偏好数据的 chosen/rejected 长度比例、领域分布、有无字面级泄漏
推理用户输入分布与训练分布的差异（prompt drift）
评估分领域 / 分长度 / 分语言切片看模型表现

反例：直接 load_dataset(...) 就开始训，loss 不收敛了再回头看数据 —— 这是 data-blind 工程，浪费 GPU 也浪费时间。

8. 实操清单​

每次接触一份新数据时强制走的流程：

  1. 样本浏览：随机抽 10 条人眼看
  2. 规模统计：文档数、总字数、token 数估算
  3. 长度分布：均值、中位、分位数、最长、最短
  4. 字符类别：unicodedata.category 全表
  5. 重复检测：MinHash / 哈希全文
  6. 标签/元数据分布：类别比例、时间分布
  7. 抽样验证 schema：每个字段类型、缺失率
  8. 写 datacard：把以上发现固化为可入库的 markdown

只有走完这 8 步，才能开始写处理代码。
