AI 层需求提炼（6 个硬性维度）​

#需求维度代码证据权重
①确定性工作流编排ScoringAgent 有明确的 6 步流水线：collect_data → extract_features → [score_×4] → compute_composite → generate_summary🔴 极高
②FSM 状态机协同已实现 CustomerFSM，Agent 评分结果需要触发 FSM 事件（如 NO_INTERACTION_5D → 失去联系）🔴 极高
③多模型路由config.py 配置了 CLAUDE_API_KEY / GEMINI_API_KEY / OPENAI_API_KEY，README 明确要求多模型🟡 高
④RAG 管线Qdrant 已集成（AsyncQdrantClient），decision/rag/ 空目录待实现🟡 高
⑤异步优先全栈 async（FastAPI + asyncpg + motor），所有 Agent 方法都是 async def🟡 高
⑥Celery 调度集成ai_tasks.py 通过 Celery 定时/按需触发 Agent（run_scoring 每小时，update_profiles 每日）🟠 中

一、LangGraph — ⭐⭐⭐⭐⭐ 最佳匹配​

形象比喻：精密的齿轮与传动轴

适配度：95/100

需求匹配情况说明
① 确定性工作流✅ 完美StateGraph 的 node→edge 模型天然就是为 ScoringAgent 的 6 步流水线设计的
② FSM 协同✅ 完美LangGraph 本身就是状态图引擎，可以在 node 中直接调用已有的 CustomerFSM.trigger()
③ 多模型路由✅ 良好通过 langchain-core 的 BaseChatModel 接口统一对接，node 内可按维度选不同模型
④ RAG 管线✅ 良好作为 node 嵌入图中，Qdrant retriever → LLM → 结果，流程清晰
⑤ 异步✅ 原生ainvoke / astream 原生异步
⑥ Celery✅ 直接Celery task 内调用 await graph.ainvoke(state) 即可

决定性优势​

  1. 项目已经依赖它 — pyproject.toml 中已有 langgraph>=0.2，零迁移成本
  2. ScoringAgent 的流水线（collect→extract→score×4→composite→summary）1:1 映射为 StateGraph 的 nodes
  3. 已有的 ScoringState(TypedDict) 直接就是 LangGraph 的 State Schema
  4. RouterAgent 的 "根据事件类型路由" 就是 LangGraph 的 conditional_edge
  5. 与已实现的 CustomerFSM 互补而非竞争 — FSM 管客户状态，LangGraph 管 AI 工作流

代码映射示例​

# scoring_agent.py 用 LangGraph 实现
from langgraph.graph import StateGraph, END

graph = StateGraph(ScoringState)
graph.add_node("collect", collect_data)
graph.add_node("extract", extract_features)
graph.add_node("score_price", score_price)
graph.add_node("score_demand", score_demand)
graph.add_node("score_consensus", score_consensus)
graph.add_node("score_trust", score_trust)
graph.add_node("composite", compute_composite)
graph.add_node("summary", generate_summary)

# 并行评分 → 汇总
graph.add_edge("collect", "extract")
graph.add_edge("extract", "score_price")  # 可用 fan-out 并行
graph.add_edge("extract", "score_demand")
graph.add_edge("extract", "score_consensus")
graph.add_edge("extract", "score_trust")
# fan-in → composite → summary → END

适用场景​

  • 需要极高的定制化，标准的智能体循环无法满足需求
  • 你的工作流是确定性流程代码与智能体（Agentic）流程的复杂混合
  • 需要工业级的生产环境部署，对容错、状态管理和人为干预有强需求

二、LangChain (create_agent) — ⭐⭐⭐ 可用但不够​

形象比喻：万能转换插头与工具箱

适配度：55/100

需求匹配情况说明
① 确定性工作流⚠️ 不足create_agent 是 ReAct 循环（思考→工具→观察→思考...），无法精确控制6步流水线的执行顺序
② FSM 协同⚠️ 勉强需要把 FSM 包装成 Tool 让 Agent 调用，但调用时机不可控
③ 多模型路由✅ 最强langchain-core 的统一接口正是它的核心卖点
④ RAG 管线✅ 良好丰富的 Retriever/VectorStore 抽象
⑤ 异步✅ 良好支持但不如 LangGraph 原生
⑥ Celery✅ 直接同样可以在 task 中调用

核心问题​

Smart Sales 的 Agent 不是"给我一堆工具自由发挥"的模式，而是严格的、可预测的业务流水线。LangChain 的 create_agent 太"自由"了 — 你不希望评分 Agent 自己决定跳过某个维度的评分。

推荐用法​

作为组件库使用（model 接口、tool 抽象），不作为 Agent 编排层。事实上项目已经这样做了 — 依赖 langchain-core 而非完整的 langchain。

三、Deep Agents — ⭐⭐ 不适合​

形象比喻：自带武器库的单兵机甲

适配度：20/100

需求匹配情况说明
① 确定性工作流❌ 冲突Deep Agents 是"自治型"智能体，自己规划任务、自己拆解步骤，与确定性流水线理念相反
② FSM 协同❌ 不兼容它有自己的内部状态管理（虚拟文件系统），与外部 FSM 集成困难
③ 多模型路由⚠️ 有限主要为单模型深度任务设计
④ RAG 管线⚠️ 间接可以通过 Tool 接入，但不是它的核心强项
⑤ 异步✅ 支持—
⑥ Celery⚠️ 复杂重量级 Agent 与轻量级定时任务不匹配

核心特性​

  • 内置高级能力：自带任务规划与拆解工具、虚拟文件系统、长对话自动压缩等
  • 子智能体生成：可以通过内置工具动态生成专门的子智能体
  • 可插拔后端：支持将文件系统/状态存储在内存、本地磁盘、沙盒等

核心问题​

Deep Agents 为长时间、开放式、复杂推理任务设计（如编程、深度研究）。Smart Sales 的场景是短时间、封闭式、可预测的业务分析 — 每次评分都是同样的流程，只是数据不同。用"单兵机甲"去做"流水线工人"的活，杀鸡用牛刀且不可控。

适用场景​

  • 需要处理极其复杂的、多步骤的任务（如复杂的编码、深度研究、长周期项目）
  • 开发者希望最快速度拉起一个功能完备的强大智能体，不想自己从零去写规划逻辑

四、CrewAI — ⭐⭐⭐ 表面契合，实际过度​

形象比喻：管理良好的外包公司

适配度：45/100

需求匹配情况说明
① 确定性工作流⚠️ 部分支持顺序（Sequential）执行，但"角色扮演"的抽象对评分流水线来说是多余的
② FSM 协同⚠️ 需适配没有原生状态机概念，需要手动桥接
③ 多模型路由✅ 支持每个 Agent 可配不同 LLM
④ RAG 管线✅ 支持通过 Knowledge/Tool 集成
⑤ 异步⚠️ 弱原生异步支持不如 LangGraph 成熟
⑥ Celery⚠️ 复杂Crew 的生命周期管理与 Celery 有摩擦

核心理念​

多智能体团队协作（Multi-Agent Collaboration）。像组建公司一样去构建 AI 系统 — 你需要定义 Agent（员工）（比如：资深研究员、文案编辑），分配 Task（任务），并把他们组建成一个 Crew（团队），以顺序或层级的方式工作。

与 LangChain 生态的区别​

  • 高度主张（Opinionated）vs 通用底层：LangGraph 是一个通用的底层状态机（像一盒散装乐高），你可以用它拼出任何逻辑，但非常耗脑；而 CrewAI 直接提供了一个"团队协作"的成品模板
  • 生态兼容：CrewAI 早期在底层大量使用了 LangChain 的 Tools 和机制，是站在 LangChain 的肩膀上

核心问题​

CrewAI 的"角色"抽象（Agent = 员工，Task = 任务，Crew = 团队）在你的场景下是语义浪费 — ScoringAgent 不需要 "backstory"（角色背景），不需要 "delegation"（任务委派），它只需要按固定步骤执行。额外引入一个框架的依赖和概念成本不划算。

而且它会与已有的 langchain-core + langgraph 形成依赖冲突风险。

适用场景​

需要多个不同角色的 Agent 接力完成一个项目，用 CrewAI 远比用 LangGraph 从零手搓状态机要直观和迅速。

五、Agno (前 Phidata) — ⭐⭐⭐⭐ 轻量替代​

形象比喻：轻量锋利的瑞士军刀

适配度：65/100

需求匹配情况说明
① 确定性工作流⚠️ 需手动没有图编排，需要自己用 Python 代码串联步骤
② FSM 协同✅ 灵活轻量意味着与外部 FSM 集成无阻力
③ 多模型路由✅ 原生支持多 provider
④ RAG 管线✅ 内置原生 Knowledge/VectorDB 支持
⑤ 异步✅ 支持—
⑥ Celery✅ 轻松轻量级，嵌入 Celery task 无压力

核心理念​

轻量级、原生 Python、极速（Lightweight & Fast）。一个专注于用最简单直接的纯 Python 代码，为大模型注入记忆（Memory）、知识库（RAG）和工具（Tools）的框架。它原生支持多模态（文本、图像、音视频）。

与 LangChain 生态的区别​

  • 直接对抗 LangGraph 的"重"：LangChain/LangGraph 经常被开发者吐槽过度封装、抽象层级太深、概念复杂（Node、Edge、State、Channel 等等）。Agno 的卖点就是反臃肿，官方甚至宣称其智能体创建速度比 LangGraph 快万倍，内存占用仅为几十分之一
  • 低心智负担：你不需要学习复杂的"图（Graph）"概念，Agno 用极其直白的 Python 语法就能拉起一个带记忆和工具的智能体

优劣势对比​

优势：如果你觉得 LangGraph 太重、学习曲线太陡，Agno 是一个不错的降级选择。它让你用纯 Python 代码编排，没有 Node/Edge/State 的概念负担。

劣势：但你已经依赖了 LangGraph，切换到 Agno 意味着：

  1. 移除已有依赖
  2. 放弃 LangGraph 的持久化/人机协同/时间旅行等生产级能力
  3. 工作流编排回退到手写 Python 代码

适用场景​

追求极致性能、讨厌复杂框架抽象的极客开发者。

六、AutoAgent — ⭐ 完全不适合​

形象比喻：能听懂人话的智能工厂厂长

适配度：5/100

AutoAgent 是零代码/自然语言生成 Agent 的工具，面向非程序员。Smart Sales 是一个严肃的、需要精确控制的生产系统，需要代码级别的确定性保证。完全不在同一个赛道。

核心理念​

全自动化、零代码（Zero-Code LLM Agent Framework）。它不需要写一行 Python 代码，只需要用自然语言和系统对话，告诉它需求，它会自动为你生成、优化和编排一套多智能体工作流，甚至自己编写需要的工具代码。

适用场景​

毫无编程经验，或者想通过对话快速搭建整套系统的场景。

七、框架选型最终结论​

适配度排名​

│  1. LangGraph          ████████████ 95   │  ← 已在依赖中，1:1 映射
│  2. Agno               ███████      65   │  ← 轻量替代方案
│  3. LangChain Agent    ██████       55   │  ← 组件层用，不做编排
│  4. CrewAI             █████        45   │  ← 过度抽象

└─────────────────────────────────────────┘

🏆 推荐方案：LangGraph（继续深化使用）​

理由铁三角：

  1. 零迁移成本 — pyproject.toml 已有 langgraph>=0.2 + langchain-core>=0.3
  2. 架构 1:1 映射 — 已有的 ScoringState(TypedDict) 就是 LangGraph State；6 步流水线就是 6 个 Node；RouterAgent 的路由逻辑就是 conditional_edge
  3. 与 FSM 互补 — LangGraph 管"AI 怎么思考"（评分/画像工作流），CustomerFSM 管"客户在哪个阶段"（生命周期状态），各司其职

推荐技术栈组合​

层级技术职责
模型接口langchain-core (已有)统一 ChatModel 抽象，对接 Claude/Gemini/GPT-4o
工作流编排langgraph (已有)StateGraph 实现 Scoring/Profile/Router Agent
客户状态CustomerFSM (已实现)管理客户生命周期，被 Agent 工作流触发
向量检索qdrant-client (已有)RAG 管线的 retriever 后端
任务调度Celery (已有)定时/按需触发 Agent 工作流

不需要引入任何新框架，用好手头已有的就是最优解。

九、核心阵营形象化对比​

框架核心设计哲学最适合的场景形象比喻
LangGraph底层图结构、绝对的状态控制权混合了确定性代码与非确定性 AI 的复杂企业级工作流极其精密的齿轮与传动轴
LangChain组件化、大一统的 API 接口快速接入不同的模型和第三方工具万能的转换插头与工具箱
Deep Agents官方预设的最强单兵/子智能体需要处理长文本、多步骤、需文件系统的重型任务自带武器库的单兵机甲
CrewAI角色扮演、团队工序编排需要多角色（如：开发+测试+产品）流水线协作的项目管理良好的外包公司
Agno极致轻量、纯正 Pythonic追求极速启动、极低内存消耗、讨厌复杂框架抽象的场景轻量锋利的瑞士军刀
AutoAgent自然语言生成、零代码毫无编程经验，或者想通过对话快速搭建整套系统的场景能听懂人话的智能工厂厂长

十、下一步实施建议​

  1. 优先实现 ScoringAgent — 用 LangGraph StateGraph 实现 6 步评分流水线，作为第一个完整 Agent 示例
  2. 集成 LLM 客户端 — 基于 langchain-core 实现 Claude/Gemini/GPT-4o 的统一适配器
  3. RAG 管线落地 — 在 decision/rag/ 中实现 embedding → Qdrant → retrieval 的完整链路
  4. 状态持久化 — 利用 LangGraph 的 checkpointing 机制，实现长时任务的断点续行
  5. 人机协同 — 在关键节点（如评分结果生成后）添加人工审批断点
