## 视口与画布项目 (Viewport and Canvas items)

[CanvasItem](https://docs.godotengine.org/en/stable/classes/class_canvasitem.html) 是所有 2D 节点的基类，无论是普通的 2D 节点（如 [Node2D](https://docs.godotengine.org/en/stable/classes/class_node2d.html)）还是 [Control](https://docs.godotengine.org/en/stable/classes/class_control.html) 节点。两者都继承自 `CanvasItem`。你可以将画布项目组织成树状结构，每个项目都会继承其父级的变换（Transform）：当父级移动时，其子级也会随之移动。

`CanvasItem` 节点及其继承类都是 [Viewport (视口)](https://docs.godotengine.org/en/stable/classes/class_viewport.html) 的直接或间接子节点，视口负责显示它们。

视口的 `canvas_transform` 属性允许对其包含的 `CanvasItem` 层次结构应用自定义的 2D 变换。像 [Camera2D](https://docs.godotengine.org/en/stable/classes/class_camera2d.html) 这样的节点就是通过更改该变换来工作的。

为了实现滚动（Scrolling）等效果，操作视口的画布变换属性比移动根画布项目及整个场景要高效得多。

然而，通常我们不希望游戏或应用中的 **所有事物** 都受画布变换的影响。例如：

*   **视差背景 (Parallax Backgrounds)**：移动速度比关卡其余部分慢的背景。
*   **UI**：想象一个叠加在游戏世界视图上的用户界面（UI）或抬头显示（HUD）。我们希望生命计数器、分数显示和其他元素即使在游戏世界视图发生变化时，也能保持其屏幕位置。
*   **过渡效果 (Transitions)**：我们可能希望用于过渡的视觉效果（渐隐、混合）保持在固定的屏幕位置。

如何在一个单一场景树中解决这些问题？

## 画布图层 (CanvasLayers)

答案是 [CanvasLayer](https://docs.godotengine.org/en/stable/classes/class_canvaslayer.html)，这是一个为其所有子节点和孙节点添加独立 2D 渲染层的节点。

视口的子节点默认在图层 “0” 绘制，而 `CanvasLayer` 可以在任何数字图层上绘制。图层编号越大的层将绘制在图层编号较小的层之上。`CanvasLayer` 也有自己的变换，且不依赖于其他图层的变换。这使得 UI 可以在我们观察游戏世界的视角变化时，固定在屏幕空间（Screen-space）中。

一个典型的例子是创建视差背景。这可以通过一个处于图层 “-1” 的 `CanvasLayer` 来实现。而包含得分、生命计数器和暂停按钮的屏幕则可以创建在图层 “1” 上。

`CanvasLayer` 独立于树顺序，它们只取决于其图层编号，因此可以在需要时随时实例化。

> **注意**
>
> `CanvasLayer` 并不是控制节点绘制顺序（遮挡关系）的唯一方式。确保节点正确绘制在其他节点“前面”或“后面”的标准方法是操作场景面板中节点的顺序。虽然这可能有些反直觉，但在场景面板中 **排在上面的节点会绘制在下面节点的后面**。此外，2D 节点还有 `z_index` 属性用于精细控制绘制顺序。
