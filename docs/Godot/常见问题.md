## Godot 支持哪些编程语言？

Godot 官方支持的语言是 GDScript、C# 和 C++。请参阅 [脚本](https://docs.godotengine.org/en/stable/tutorials/scripting/index.html#toc-learn-scripting) 章节中每种语言的子类别。

## 什么是 GDScript，为什么要使用它？

GDScript 是 Godot 的集成脚本语言。它从零开始构建，旨在以最少的代码最大限度地发挥 Godot 的潜力，让新手和专家级开发者都能尽快利用 Godot 的优势。如果你以前用 Python 之类的语言写过任何东西，那么你会有宾至如归的感觉。有关 GDScript 提供的功能的示例和完整概述，请查看 [GDScript 脚本指南](https://docs.godotengine.org/en/stable/tutorials/scripting/gdscript/index.html#doc-gdscript)。

使用 GDScript 有几个原因，但最显著的原因是整体 **复杂性的降低**。

为 Godot 创建一个紧密集成、自定义脚本语言的初衷有两个：首先，它减少了启动和运行 Godot 所需的时间，为开发者提供了一种以生产力为重点的快速接触引擎的方式；其次，它减轻了整体维护负担，减少了问题的维度，并允许引擎的开发者专注于解决与引擎核心相关的错误和改进功能，而不是花费大量时间尝试让一小部分增量功能在大量语言中工作。

由于 Godot 是一个开源项目，从一开始就必须优先考虑更集成和无缝的体验，而不是通过支持更熟悉的编程语言来吸引更多用户，特别是当支持这些更熟悉的语言会导致更糟糕的体验时。我们理解如果你宁愿在 Godot 中使用另一种语言（请参阅上面的支持选项列表）。即便如此，如果你还没有尝试过 GDScript，请尝试 **三天**。就像 Godot 一样，一旦你看到它有多强大以及你的开发变得多么迅速，我们认为你会喜欢上 GDScript 的。

有关熟悉 GDScript 或动态类型语言的更多信息，请参阅 [GDScript：动态语言导论](https://docs.godotengine.org/en/stable/tutorials/scripting/gdscript/gdscript_advanced.html#doc-gdscript-more-efficiently) 教程。

## 创建 GDScript 的动力是什么？

在早期，引擎使用 [Lua](https://www.lua.org/) 脚本语言。得益于 LuaJIT，Lua 运行很快，但创建到面向对象系统（通过使用回退机制）的绑定既复杂又缓慢，并且需要大量的代码。在对 [Python](https://www.python.org/) 进行了一些实验后，也证明难以嵌入。

为 Godot 创建自定义脚本语言的主要原因是：

1.  大多数脚本虚拟机（Lua、Python、Squirrel、JavaScript、ActionScript 等）中的多线程支持较差，而 Godot 使用线程。
2.  大多数脚本虚拟机中的类扩展支持较差，适应 Godot 的工作方式效率非常低（Lua、Python、JavaScript）。
3.  许多现有语言在绑定到 C++ 时接口非常糟糕，导致大量代码、错误、瓶颈和整体低效（Lua、Python、Squirrel、JavaScript 等）。我们想专注于一个伟大的引擎，而不是大量的集成。
4.  没有原生向量类型（Vector3、Transform3D 等），在使用自定义类型时导致性能大幅降低（Lua、Python、Squirrel、JavaScript、ActionScript 等）。
5.  垃圾回收器会导致停顿或不必要的巨大内存使用（Lua、Python、JavaScript、ActionScript 等）。
6.  难以与代码编辑器集成以提供代码补全、实时编辑等（所有这些语言）。

GDScript 旨在解决上述及更多问题。

## 哪种编程语言最快？

在大多数游戏中，*脚本语言* 本身并不是性能问题的原因。相反，性能受到低效算法（在所有语言中都很慢）、GPU 性能或物理或导航等常见 C++ 引擎代码的影响。Godot 支持的所有语言对于通用脚本来说都足够快。你应该根据其他因素选择语言，例如易用性、熟悉度、平台支持或语言特性。

总的来说，C# 和 GDScript 的性能在同一个数量级内，而 C++ 比两者都快。

将 GDScript 性能与 C# 进行比较比较棘手，因为在某些特定情况下 C# 可能更快。C# *语言* 本身往往比 GDScript 快，这意味着在调用 Godot 引擎代码较少的情况下，C# 可能更快。然而，由于 *编组 (marshalling)* 的成本，在进行许多 Godot API 调用时，C# 可能比 GDScript 慢。C# 的性能也可能受到垃圾回收的影响，垃圾回收会在随机且不可预测的时刻发生。这可能导致复杂项目中的卡顿问题，而且这并不局限于 Godot。

使用 [GDExtension](https://docs.godotengine.org/en/stable/tutorials/scripting/gdextension/what_is_gdextension.html#doc-what-is-gdextension) 的 C++ 几乎总是比 C# 或 GDScript 快。然而，C++ 不如 C# 或 GDScript 易用，且开发速度较慢。

你还可以在单个项目中使用多种语言，通过 [跨语言脚本](https://docs.godotengine.org/en/stable/tutorials/scripting/cross_language_scripting.html#doc-cross-language-scripting)，或者通过同时使用 GDExtension 和脚本语言。请注意，这样做会有其自身的复杂性。

## Godot 会支持 [插入闭源 SDK，如 FMOD、GameWorks 等] 吗？

Godot 的目标是创建一个免费且开源、采用 MIT 许可的、模块化且可扩展的引擎。核心引擎开发社区没有计划支持任何第三方闭源/专有 SDK，因为集成这些 SDK 将违背 Godot 的理念。

也就是说，由于 Godot 是开源且模块化的，没有什么能阻止你或任何其他感兴趣的人将这些库添加为模块，并在你的游戏中随附它们（无论是以开源还是闭源形式）。

要了解如何仍然可以提供对你选择的 SDK 的支持，请查看下面的“插件”问题。

如果你知道一个 Godot 不支持但提供免费开源集成的第三方 SDK，请考虑自己开始集成工作。Godot 不归某一个人所有；它属于社区，并随着像你这样雄心勃勃的社区贡献者而成长。

## 如何在我的系统上安装 Godot 编辑器（用于桌面集成）？

由于你实际上不需要在系统上安装 Godot 即可运行它，这意味着桌面集成不会自动执行。有两种方法可以克服这个问题。你可以从 [Steam](https://store.steampowered.com/app/404790/Godot_Engine/) (所有平台)、[Scoop](https://scoop.sh/) (Windows)、[Homebrew](https://brew.sh/) (macOS) 或 [Flathub](https://flathub.org/apps/details/org.godotengine.Godot) (Linux) 安装 Godot。这将自动执行桌面集成所需的步骤。

或者，你可以手动执行安装程序为你做的步骤：

### Windows

*   将 Godot 可执行文件移动到一个稳定的位置（即你的“下载”文件夹之外），这样你以后就不会不小心移动它并破坏快捷方式。
*   右键单击 Godot 可执行文件并选择 **创建快捷方式**。
*   将创建的快捷方式移动到 `%APPDATA%\Microsoft\Windows\Start Menu\Programs`。这是将出现在“开始”菜单中的快捷方式的用户级位置。你还可以通过右键单击可执行文件并选择 **固定到任务栏** 来将 Godot 固定在任务栏中。

### macOS

将解压后的 Godot 应用程序拖到 `/Applications/Godot.app`，然后根据需要将其拖到 Dock。只要 Godot 在 `/Applications` 或 `~/Applications` 中，Spotlight 就能找到它。

### Linux

*   将 Godot 二进制文件移动到一个稳定的位置（即你的“下载”文件夹之外），这样你以后就不会不小心移动它并破坏快捷方式。
*   重命名 Godot 二进制文件并将其移动到 `PATH` 环境变量中存在的位置。这通常是 `/usr/local/bin/godot` 或 `/usr/bin/godot`。执行此操作需要管理员权限，但这还允许你通过输入 `godot` 从 [终端运行 Godot 编辑器](https://docs.godotengine.org/en/stable/tutorials/editor/command_line_tutorial.html#doc-command-line-tutorial)。
    *   如果你无法将 Godot 编辑器二进制文件移动到受保护的位置，可以将二进制文件保存在主目录的某个位置，并修改下面链接的 `.desktop` 文件中的 `Path=` 行，使其包含 Godot 二进制文件的完整 *绝对* 路径。
*   将 [此 .desktop 文件](https://raw.githubusercontent.com/godotengine/godot/master/misc/dist/linux/org.godotengine.Godot.desktop) 保存到 `$HOME/.local/share/applications/`。如果你拥有管理员权限，也可以将 `.desktop` 文件保存到 `/usr/local/share/applications` 以使快捷方式对所有用户可用。

## Godot 编辑器是便携式应用程序吗？

在默认配置下，Godot 是 *半便携式的*。它的可执行文件可以从任何位置运行（包括不可写的位置），并且不需要管理员权限。

然而，配置文件将被写入用户级的配置或数据目录。这通常是一个很好的方法，但这意味着如果你复制包含 Godot 可执行文件的文件夹，配置文件将不会跨机器携带。有关更多信息，请参阅 [Godot 项目中的文件路径](https://docs.godotengine.org/en/stable/tutorials/io/data_paths.html#doc-data-paths)。

如果需要 *真正的* 便携式操作（例如在 USB 盘上使用），请按照 [自包含模式](https://docs.godotengine.org/en/stable/tutorials/io/data_paths.html#doc-data-paths-self-contained-mode) 中的步骤操作。

## 为什么 Godot 的目标是保持其核心功能集小而精？

Godot 有意不包含可以由插件实现的功能，除非它们被非常频繁地使用。一个不常使用的例子是高级人工智能功能。

这有几个原因：

*   **代码维护和错误表面。** 每当我们接受 Godot 存储库中的新代码时，现有的贡献者通常会承担维护它的责任。一些贡献者在代码合并后并不总是留在身边，这可能使我们难以维护相关代码。这可能导致功能维护不善，错误永远无法修复。最重要的是，随着时间的推移，需要测试和检查回归的“API 表面”不断增加。
*   **易于贡献。** 通过保持代码库小巧整洁，它可以保持快速且易于从源码编译。这使得新贡献者更容易开始使用 Godot，而无需他们购买高端硬件。
*   **为编辑器保持较小的二进制文件大小。** 并非每个人都有快速的互联网连接。确保每个人都可以在不到 5 分钟的时间内下载 Godot 编辑器、解压并运行它，使 Godot 对所有国家的开发者都更具可访问性。
*   **为导出模板保持较小的二进制文件大小。** 这直接影响使用 Godot 导出的项目的大小。在移动和 Web 平台上，保持较小的文件大小对于确保在性能较低的设备上快速安装和加载非常重要。同样，许多国家的宽带互联网并不容易获得。此外，这些国家通常实行严格的数据使用上限。

出于上述所有原因，我们必须有选择地接受 Godot 的核心功能。这就是为什么我们的目标是在未来的 Godot 版本中将一些核心功能移动到官方支持的插件中。在二进制文件大小方面，这还有一个优点，即让你只为你实际在项目中使用的功能买单。（与此同时，你可以 [编译禁用未使用功能的自定义导出模板](https://docs.godotengine.org/en/stable/engine_details/development/compiling/optimizing_for_size.html#doc-optimizing-for-size) 以优化项目分发的大小。）

## 应该如何创建资产来处理多种分辨率和宽高比？

这个问题经常出现，这可能是由于 Apple 最初将设备分辨率翻倍时造成的误解。它让人们认为在不同分辨率下拥有相同的资产是一个好主意，因此许多人继续朝着这条路走下去。这最初在一定程度上有效，并且仅适用于 Apple 设备，但后来出现了几种具有不同分辨率和宽高比的 Android 和 Apple 设备，其尺寸和 DPI 范围非常广。

实现这一目标最常见且最合适的方法是，为游戏使用单一的基础分辨率，并且只处理不同的屏幕宽高比。这主要在 2D 中需要，因为在 3D 中，这只是相机垂直或水平视野 (FOV) 的问题。

1.  为你的游戏选择一个单一的基础分辨率。即使有的设备分辨率高达 1440p，有的设备低至 400p，你设备中的常规硬件缩放将以极低或无性能成本处理此问题。最常见的选择是接近 1080p (1920x1080) 或 720p (1280x720)。请记住，分辨率越高，资产越大，占用的内存越多，加载时间越长。
2.  使用 Godot 中的拉伸选项；画布项目 (canvas items) 拉伸同时保持宽高比效果最好。查看关于如何实现此目标的 [多分辨率](https://docs.godotengine.org/en/stable/tutorials/rendering/multiple_resolutions.html#doc-multiple-resolutions) 教程。
3.  确定最小分辨率，然后决定你是希望游戏针对不同的宽高比进行垂直或水平拉伸，还是使用固定的宽高比并出现黑边。这也在 [多分辨率](https://docs.godotengine.org/en/stable/tutorials/rendering/multiple_resolutions.html#doc-multiple-resolutions) 中有解释。
4.  对于用户界面，使用 [锚定 (anchoring)](https://docs.godotengine.org/en/stable/tutorials/ui/size_and_anchors.html#doc-size-and-anchors) 来确定控件应该留在何处以及如何移动。如果 UI 更复杂，请考虑学习容器 (Containers)。

就这样！你的游戏应该可以在多种分辨率下工作。

## 我想做贡献！我该如何开始？

太棒了！作为一个开源项目，Godot 的繁荣离不开像你这样的开发者的创新和抱负。

开始为 Godot 做出贡献的最佳方式是使用它并报告你可能遇到的任何 [问题](https://github.com/godotengine/godot/issues)。一份带有清晰复现步骤的高质量错误报告有助于你的贡献者同行快速有效地修复错误。你还可以报告在 [在线文档](https://github.com/godotengine/godot-docs/issues) 中发现的问题。

如果你准备好提交你的第一个 PR，请从上面的链接中选择任何引起你共鸣的问题，并尝试修复它。你需要学习如何从源码编译引擎，或者如何构建文档。你还需要熟悉 Git，这是 Godot 开发者使用的版本控制系统。

我们在 [针对贡献者的文档](https://contributing.godotengine.org/en/latest/organization/how_to_contribute.html) 中解释了如何处理引擎源码、如何编辑文档以及还有哪些其他的贡献方式。

## 我对 Godot 有个好主意。我该如何分享它？

我们一直在寻找有关如何改进引擎的建议。用户反馈是我们决策过程的主要驱动力，你在处理项目时可能面临的限制是我们考虑引擎增强功能的重要参考。

如果你遇到易用性问题或在当前版本的 Godot 中缺少某个功能，请先在我们的 [社区](https://godotengine.org/community/) 中讨论。社区成员可能会建议其他（或许更好的）方法来实现所需的结果。而且你可以了解其他用户是否也遇到了同样的问题，并一起找出好的解决方案。

如果你为引擎想出了一个明确的主意，欢迎开启一个 [提案问题 (proposal issue)](https://github.com/godotengine/godot-proposals/issues)。在描述你的问题和拟议的解决方案时，尽量做到具体且具体 —— 只有可操作的提案才能被考虑。虽然不是必需的，但如果你想自己实现它，那总是很受欢迎！

如果你只有一个笼统的想法而没有具体的细节，你可以开启一个 [提案讨论 (proposal discussion)](https://github.com/godotengine/godot-proposals/discussions)。这些讨论可以是任何你想要的内容，并允许进行自由形式的讨论以寻找解决方案。一旦找到解决方案，就可以开启一个提案问题。

请在创建提案之前阅读 [readme](https://github.com/godotengine/godot-proposals/blob/master/README.md) 文档，以了解有关该流程的更多信息。

## 是否可以使用 Godot 创建非游戏应用程序？

是的！Godot 具有广泛的内置 UI 系统，其较小的分发尺寸使其可以成为 Electron 或 Qt 等框架的合适替代品。

在创建非游戏应用程序时，请务必在“项目设置”中启用 [低处理器模式 (low-processor mode)](https://docs.godotengine.org/en/stable/classes/class_projectsettings.html#class-projectsettings-property-application-run-low-processor-mode) 以降低 CPU 和 GPU 使用率。

查看 [Material Maker](https://github.com/RodZill4/material-maker) 和 [Pixelorama](https://github.com/Orama-Interactive/Pixelorama) 以了解使用 Godot 制作的开源应用程序示例。

## 是否可以将 Godot 作为库使用？

如果你想用 Godot 制作游戏，请记住 Godot 设计为与其编辑器一起使用。我们建议你尝试一下，因为它很长一段时间内可能会为你节省时间。

对于更专业的应用，研究将 Godot 作为库使用可能是有意义的。自 Godot 4.6 起，LibGodot 形式提供了将 Godot 作为静态或共享库使用的 **实验性** 支持。目前在 Windows、macOS 和 Linux 上受支持。计划在未来版本中支持 Android 和 iOS。

你可以在 [migeran/libgodot\_project GitHub 存储库](https://github.com/migeran/libgodot_project) 中找到将 Godot 作为库使用的示例应用程序。

## Godot 使用什么用户界面工具包？

Godot 不使用 GTK、Qt 或 wxWidgets 等标准 GUI 工具包。相反，Godot 使用其自己的用户界面工具包，该工具包始终使用硬件加速进行渲染。虽然可以使用在 CPU 上模拟图形 API 的外部解决方案，但没有内置的软件回退机制。

该工具包以控件节点 (Control nodes) 的形式公开，用于渲染编辑器（编辑器是用 C++ 编写的）。这些控件节点也可以在 Godot 支持的任何脚本语言的项目中使用。

这种自定义工具包使得从硬件加速中受益并在所有平台上拥有统一的外观成为可能。最重要的是，它不必处理 GTK 或 Qt 带来的 LGPL 许可陷阱。最后，这意味着 Godot 在“吃自己的狗粮”，因为编辑器本身是 Godot UI 系统最复杂的用户之一。

这种自定义 UI 工具包可以 [嵌入到其他应用程序中](https://docs.godotengine.org/en/stable/about/faq.html#doc-faq-use-godot-as-library)（实验性）。然而，首选的使用方式是 [使用 Godot 通过编辑器创建非游戏应用程序](https://docs.godotengine.org/en/stable/about/faq.html#doc-faq-non-game-applications)。

## 为什么 Godot 使用 SCons 构建系统？

Godot 使用 [SCons](https://www.scons.org/) 构建系统。在不久的将来没有计划切换到不同的构建系统。我们选择 SCons 而非其他替代方案有很多原因。例如：

*   Godot 可以为十几种不同的平台编译：所有 PC 平台、所有移动平台、许多游戏主机和 WebAssembly。
*   开发者经常需要 **同时** 为多个平台编译，甚至是同一平台的多个不同目标。他们无法承担每次都重新配置和重新构建项目的成本。SCons 可以毫无困难地做到这一点，而不会破坏构建。
*   无论进行多少更改、配置、添加、删除等，SCons 都 *永远不会* 破坏构建。
*   Godot 的构建过程并不简单。有几个文件是由代码生成的（绑定器），其他的被解析（着色器），还有一些需要提供定制化（[模块](https://docs.godotengine.org/en/stable/engine_details/engine_api/custom_modules_in_cpp.html#doc-custom-modules-in-cpp)）。这需要复杂的逻辑，在实际的编程语言（如 Python）中编写比使用主要用于构建的宏语言更容易。
*   Godot 的构建过程大量使用交叉编译工具。每个平台都有特定的检测过程，所有这些都必须作为特定情况处理，并为每个情况编写专门的代码。

如果你打算自己构建 Godot，请尝试保持开放的心态并至少熟悉一下 SCons。

## 为什么 Godot 不使用 STL (标准模板库)？

像许多其他库一样（例如 Qt），Godot 不使用 STL（除了极少数例外，如线程原语）。我们认为 STL 是一个伟大的通用库，但我们对 Godot 有特殊要求。

*   STL 模板创建非常大的符号，这导致巨大的调试二进制文件。相反，我们使用很少的模板和非常短的名称。
*   我们的大多数容器满足特殊需求，例如 Vector，它使用写时复制，我们用它来传递数据，或者 RID 系统，它需要 O(1) 访问时间以保证性能。同样，我们的哈希映射实现旨在与引擎内部类型无缝集成。
*   我们的容器内置了内存跟踪，这有助于更好地跟踪内存使用情况。
*   对于大型数组，我们使用池化内存，它可以映射到预分配的缓冲区或虚拟内存。
*   我们使用自定义的 String 类型，因为 STL 提供的类型过于基础且缺乏适当的国际化支持。

查看 [Godot 的容器类型](https://docs.godotengine.org/en/stable/engine_details/architecture/core_types.html#doc-core-types) 以获取替代方案。

## 为什么 Godot 不使用异常？

我们认为游戏无论如何都不应该崩溃。如果发生意外情况，Godot 会打印一个错误（甚至可以追踪到脚本），但随后它会尝试尽可能优雅地恢复并继续运行。

此外，异常显著增加了可执行文件的二进制大小，并导致编译时间增加。

## Godot 使用 ECS (实体组件系统) 吗？

Godot **不** 使用 ECS，而是依赖于继承。虽然没有普遍更好的方法，但我们发现使用基于继承的方法具有更好的易用性，同时对于大多数用例来说仍然足够快。

即便如此，没有什么能阻止你在项目中使用组合，方法是使用单独的脚本创建子节点。然后可以在运行时添加和移除这些节点，以动态地添加和移除行为。

有关 Godot 设计选择的更多信息，可以在 [这篇文章](https://godotengine.org/article/why-isnt-godot-ecs-based-game-engine) 中找到。

## 为什么 Godot 不强制用户实现 DOD (面向数据设计)？

虽然 Godot 内部尝试尽可能地使用缓存一致性，但我们认为用户不需要被强制使用 DOD 实践。

DOD 主要是一种缓存一致性优化，只有在处理每帧处理且几乎不修改的成千上万个对象时才能提供显著的性能提升。也就是说，如果你每帧移动几百个精灵或敌人，DOD 不会导致性能的实质性提升。在这种情况下，你应该考虑一种不同的优化方法。

绝大多数游戏不需要这个，而且当你在大多数情况下需要时，Godot 提供了方便的助手来完成这项工作。

如果一个游戏需要处理如此大量对象，我们的建议是对性能繁重的任务使用 C++ 和 GDExtensions，对游戏的其余部分使用 GDScript（或 C#）。
