核心类型

Godot 拥有丰富的一系列类和模板，它们构成了引擎的核心，一切都建立在它们之上。

本参考文档将尝试按顺序列出它们，以便更好地理解。

分配内存 (Allocating memory)​

Godot 有许多技巧来确保内存安全并跟踪内存使用情况。因此，不应使用常规的 C 和 C++ 库调用。相反，Godot 提供了一些替代方案。

对于 C 风格的分配，Godot 提供了几个宏：

memalloc(size)
memrealloc(pointer)
memfree(pointer)

这些宏分别等同于 C 标准库中常用的 malloc()、realloc() 和 free()。

对于 C++ 风格的分配，提供了特殊的宏：

memnew(Class)
memnew(Class(args))
memdelete(instance)

memnew_arr(Class, amount)
memdelete_arr(pointer_to_array)

这些分别等同于 new、delete、new[] 和 delete[]。

memnew/memdelete 还使用了一些 C++ 魔法，自动调用初始化后 (post-init) 和释放前 (pre-release) 的函数。例如，这用于在 Object 创建后立即通知它们，并在它们被删除之前通知它们。

  • core/os/memory.h

容器 (Containers)​

Godot 提供了一套自己的容器，这意味着代码库中通常不使用 std::string 和 std::vector 等 STL 容器。有关更多信息，请参阅 为什么 Godot 不使用 STL（标准模板库）？。

📜 图标表示该类型是 Variant 的一部分。这意味着它可以作为暴露给脚本 API 的方法的参数或返回值。

Godot 数据类型最接近的 C++ STL 数据类型备注
String 📜std::string将其作为“默认”字符串类型使用。 String 使用 UTF-32 编码，由于其固定的字符大小，简化了处理过程。
Vectorstd::vector将其作为“默认”向量/数组类型使用。 使用写时拷贝 (COW) 语义。这意味着它通常较慢，但几乎可以免费进行拷贝。在不需要 COW 且性能至关重要的地方，请改用 LocalVector。
HashSetstd::unordered_set将其作为“默认”集合类型使用。
AHashMapstd::unordered_map将其作为“默认”哈希映射类型使用。 不保留插入顺序。注意，指向映射中的指针以及迭代器在发生变动时是不稳定的。如果需要这些特性，请改用 HashMap。
StringName 📜std::string使用字符串驻留 (string interning) 进行快速比较。对于频繁引用且在引擎中多处使用的静态字符串，请使用此类型。
LocalVectorstd::vector语义上更接近 std::vector，不使用写时拷贝 (COW)，因此比 Vector 快。当不需要廉价拷贝时，优先选择它而不是 Vector。
Array 📜std::vector值可以是任何 Variant 类型。不强加静态类型。使用共享引用计数，类似于 std::shared_ptr。内部使用 Vector<Variant>。
TypedArray 📜std::vectorArray 的子类，但对其元素进行了静态类型约束。不要与 Packed*Array 混淆，后者内部是一个 Vector。
Packed*Array 📜std::vectorVector 的别名，例如 PackedColorArray = Vector<Color>。仅提供有限的紧凑数组类型列表（否则请使用 TypedArray）。
Liststd::list链表类型。通常比其他数组/向量类型慢。在编写新代码时，除非使用 List 可以避免类型转换，否则优先选择其他类型。
FixedVectorstd::array具有固定容量的向量（更类似于 boost::container::static_vector）。这种容器类型比其他类似向量的类型更高效，因为它不进行堆分配。
Spanstd::span表示对连续数组的只读访问，无需拷贝任何数据。注意 Span 被设计为高性能 API：它不执行像你可能习惯的其他 Godot 容器那样的参数正确性检查。请谨慎使用。Span 可以从大多数类似数组的容器构造（例如 vector.span()）。
RBSetstd::set使用 红黑树 以实现更快的访问。
VSetstd::flat_set使用写时拷贝 (COW) 语义。这意味着它通常较慢，但几乎可以免费进行拷贝。VSet 的性能优势尚未确定，因此请优先使用其他类型。
HashMapstd::unordered_map防御性（稳健但慢）的映射类型。保留插入顺序。指向键和值的指针以及迭代器在变动时是稳定的。当需要这些特性时，请使用此映射类型。否则请使用 AHashMap。
RBMapstd::map使用 红黑树 查找键的映射类型。RBMap 的性能优势尚未确定，因此请优先使用其他类型。
Dictionary 📜std::unordered_map键和值可以是任何 Variant 类型。不强加静态类型。使用共享引用计数，类似于 std::shared_ptr。保留插入顺序。内部使用 HashMap<Variant>。
TypedDictionary 📜std::unordered_mapDictionary 的子类，但对其键和值进行了静态类型约束。
Pairstd::pair存储单个键值对。另请参阅同文件中的 KeyValue，它使用只读键。

重新定位安全性 (Relocation safety)​

Godot 的容器假设其元素是 可平凡重新定位的 (trivially relocatable)。

这意味着，如果你在其中存储了具有指向自身的指针的数据类型，或者其他 非平凡重新定位 的数据类型，Godot 可能会崩溃。注意，存储指向非平凡重新定位对象的 指针（例如某些 Object 子类）是没有问题且受支持的。

假设平凡重新定位的原因是它允许我们使用重要的优化技术，例如通过 memcpy 或 realloc 进行重新定位。

GH-100509 跟踪了这一决策。

多线程 / 并发 (Multithreading / Concurrency)​

另请参阅​

你可以在 使用多线程 中找到关于多线程策略的更多信息。

Godot 的容器都不是线程安全的。当你预期有多个线程访问它们时，必须使用多线程保护。

注意，这里列出的一些类型也可以通过绑定 (bindings) 获得，但绑定类型包装在 RefCounted 中（位于 CoreBind:: 命名空间）。出于效率原因，请尽可能优先选择这里列出的原始类型。

Godot 数据类型最接近的 C++ STL 数据类型备注
Mutexstd::recursive_mutex递归互斥锁类型。使用 MutexLock lock(mutex) 进行加锁。
BinaryMutexstd::mutex非递归互斥锁类型。使用 MutexLock lock(mutex) 进行加锁。
RWLockstd::shared_timed_mutex读写感知互斥锁类型。使用 RWLockRead lock(mutex) 或 RWLockWrite lock(mutex) 进行加锁。
SafeBinaryMutexstd::mutex可与 ConditionVariable 配合使用的递归互斥锁类型。使用 MutexLock lock(mutex) 进行加锁。
ConditionVariablestd::condition_variable条件变量类型，与 SafeBinaryMutex 配合使用。
Semaphorestd::counting_semaphore计数信号量类型。
SafeNumericstd::atomic模板化的原子类型，专为数字设计。
SafeFlagstd::atomic_bool布尔原子类型。
SafeRefCountstd::atomic专为引用计数设计的原子类型。如果引用计数为 0，将拒绝增加计数。

数学类型 (Math types)​

在 core/math 目录中有几种线性数学类型可用：

  • core/math

NodePath​

这是一种特殊的数据类型，用于存储场景树中的路径并以优化后的方式引用它们：

  • core/string/node_path.h

RID​

RID 是 资源 ID (Resource IDs)。服务器使用这些 ID 来引用存储在其中的数据。RID 是不透明的 (opaque)，意味着它们引用的数据不能直接访问。RID 是唯一的，即使是对于不同类型的引用数据也是如此：

  • core/templates/rid.h
