避免对一切都使用节点

节点的生成成本很低，但即使是节点也有其极限。一个项目可能会有成千上万个节点在同时运作。然而，它们的行为越复杂，每一个节点给项目性能带来的负担就越大。

Godot 提供了更轻量级的对象用于创建供节点使用的 API。在设计如何构建项目功能时，请务必记住这些选项。

  1. Object：终极轻量级对象，原始的 Object 必须使用手动内存管理。话虽如此，创建自己的自定义数据结构，甚至是节点结构，且比 Node 类更轻量，并不是太难。

    • 示例：参考 Tree 节点。它支持高度定制的目录表，可以拥有任意数量的行和列。然而，它用来生成可视化效果的数据实际上是一个由 TreeItem Object 组成的树。
    • 优点：将 API 简化为作用域更小的对象有助于提高其可访问性并缩短迭代时间。与其处理整个 Node 库，不如创建一组精简的 Object，节点可以从中生成并管理相应的子节点。

    注意

    处理它们时应当小心。你可以将一个 Object 存储到变量中，但这些引用可能会在没有任何警告的情况下变得无效。例如，如果对象的创建者突然决定删除它，当你下次访问它时，这会触发错误状态。

  2. RefCounted：仅比 Object 稍微复杂一点。它们会跟踪对自身的引用，仅在不再有对自身的引用时才释放加载的内存。在大多数需要在自定义类中存储数据的情况下，这些都非常有用。

    • 示例：参考 FileAccess 对象。它的功能就像一个常规 Object，只是你不需要自己手动删除它。
    • 优点：与 Object 相同。

  3. Resource：仅比 RefCounted 稍微复杂一点。它们天生具有将对象属性序列化/反序列化（即保存和加载）到 Godot 资源文件或从资源文件加载的能力。

    • 示例：脚本（Scripts）、PackedScene（用于场景文件），以及其他类型如各个 AudioEffect 类。它们中的每一个都可以被保存和加载，因此它们继承自 Resource。
    • 优点：关于 Resource 相比传统数据存储方法的优势，已经有很多论述了。然而，在“使用资源而非节点”的背景下，它们的主要优势在于 检查器（Inspector）兼容性。虽然它们几乎和 Object/RefCounted 一样轻量，但它们仍然可以在检查器中显示和导出属性。这使得它们在易用性方面能够履行类似于“子节点”的职责，但如果你计划在场景中拥有大量此类资源/节点，它们还能提高性能。
