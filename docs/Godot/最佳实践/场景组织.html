场景组织

本文涵盖了与有效组织场景内容相关的主题。你应该使用哪些节点？应该把它们放在哪里？它们应该如何交互？

如何有效地建立关系 (How to build relationships effectively)​

当 Godot 用户开始制作自己的场景时，他们经常会遇到以下问题：

他们创建了第一个场景并填充了内容，随着“应该把东西拆分出来”的念头不断积累，最终将场景的分支保存为独立的场景。然而，他们随后发现，以前可以依赖的硬引用（hard references）不再可行。在多个地方重用该场景会产生问题，因为节点路径找不到目标，且在编辑器中建立的信号连接也会断开。

要解决这些问题，你必须在实例化子场景时，使其不依赖于其环境的细节。你需要能够相信子场景会自我创建，而不必挑剔它是如何被使用的。

面向对象编程（OOP）中要考虑的最大事项之一是保持功能集中、单一职责的类，并与代码库的其他部分保持 松耦合 (loose coupling)。这可以保持对象体量微小（易于维护）并提高其可重用性。

这些 OOP 最佳实践对场景结构和脚本使用的最佳实践有 多重 启示。

如果可能的话，你应该将场景设计为无依赖。 也就是说，你应该创建将所需的一切都保留在自身内部的场景。

如果一个场景必须与外部上下文交互，经验丰富的开发人员建议使用 依赖注入 (Dependency Injection)。这种技术涉及让高级 API 提供低级 API 的依赖项。为什么要这样做？因为依赖于外部环境的类可能会无意中触发错误和意外行为。

为此，你必须公开数据，然后依靠父级上下文来对其进行初始化：

  1. 连接到信号。极其安全，但应仅用于“响应”行为，而非启动行为。按照惯例，信号名称通常是过去时动词，如 “entered”、“skill_activated” 或 “item_collected”。

    # 父级
    $Child.signal_name.connect(method_on_the_object)

    # 子级
    signal_name.emit() # 触发由父级指定的行为。

  2. 调用方法。用于启动行为。

    # 父级
    $Child.method_name = "do"

    # 子级，假设其具有 String 属性 'method_name' 和方法 'do'。
    call(method_name) # 调用父级指定的方法（该方法必须由子级拥有）。

  3. 初始化 Callable 属性。比方法更安全，因为不需要方法的所有权。用于启动行为。

    # 父级
    $Child.func_property = object_with_method.method_on_the_object

    # 子级
    func_property.call() # 调用父级指定的方法（可以来自任何地方）。

  4. 初始化 Node 或其他 Object 引用。

    # 父级
    $Child.target = self

    # 子级
    print(target) # 使用父级指定的节点。

  5. 初始化 NodePath。

    # 父级
    $Child.target_path = ".."

    # 子级
    get_node(target_path) # 使用父级指定的 NodePath。

这些选项隐藏了子节点的访问点。这反过来使子节点与其环境保持 松耦合。你可以在另一个上下文中重用它，而无需对其 API 进行任何额外更改。

注意

尽管上面的示例说明了父子关系，但同样的原则适用于所有的对象关系。同级节点（Siblings）应该只意识到它们自己的层次结构，而由祖先节点来调解它们的通信和引用。

# 父级
$Left.target = $Right.get_node("Receiver")

# Left
var target: Node
func execute():
	# 使用 'target' 执行某些操作。

# Right
func _init():
	var receiver = Receiver.new()
	add_child(receiver)

同样的原则也适用于维持对其他对象依赖的非节点（non-Node）对象。哪个对象拥有其他对象，就应该由它来管理它们之间的关系。

警告

尽管如此，你应该倾向于将数据保留在“内部”（即场景内部），因为将依赖项放在外部上下文中（即使是松耦合的），仍意味着该节点会预期其环境中的某些内容为真。项目的设计哲学应该防止这种情况发生。否则，代码固有的责任将迫使开发人员使用文档来在微观范围内跟踪对象关系；这也就是所谓的“开发地狱”。编写依赖外部文档才能安全使用的代码，默认情况下是容易出错的。

为了避免创建和维护此类文档，你可以将被依赖的节点（上面的“子级”）转换为实现 _get_configuration_warnings() 的工具脚本（tool script）。从中返回一个非空的 PackedStringArray 会使“场景”面板在节点旁生成一个带有字符串作为工具提示的警告图标。这与 Area2D 节点在其未定义子 CollisionShape2D 节点时显示的图标相同。然后，编辑器通过脚本代码对场景进行自我记录。不需要通过文档进行内容重复。

这样的 GUI 可以更好地告知项目用户关于 Node 的关键信息。它有外部依赖吗？这些依赖是否已满足？其他程序员，尤其是设计师和文案，将需要消息中清晰的指令来告诉他们如何配置它。

那么，为什么所有这些复杂的“乾坤大挪移”能起作用呢？因为场景在独立运作时表现最佳。如果无法独立运作，那么匿名地与他人合作（具有最小的硬依赖，即松耦合）是次优选择。不可避免地，可能需要对类进行更改，如果这些更改导致它以预见之外的方式与其他场景交互，那么事情就会开始崩溃。所有这些间接操作（indirection）的全部目的就是为了避免陷入这样一种境地：更改一个类会导致对依赖于它的其他类产生不利影响。

脚本和场景作为引擎类的扩展，应遵守 所有 OOP 原则。示例包括...

  • SOLID
  • DRY (Don't Repeat Yourself)
  • KISS (Keep It Simple, Stupid)
  • YAGNI (You Ain't Gonna Need It)

选择节点树结构 (Choosing a node tree structure)​

你可能开始开发一款游戏，却被面前巨大的可能性搞得不知所措。你可能知道自己想做什么，想要哪些系统，但是你把它们 放 在哪里呢？如何制作你的游戏始终取决于你自己。你可以以无数种方式构建节点树。如果你不确定，本指南可以为你提供一个不错的起始结构示例。

一个游戏应该总有一个“入口点”；某个你可以明确追踪事物开始的地方，以便你可以跟随逻辑继续延伸到别处。它也作为程序中所有其他数据和逻辑的鸟瞰图。对于传统应用程序，这通常是一个 “main” 函数。在 Godot 中，它是一个 Main 节点。

  • Node "Main" (main.gd)

main.gd 脚本将作为你游戏的主要控制器。

然后你有一个游戏内的“世界”（2D 或 3D 的）。这可以是 Main 的子节点。此外，你将需要一个用于游戏的主要 GUI，用于管理项目所需的各种菜单和小部件。

  • Node "Main" (main.gd)
    • Node2D/Node3D "World" (game_world.gd)
    • Control "GUI" (gui.gd)

在切换关卡时，你可以更换 “World” 节点的子节点。手动更改场景 让你能够完全控制游戏世界的转换方式。

下一步是考虑你的项目需要哪些游戏系统。如果你有一个系统...

  1. 在内部跟踪其所有数据
  2. 应该是全局可访问的
  3. 应该独立存在

...那么你应该创建一个 自动加载 (autoload) '单例' 节点。

注意

对于小型游戏，一个控制力较弱的简单替代方案是拥有一个“Game”单例，它只需调用 SceneTree.change_scene_to_file() 方法来更换主场景的内容。这种结构或多或少将 “World” 保留为主要游戏节点。

任何 GUI 也需要是单例、 “World” 的过渡部分，或者作为根节点的直接子节点手动添加。否则，GUI 节点也会在场景转换期间删除自己。

如果你有修改其他系统数据的系统，你应该将这些定义为它们自己的脚本或场景，而不是自动加载（autoloads）。有关更多信息，请参阅 自动加载与常规节点。

游戏中的每个子系统都应在 SceneTree 中拥有自己的部分。只有在节点实际上是其父节点的元素的情况下，才应使用父子关系。移除父节点是否合乎逻辑地意味着子节点也应被移除？如果不是，那么它应该在层次结构中作为兄弟节点或其他关系拥有自己的位置。

注意

在某些情况下，你需要这些分离的节点 同时也 相对于彼此进行定位。你可以为此目的使用 RemoteTransform / RemoteTransform2D 节点。它们将允许目标节点有条件地从 Remote* 节点继承选定的变换元素。要分配 target NodePath，请使用以下方法之一：

  1. 一个可靠的第三方（很可能是父节点）来调解分配。
  2. 一个分组（group），用于获取所需节点的引用（假设目标永远只有一个）。

何时应该这样做是主观的。当由于必须微调节点何时在 SceneTree 中移动以保存自身而产生困境时。例如...

  • 将 “player” 节点添加到 “room” 中。

  • 需要更换房间，因此你必须删除当前房间。

  • 在删除房间之前，你必须保存和/或移动玩家。

    如果内存不是问题，你可以...

    • 创建新房间。
    • 将玩家移动到新房间。
    • 删除旧房间。

    如果内存是问题，那么你将需要...

    • 将玩家移动到树中的其他位置。
    • 删除房间。
    • 实例化并添加新房间。
    • 将玩家重新添加到新房间。

问题在于，这里的玩家是一个“特殊情况”，开发人员必须 知道 他们需要为项目以这种方式处理玩家。作为一个团队，可靠地分享这些信息的唯一方法是 记录文档。在文档中保留实现细节是危险的。它是维护负担，会使代码可读性变差，并不必要地膨胀了项目的知识内容。

在具有较大型资产的更复杂游戏中，将玩家完全保留在 SceneTree 的其他地方可能是一个更好的主意。这会导致：

  1. 更强的一致性。
  2. 没有必须在某处记录和维护的“特殊情况”。
  3. 不会因为没考虑到这些细节而出现错误的机会。

相反，如果你确实需要一个 不 继承其父级变换的子节点，你有以下选择：

  1. 声明式 解决方案：在它们之间放置一个 Node。因为它没有变换，所以它们不会将此信息传递给它的子节点。
  2. 命令式 解决方案：为 CanvasItem 或 Node3D 节点使用 top_level 属性。这将使节点忽略其继承的变换。

注意

如果构建网络游戏，请记住哪些节点和游戏系统与所有玩家相关，而哪些仅与权威服务器相关。例如，用户并不都需要每个玩家的 “PlayerController” 逻辑副本——他们只需要自己的。将它们保留在与 “world” 分开的分支中可以帮助简化游戏连接等的管理。

场景组织的关键是根据 关系 而非 空间 来考虑 SceneTree。节点是否依赖于其父节点的存在？如果不是，那么它们可以完全在别处蓬勃发展。如果它们是依赖的，那么理所当然它们应该是该父节点的子节点（如果还没整合的话，很可能是该父节点场景的一部分）。

这是否意味着节点本身就是组件？完全不是。Godot 的节点树形成的是一种聚合关系（aggregation），而不是组合关系（composition）。但是，虽然你仍然可以灵活地移动节点，但最好的情况是默认情况下不需要此类移动。
