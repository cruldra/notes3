## 简介 (Introduction)

Godot 的目标是做到对版本控制系统（VCS）友好，并生成大部分可读且可合并的文件。

## 版本控制插件 (Version control plugins)

Godot 还支持在编辑器内部直接使用版本控制系统。但是，编辑器中的版本控制需要针对你正在使用的特定 VCS 的插件。

截至 2025 年 10 月，目前仅有一个官方的 Git 插件可用，但社区可能会创建额外的 VCS 插件。

### 官方 Git 插件 (Official Git plugin)

官方插件支持在编辑器内部使用 Git。你可以在 [GitHub](https://github.com/godotengine/godot-git-plugin/releases) 上找到最新版本。

关于如何使用 Git 插件的文档可以在其 [维基页面 (wiki)](https://github.com/godotengine/godot-git-plugin/wiki) 中找到。

## 应从 VCS 中排除的文件 (Files to exclude from VCS)

> **注意**
>
> 此列表列出了在 Godot 4.1 及更高版本中应从版本控制中忽略的文件和文件夹。
>
> 在 Godot 3.x 和 Godot 4.0 中，应从版本控制中忽略的文件列表是 **完全** 不同的。这一点很重要，因为 Godot 3.x 和 4.0 可能会在 `export_presets.cfg` 中存储敏感凭据（而 Godot 4.1 及更高版本则不会）。
>
> 如果你使用的是 Godot 3，请查看该文档页面的 `3.6` 版本。

当第一次在编辑器中打开项目时，Godot 会自动创建一些文件和文件夹。为了避免生成的临时数据膨胀你的版本控制仓库，你应该将它们添加到 VCS 的忽略列表中：

*   `.godot/`：此文件夹存储各种项目缓存数据。
*   `*.translation`：这些是从 CSV 文件生成的二进制导入 [翻译文件](https://docs.godotengine.org/en/stable/tutorials/i18n/internationalizing_games.html)。

你可以在创建项目时让 Godot 项目管理器自动为你生成版本控制元数据。选择 **Git** 选项时，这会在项目根目录创建 `.gitignore` 和 `.gitattributes` 文件：

![在项目管理器的“新建项目”对话框中创建版本控制元数据](https://docs.godotengine.org/en/stable/_images/version_control_systems_generate_metadata.webp)

在现有项目中，选择编辑器顶部的 **项目 (Project)** 菜单，然后选择 **版本控制 (Version Control) > 生成版本控制元数据 (Generate Version Control Metadata)**。这将创建与在项目管理器中执行操作相同的文件。

## 在 Windows 上使用 Git (Working with Git on Windows)

大多数 Windows 版 Git 客户端配置为将 `core.autocrlf` 设置为 `true`。由于换行符会自动从 LF 转换为 CRLF，这可能导致文件不必要地被 Git 标记为已修改。

最好将此选项设置为：

```bash
git config --global core.autocrlf input
```

使用项目管理器或编辑器创建版本控制元数据将自动通过 `.gitattributes` 文件强制使用 LF 换行符。在这种情况下，你不需要更改 Git 配置。

## Git LFS (Large File Storage)

Git LFS 是一项 Git 扩展，允许你管理仓库中的大型文件。它在 Git 内部使用文本指针替换大型文件，同时将文件内容存储在远程服务器上。这对于管理大型资产（如纹理、音频文件和 3D 模型）非常有用，而不会使你的 Git 仓库变得臃肿。

> **注意**
>
> 使用 Git LFS 时，你需要确保在向仓库提交任何文件 **之前** 完成设置。如果你已经将文件提交到仓库，则需要将它们从仓库中移除，并在设置 Git LFS 后重新添加。
>
> 虽然可以使用 `git lfs migrate` 来转换仓库中的现有文件，但这操作较为复杂，需要对 Git 有深入的了解。
>
> 一种常见的方法是使用 Git LFS（以及正确的 `.gitattributes`）建立一个新仓库，然后将文件从旧仓库拷贝到新仓库。这样，你可以确保所有文件从一开始就由 LFS 跟踪。

要在 Godot 中使用 Git LFS，你需要安装 Git LFS 扩展并配置它以跟踪你想管理的文件类型。你可以通过在终端运行以下命令来实现：

```bash
git lfs install
```

这将在你的仓库中创建一个 `.gitattributes` 文件，告诉 Git 对指定的文件类型使用 LFS。你可以通过修改 `.gitattributes` 文件来添加更多文件类型。例如，要跟踪所有 GLB 文件，可以在终端运行以下命令：

```bash
git lfs track "*.glb"
```

当你添加或修改由 LFS 跟踪的文件时，Git 会自动将它们存储在 LFS 中，而不是常规的 Git 历史记录中。你可以像处理常规 Git 文件一样推送（push）和拉取（pull）LFS 文件，但请记住，LFS 文件是与 Git 历史记录的其余部分分开存储的。这意味着你可能需要在克隆仓库的任何机器上都安装 Git LFS 才能访问 LFS 文件。

下面是一个可以作为 Git LFS 起点的 `.gitattributes` 文件示例。选择这些文件类型是因为它们很常用，但你可以修改列表以包含项目中可能拥有的任何二进制类型。

```text
# 为 Git 视为文本文件的所有文件规范化换行符。
* text=auto eol=lf

# Git LFS 跟踪 (资产)

# 3D 模型
*.fbx filter=lfs diff=lfs merge=lfs -text
*.gltf filter=lfs diff=lfs merge=lfs -text
*.glb filter=lfs diff=lfs merge=lfs -text
*.blend filter=lfs diff=lfs merge=lfs -text
*.obj filter=lfs diff=lfs merge=lfs -text

# 图像
*.png filter=lfs diff=lfs merge=lfs -text
*.svg filter=lfs diff=lfs merge=lfs -text
*.jpg filter=lfs diff=lfs merge=lfs -text
*.jpeg filter=lfs diff=lfs merge=lfs -text
*.gif filter=lfs diff=lfs merge=lfs -text
*.tga filter=lfs diff=lfs merge=lfs -text
*.webp filter=lfs diff=lfs merge=lfs -text
*.exr filter=lfs diff=lfs merge=lfs -text
*.hdr filter=lfs diff=lfs merge=lfs -text
*.dds filter=lfs diff=lfs merge=lfs -text

# 音频
*.mp3 filter=lfs diff=lfs merge=lfs -text
*.wav filter=lfs diff=lfs merge=lfs -text
*.ogg filter=lfs diff=lfs merge=lfs -text

# 字体与图标
*.ttf filter=lfs diff=lfs merge=lfs -text
*.otf filter=lfs diff=lfs merge=lfs -text
*.ico filter=lfs diff=lfs merge=lfs -text

# Godot LFS 特定文件
*.scn filter=lfs diff=lfs merge=lfs -text
*.res filter=lfs diff=lfs merge=lfs -text
*.material filter=lfs diff=lfs merge=lfs -text
*.anim filter=lfs diff=lfs merge=lfs -text
*.mesh filter=lfs diff=lfs merge=lfs -text
*.lmbake filter=lfs diff=lfs merge=lfs -text
```

有关 Git LFS 的更多信息，请查看官方文档：[https://git-lfs.github.com/](https://git-lfs.github.com/) 和 [https://docs.github.com/en/repositories/working-with-files/managing-large-files](https://docs.github.com/en/repositories/working-with-files/managing-large-files)。
