有没有想过应该用策略 Y 还是 Z 来解决问题 X？本文涵盖了与这些困境相关的各种主题。

## 添加节点和更改属性：哪个在前？ (Adding nodes and changing properties: which first?)

在运行时通过脚本初始化节点时，你可能需要更改属性，如节点的名称或位置。一个常见的困境是：你应该何时更改这些值？

最佳实践是在将节点添加到场景树 **之前** 更改其属性值。某些属性的 setter（设置方法）包含更新其他相关值的代码，而这些代码可能会很慢！在大多数情况下，这些代码对游戏性能没有影响，但在重度使用场景（如程序化生成）中，它可能会让你的游戏慢得像爬行。

由于这些原因，通常最佳实践是在将节点添加到场景树之前设置其初始值。当然也有一些例外情况，即某些值在添加到场景树之前 *无法* 设置，例如设置全局位置（global position）。

## 加载 vs. 预加载 (Loading vs. preloading)

在 GDScript 中，存在全局 [preload](https://docs.godotengine.org/en/stable/classes/class_%40gdscript.html#class-gdscript-method-preload) 方法。它尽可能早地加载资源，以便将“加载”操作前置，避免在性能敏感的代码执行过程中加载资源。

其对应的方法 [load](https://docs.godotengine.org/en/stable/classes/class_%40gdscript.html#class-gdscript-method-load) 仅在程序运行到加载语句时才加载资源。也就是说，它会原地加载资源，这在敏感进程中发生时会导致卡顿。`load()` 函数也是 [ResourceLoader.load(path)](https://docs.godotengine.org/en/stable/classes/class_resourceloader.html) 的别名，所有脚本语言都可以访问。

那么，预加载和加载究竟在何时发生，又该如何选择呢？让我们看一个例子：

```gdscript
# my_buildings.gd
extends Node

# 注意常量脚本/场景与它们的属性变体有不同的命名方案。

# 此值是一个常量，因此在 Script 对象加载时产生。
# 脚本正在预加载该值。这里的优势是编辑器可以提供自动补全，因为路径必须是静态的。
const BuildingScn = preload("res://building.tscn")

# 1. 脚本预加载该值，因此它将作为 'my_buildings.gd' 脚本文件的依赖项加载。
#    但是，因为这是一个属性而非常量，直到脚本使用 .new() 实例化时，
#    对象才会将预加载的 PackedScene 资源拷贝到属性中。
#
# 2. 预加载的值仅凭 Script 对象本身是无法访问的。因此，在这里预加载该值
#    实际上并没有提供任何好处。
#
# 3. 因为用户导出了该值，如果此脚本存储在场景文件的一个节点上，
#    场景实例化代码无论如何都会覆盖预加载的初始值（造成浪费）。
#    通常最好为导出属性提供 `null`、空值或其他无效的默认值。
#
# 4. 仅使用 .new() 实例化脚本会触发 `load("office.tscn")`，
#    忽略通过导出设置的任何值。
@export var a_building : PackedScene = preload("office.tscn")

# 糟糕！这会导致错误！
# 必须为常量分配常量值。因为 `load` 从本质上执行的是运行时查找，
# 因此不能使用它来初始化常量。
const OfficeScn = load("res://office.tscn")

# 成功加载，且仅在实例化脚本时加载！太棒了！
var office_scn = load("res://office.tscn")
```

预加载允许脚本在加载脚本的那一刻处理所有加载工作。预加载很有用，但也有不想使用它的时候。在决定使用哪种方式时，请考虑以下几点：

1.  如果无法确定脚本何时可能加载，那么预加载资源（特别是场景或脚本）可能会导致额外的、意料之外的加载。这可能会在原始脚本的加载操作之上产生无意的、时长不确定的加载时间。
2.  如果有其他东西可以替换该值（如场景的导出初始化），那么预加载该值就没有意义。如果你打算始终单独创建脚本，这一点就不重要。
3.  如果只想“导入”另一个类资源（脚本或场景），那么使用预加载常量通常是最佳方案。但在特殊情况下，你可能不希望这样做：
    1.  如果“导入”的类容易发生更改，那么它应该是一个属性，使用 `@export` 或 `load()` 进行初始化（甚至可能直到稍后才初始化）。
    2.  如果脚本需要大量依赖项，且你不希望消耗过多内存，那么你可能希望随着情况的变化在运行时加载和卸载各种依赖项。如果将资源预加载到常量中，那么卸载这些资源的唯一方法就是卸载整个脚本。如果将它们作为属性加载，则可以将这些属性设置为 `null` 并移除对资源的所有引用（由于它们是扩展自 `RefCounted` 的类型，这将导致资源从内存中自我删除）。

## 大型关卡：静态 vs. 动态 (Large levels: static vs. dynamic)

如果要创建一个大型关卡，哪种情况最合适？是将关卡创建为一个静态空间更好？还是分块加载关卡并根据需要移动世界内容更好？

简单的答案是，“当性能需要时”。与这两个选项相关的困境是编程中历史悠久的选择之一：是优化内存胜过速度，还是反之亦然？

天真的回答是使用一次性加载所有内容的静态关卡。但是，根据项目的不同，这可能会消耗大量内存。浪费用户的 RAM 会导致程序运行缓慢，或者因为计算机尝试同时执行的其他任务而直接崩溃。

无论如何，都应该将较大的场景分解为较小的场景（以帮助资产重用）。然后，开发人员可以设计一个节点来实时管理资源和节点的创建/加载以及删除/卸载。具有大型且多样化环境或程序化生成元素的游戏经常实施这些策略以避免浪费内存。

另一方面，编写动态系统更复杂；它使用更多的编程逻辑，从而产生错误和 Bug 的机会。如果不小心，可能会开发出一个增加应用程序技术债的系统。

因此，最好的选择是……

1.  对于较小的游戏，使用静态关卡。
2.  如果在一款中/大型游戏中有时间/资源，创建一个可以用代码管理节点和资源的库或插件。如果随着时间的推移对其进行精炼以提高易用性和稳定性，那么它可能会演变成跨项目的可靠工具。
3.  在中/大型游戏中使用动态逻辑，因为你拥有编程技能，但没有时间或资源来精炼代码（游戏必须做完）。稍后可能会进行重构，将代码外包到插件中。

有关在运行时交换场景的各种方式的示例，请参阅 [“手动更改场景”](https://docs.godotengine.org/en/stable/tutorials/scripting/change_scenes_manually.html) 文档。
