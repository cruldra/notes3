数据偏好

有没有想过应该用数据结构 Y 还是 Z 来解决问题 X？本文涵盖了与这些困境相关的各种主题。

注意

本文提到了“某某时间（[something]-time）”操作。这一术语来自算法分析的 大 O 表示法。

简而言之，它描述了运行时间长度的最坏情况。用通俗的话来说：

“随着问题域规模的增加，算法的运行时间长度……”

  • 常数时间 (Constant-time)，O(1)：“……不增加。”
  • 对数时间 (Logarithmic-time)，O(log n)：“……以缓慢的速度增加。”
  • 线性时间 (Linear-time)，O(n)：“……以相同的速度增加。”
  • 等等。

想象一下，如果必须在一帧内处理 300 万个数据点。使用线性时间算法是不可能实现该功能的，因为数据的庞大体量会使运行时间远远超出分配的时间。相比之下，使用常数时间算法可以毫无问题地处理该操作。

总的来说，开发人员希望尽可能避免进行线性时间操作。但是，如果保持线性时间操作的规模较小，且不需要频繁执行该操作，那么它可能是可以接受的。平衡这些要求并为工作选择正确的算法/数据结构，是程序员技能价值的一部分。

Array vs. Dictionary vs. Object​

Godot 将脚本 API 中的所有变量存储在 Variant 类中。Variant 可以存储与 Variant 兼容的数据结构，如 Array 和 Dictionary，以及 Object。

Godot 将 Array 实现为 Vector<Variant>。引擎将 Array 内容存储在一段连续的内存区域中，即它们彼此相邻排成一行。

注意

对于不熟悉 C++ 的人来说，Vector 是传统 C++ 库中数组对象的名称。它是一种“模板”类型，意味着它的记录只能包含特定的类型（由尖括号表示）。因此，例如，一个 PackedStringArray 大致相当于 Vector<String>。

连续内存存储意味着以下操作性能：

  • 遍历 (Iterate)：最快。非常适合循环。
    • 原理：只需增加计数器即可获取下一条记录。
  • 插入 (Insert)、删除 (Erase)、移动 (Move)：位置依赖型。通常较慢。
    • 原理：添加/删除/移动内容涉及移动相邻的记录（以腾出空间/填充空间）。
    • 从末尾 添加/删除很快。
    • 从任意位置 添加/删除较慢。
    • 从前端 添加/删除最慢。
    • 如果需要从前端进行多次插入/删除，那么……
      1. 反转数组。
      2. 在一个循环中 在末尾 执行数组更改。
      3. 再次反转数组。 这仅产生 2 次数组拷贝（仍为常数时间，但较慢），而直接操作平均每次需要拷贝大约 1/2 的数组，执行 N 次（线性时间）。
  • 获取 (Get)、设置 (Set)：按 位置 获取最快。例如，可以请求第 0 条、第 2 条、第 10 条记录等，但不能指定你想要的具体记录内容。
    • 原理：从数组起始位置到所需索引的 1 次加法操作。
  • 查找 (Find)：最慢。识别值的索引/位置。
    • 原理：必须遍历数组并比较值，直到找到匹配项。
    • 性能还取决于是否需要详尽搜索。
    • 如果保持有序，自定义搜索操作可以将其提升至对数时间（相对较快）。不过，非专业用户可能不习惯这样做。通过在每次编辑后重新排序 Array 并编写感知顺序的搜索算法来实现。

Godot 将 Dictionary 实现为 HashMap<Variant, Variant, VariantHasher, StringLikeVariantComparator>。引擎存储一个键值对的小数组（初始化为 2^3 即 8 条记录）。当尝试访问一个值时，需要提供一个键。然后它会 哈希 (hash) 该键，即将其转换为一个数字。“哈希值”用于计算数组中的索引。作为一个数组，HashMap 然后在键映射到值的“表”中进行快速查找。当 HashMap 变得太满时，它会增加到 2 的下一个幂（即 16 条记录，然后是 32 等）并重建结构。

哈希是为了减少键冲突的机会。如果发生冲突，表必须为该值重新计算另一个索引，并考虑到之前的位置。总之，这实现了对所有记录的常数时间访问，代价是内存和一些次要的操作效率。

  1. 对每个键进行任意次数的哈希操作。
    • 哈希操作是常数时间的，所以即使算法必须执行多次，只要哈希计算的次数不会过度依赖于表的密度，一切都会保持快速。这引出了……
  2. 维护一个不断增长的表规模。
    • HashMap 故意在表中保留不连续的未使用内存间隙，以减少哈希冲突并维持访问速度。这就是为什么它不断以 2 的幂指数级增加大小。

正如你所见，Dictionary 擅长 Array 所不擅长的任务。其操作细节概览如下：

  • 遍历 (Iterate)：快。
    • 原理：遍历映射内部的哈希向量。返回每个键。之后，用户再使用该键跳转并返回所需的值。
  • 插入、删除、移动：最快。
    • 原理：哈希给定的键。执行 1 次加法操作查找相应的值（数组开始 + 偏移量）。移动是其中两次（一次插入，一次删除）。映射必须执行一些维护以保持其性能：
      • 更新记录的有序列表。
      • 确定表密度是否强制要求扩展表容量。
    • Dictionary 会记住用户插入键的顺序。这使它能够执行可靠的遍历。
  • 获取、设置：最快。与 按键 查找相同。
    • 原理：与插入/删除/移动相同。
  • 查找 (Find)：最慢。识别值的键。
    • 原理：必须遍历记录并比较值，直到找到匹配项。
    • 注意，Godot 不提供开箱即用的此功能（因为它们并非为此任务设计的）。

Godot 将 Object 实现为简单但动态的数据内容容器。当被问及问题时，Object 会查询数据源。例如，为了回答“你是否有一个名为 'position' 的属性？”这个问题，它可能会询问其 脚本 (script) 或 ClassDB。你可以在 在 Godot 中应用面向对象原则 文章中找到有关对象是什么以及它们如何工作的更多信息。

这里的重要细节是 Object 任务的复杂性。每当它执行其中一项多源查询时，它都会运行 若干 遍历循环和 HashMap 查找。此外，查询是线性时间操作，取决于 Object 的继承层次结构大小。如果 Object 查询的类（其当前类）没有找到任何内容，请求将递交给下一个基类，一直向上直到原始 Object 类。虽然这些在孤立情况下都是快速操作，但必须进行如此多的检查这一事实，使得它们在查找数据方面比另外两种替代方案都要慢。

注意

当开发人员提到脚本 API 有多慢时，他们指的就是这条查询链。与应用程序确切知道去哪里查找任何内容的已编译 C++ 代码相比，脚本 API 操作必然会花费更长的时间。它们必须先定位任何相关数据的来源，然后才能尝试访问它。

GDScript 慢的原因是因为它执行的每一项操作都要经过这个系统。

C# 可以通过更优化的字节码以更快的速度处理一些内容。但是，如果 C# 脚本调用引擎类的内容，或者脚本尝试访问其外部的内容，它就会经过这条流水线。

NativeScript C++ 则走得更远，默认情况下将一切保留在内部。调用外部结构将通过脚本 API。在 NativeScript C++ 中，注册方法以将其暴露给脚本 API 是一项手动任务。正是在这一点上，外部的非 C++ 类将使用 API 来定位它们。

那么，假设通过扩展 RefCounted 来创建一个数据结构（如 Array 或 Dictionary），为什么还要选择 Object 而不是其他两个选项呢？

  1. 控制 (Control)：有了对象，就有了创建更精密结构的能力。可以在数据之上构建抽象层，以确保外部 API 不会因内部数据结构的变化而改变。更重要的是，Object 可以拥有信号，从而实现反应式行为。
  2. 清晰 (Clarity)：在脚本和引擎类为对象定义的数据方面，Object 是可靠的数据源。属性可能不包含预期的值，但不必担心该属性最初是否存在。
  3. 便利 (Convenience)：如果你脑海中已经有了一个类似的数据结构，那么从现有类进行扩展会使构建该数据结构的任务变得简单得多。相比之下，Array 和 Dictionary 无法满足你可能拥有的所有用例。

Object 还给予用户创建更专门化数据结构的机会。有了它，你可以设计自己的链表、二叉搜索树、堆、伸展树、图、并查集以及许多其他选项。

“为什么不使用 Node 来构建树结构呢？”有人可能会问。因为 Node 类包含了一些与自定义数据结构无关的东西。因此，在构建树结构时，构建自己的节点类型会很有帮助。

class_name TreeNode
extends Object

var _parent: TreeNode = null
var _children := []

func _notification(p_what):
	match p_what:
		NOTIFICATION_PREDELETE:
			# 析构函数。
			for a_child in _children:
				a_child.free()

从这里开始，你就可以创建具有特定功能的自己的结构，仅受想象力的限制。

枚举 (Enumerations): int vs. string​

大多数语言都提供枚举类型选项。GDScript 也不例外，但与大多数其他语言不同的是，它允许对枚举值使用整数或字符串（后者仅在 GDScript 中使用 @export_enum 注解时）。那么问题来了，“应该使用哪一种？”

简短的回答是，“哪种用着舒服就用哪种。” 这是 GDScript 特有的功能，而非 Godot 脚本的通用功能；该语言优先考虑易用性而非性能。

在技术层面，整数比较（常数时间）会比字符串比较（线性时间）更快。如果你想保持其他语言的约定，那么你应该使用整数。

使用整数的主要问题出现在想要 打印 枚举值时。作为整数，尝试打印 MY_ENUM 将打印 5 或其他什么数字，而不是像 "MyEnum" 这样的字符串。要打印整数枚举，必须编写一个 Dictionary，为每个枚举映射相应的字符串值。

如果使用枚举的主要目的是为了打印值，且希望将它们作为相关概念分组在一起，那么将它们作为字符串使用是有意义的。这样，就不需要一个单独的数据结构来执行打印操作。

AnimatedTexture vs. AnimatedSprite2D vs. AnimationPlayer vs. AnimationTree​

在何种情况下应该使用 Godot 的每种动画类？对于新的 Godot 用户来说，答案可能并不直观。

AnimatedTexture 是引擎作为动画循环而非静态图像绘制的纹理。用户可以操作……

  1. 它移动过纹理每个部分的速率 (FPS)。
  2. 纹理中包含的区域数量 (帧)。

Godot 的 RenderingServer 然后按预定速率按顺序绘制这些区域。好消息是这不涉及引擎方面的额外逻辑。坏消息是用户几乎没有控制权。

另请注意，AnimatedTexture 是一种 资源 (Resource)，与这里讨论的其他 Node 对象不同。你可以创建一个使用 AnimatedTexture 作为其纹理的 Sprite2D 节点。或者（这是其他类做不到的），你可以将 AnimatedTexture 作为图块添加到 TileSet 中，并将其与 TileMapLayer 集成，从而实现许多自动播放动画的背景，且它们都在单个批处理绘制调用中进行渲染。

AnimatedSprite2D 节点结合 SpriteFrames 资源，允许你通过精灵表（spritesheets）创建各种动画序列，在动画之间切换，并控制它们的速度、区域偏移和朝向。这使得它们非常适合控制 2D 帧动画。

如果需要触发与动画更改相关的其他效果（例如，创建粒子效果、调用函数或操作除帧动画之外的其他周边元素），那么将需要结合使用 AnimationPlayer 节点和 AnimatedSprite2D。

如果你想设计更复杂的 2D 动画系统，AnimationPlayer 也是你需要的工具，例如……

  1. 剪纸动画 (Cut-out animations)：在运行时编辑精灵的变换。
  2. 2D 网格动画 (2D Mesh animations)：为精灵的纹理定义一个区域并为其绑定骨架。然后为骨骼制作动画，骨骼会根据骨骼之间的关系成比例地拉伸和弯曲纹理。
  3. 上述两者的混合。

虽然需要 AnimationPlayer 为游戏设计每个单独的动画序列，但在动画混合（即实现动画之间的平滑过渡）方面，组合动画也会很有用。你为对象规划的动画之间可能还存在层次结构。在这些情况下，AnimationTree 大放异彩。
