何时使用场景与脚本

我们已经讨论过场景（scenes）和脚本（scripts）的区别。脚本使用命令式代码定义引擎类的扩展，而场景使用声明式代码进行定义。

因此，每个系统的能力各不相同。场景可以定义扩展类如何初始化，但不能定义其实际行为。场景通常与脚本结合使用，场景声明节点的组成，而脚本使用命令式代码添加行为。

匿名类型 (Anonymous types)​

仅使用脚本来完整定义场景内容是 可行 的。本质上，Godot 编辑器就是这么做的，只是在它的对象的 C++ 构造函数中实现的。

但是，选择使用哪一个可能是一个困境。创建脚本实例与创建引擎内类实例完全相同，而处理场景则需要更改 API：

const MyNode = preload("my_node.gd")
const MyScene = preload("my_scene.tscn")
var node = Node.new()
var my_node = MyNode.new() # 相同的方法调用。
var my_scene = MyScene.instantiate() # 不同的方法调用。
var my_inherited_scene = MyScene.instantiate(PackedScene.GEN_EDIT_STATE_MAIN) # 创建继承自 MyScene 的场景。
using Godot;

public partial class Game : Node
{
    public static CSharpScript MyNode { get; } =
        GD.Load<CSharpScript>("res://Path/To/MyNode.cs");
    public static PackedScene MyScene { get; } =
        GD.Load<PackedScene>("res://Path/To/MyScene.tscn");
    private Node _node;
    private Node _myNode;
    private Node _myScene;
    private Node _myInheritedScene;

    public Game()
    {
        _node = new Node();
        _myNode = MyNode.New().As<Node>();
        // 与调用 new() 或 MyNode.New() 不同。从 PackedScene 实例化。
        _myScene = MyScene.Instantiate();
        // 创建继承自 MyScene 的场景。
        _myInheritedScene = MyScene.Instantiate(PackedScene.GenEditState.Main);
    }
}

此外，由于引擎代码与脚本代码之间的速度差异，脚本的运行速度会比场景稍慢。节点越大、越复杂，就越有理由将其构建为场景。

命名类型 (Named types)​

脚本可以在编辑器本身中注册为一种新类型。这会使其作为一种新类型显示在节点或资源创建对话框中，并带有可选的图标。通过这种方式，用户使用该脚本的能力变得更加流线化。而不需要：

  1. 知道他们想使用的脚本的基类型（base type）。
  2. 创建该基类型的实例。
  3. 为该节点附加脚本。

对于已注册的脚本，该脚本化类型反而成为了一个像系统中其他节点和资源一样的创建选项。创建对话框甚至有一个搜索栏，可以按名称查找类型。

注册类型有两种系统：

  • 自定义类型 (Custom Types)

    • 仅限编辑器。类名在运行时不可访问。
    • 不支持继承的自定义类型。
    • 一种初始化工具。创建带有脚本的节点，仅此而已。
    • 编辑器对该脚本或其与其他引擎类型或脚本的关系没有类型感知。
    • 允许用户定义图标。
    • 适用于所有脚本语言，因为它以抽象的方式处理脚本（Script）资源。
    • 使用 EditorPlugin.add_custom_type 设置。

  • 脚本类 (Script Classes)

    • 编辑器和运行时均可访问。
    • 完整显示继承关系。
    • 创建带有脚本的节点，但也可以从编辑器更改类型或扩展该类型。
    • 编辑器知晓脚本、脚本类与引擎 C++ 类之间的继承关系。
    • 允许用户定义图标。
    • 引擎开发人员必须手动添加对各种语言的支持（包括名称公开和运行时访问）。
    • 编辑器扫描项目文件夹并为所有脚本语言注册任何公开的名称。每种脚本语言必须实现自己的支持来公开这些信息。

两种方法都会向创建对话框添加名称，但脚本类（script classes）特别允许用户在不加载脚本资源的情况下访问类型名称。从任何地方创建实例、访问常量或静态方法都是可行的。

有了这些特性，人们可能希望其类型是一个没有场景的脚本，因为这给予了用户极大的易用性。那些开发插件或为设计师创建内部工具的人会发现这种方式更轻松。

缺点是，这也意味着必须大量使用命令式编程。

脚本 vs 打包场景的性能 (Performance of Script vs PackedScene)​

在选择场景和脚本时要考虑的最后一个方面是执行速度。

随着对象体量的增加，创建和初始化它们所需的脚本大小也会大幅增长。创建节点层次结构就证明了这一点。每个节点的逻辑可能有几百行代码长。

下面的代码示例创建了一个新 Node，更改了它的名称，为其分配了一个脚本，将其未来的父级设为其所有者（以便它能与其一起保存到磁盘），最后将其添加为 Main 节点的子节点：

# main.gd
extends Node

func _init():
	var child = Node.new()
	child.name = "Child"
	child.script = preload("child.gd")
	add_child(child)
	child.owner = self
using Godot;

public partial class Main : Node
{
    public Node Child { get; set; }

    public Main()
    {
        Child = new Node();
        Child.Name = "Child";
        var childID = Child.GetInstanceId();
        Child.SetScript(GD.Load<Script>("res://Path/To/Child.cs"));
        // SetScript() 会导致 C# 包装对象被销毁，
        // 因此在继续之前使用其实例 ID 为 Child 节点获取一个新的包装对象。
        Child = (Node)GodotObject.InstanceFromId(childID);
        AddChild(Child);
        Child.Owner = this;
    }
}

像这样的脚本代码比引擎端的 C++ 代码要慢得多。每条指令都会调用脚本 API，这会导致在后端进行多次“查找”以找到要执行的逻辑。

场景有助于避免这种性能问题。PackedScene 是场景继承自的基类型，它定义的资源使用序列化数据来创建对象。引擎可以在后端批量处理场景，并提供比脚本好得多的性能。

结论 (Conclusion)​

归根结底，最好的方法是考虑以下几点：

  • 如果希望创建一个基础工具，该工具将在几个不同的项目中使用，并且所有技能水平的人（包括那些不认为自己是“程序员”的人）都可能使用它，那么它很有可能应该是一个脚本，可能还是一个带有自定义名称/图标的脚本。

  • 如果希望创建一个特定于其游戏的理念，那么它应该始终是一个场景。场景比脚本更容易跟踪/编辑，并且提供更多的安全性。

  • 如果想给场景起个名字，那么仍然可以某种程度上通过声明一个脚本类并将其场景作为常量来做到这一点。该脚本实际上变成了一个命名空间：

    # game.gd
    class_name Game # 继承自 RefCounted，所以不会出现在节点创建对话框中。
    extends RefCounted

    const MyScene = preload("my_scene.tscn")

    # main.gd
    extends Node
    func _ready():
    	add_child(Game.MyScene.instantiate())
    // Game.cs
    public partial class Game : RefCounted
    {
        public static PackedScene MyScene { get; } =
            GD.Load<PackedScene>("res://Path/To/MyScene.tscn");
    }

    // Main.cs
    public partial class Main : Node
    {
        public override void _Ready()
        {
            AddChild(Game.MyScene.Instantiate());
        }
    }
